# Brainstorm

Confidence-driven collaborative brainstorming. Interviews you until it understands what you **actually want** — not what you think you should want — then proposes approaches, builds a design, and lets you choose what happens next.

<HARD-GATE>
Do NOT write any code, scaffold any crate, invoke any implementation skill, or take any implementation action until you have presented a design (or a triage recommendation) and the user has explicitly approved it. This applies to EVERY topic regardless of perceived simplicity.

Carve-out: when the user's initial request pre-authorizes a deliverable contingent on the brainstorm's verdict (e.g. "write an ADR if changes are warranted") — or issues an unconditional build/port/"adapt" directive for a specific deliverable (e.g. "adapt this skill into a repo-local one") — presenting the design or triage recommendation (for a Port, the transformations table) IS the approval that satisfies this gate — see Guardrails §User pre-authorization. The Step 6 menu still fires afterward.
</HARD-GATE>

## Process flow

```
Read context (reference file + classify topic)
        |
        v
Confidence-driven interview loop (target: 95%)
        |
        v
Propose 2-3 approaches with tradeoffs
        |
        v
Present design section by section, get approval per section
        |
        v
[If product-behavior] Validate against docs/FOUNDATIONS.md
        |
        v
Persist the deliverable (design doc / ADR / agent task / ...)
        |
        v
Next-steps menu (user chooses)
```

**In plan mode**: the design writes to the plan file instead of `docs/plans/`, and `ExitPlanMode` replaces the next-steps menu.

**Triage/analysis brainstorms** (evaluate a report or finding-set, or answer a diagnostic question, and produce work items *or a single recommended fix*): the interview loop and the propose-approaches step are replaced by a single triage recommendation — see Step 3 §Triage brainstorms and `references/triage-and-deliverables.md`.

## Step 1: Read context

1. **Reference material.**
   - **`reference_path` argument**: read the entire file. Extract its key claims, proposals, and open questions.
   - **Inline file references**: if the user names files inline in the request, read them with the same treatment. Read all of them.
   - **Inline-pasted prose**: if the user pastes reference content (external research output, a report excerpt, a finding-set) directly into the request body, that prose IS the reference — there's no file to read.
   - **Not reference material**: the user's own first-hand problem description (bug report, symptoms, repro steps, pasted error/program output as evidence of the defect) is the request itself, not externally-authored reference content. A file named only as the alignment authority (`docs/FOUNDATIONS.md`) is also not reference material — it's governed by sub-step 3. Inline-named **internal codebase files read for orientation** (`README.md`, CI/config, source files the user points at to ground the design) are likewise not reference material — they're covered by exploration (sub-steps 6–7) and get no reference summary; reserve the 2-3 sentence summary for inline-named **authored artifacts** (a report, research doc, external spec). The discriminator is authored-reference-content vs. repo-state-for-context, not merely whether a filename was mentioned. **Transform-source exception**: when the user pastes their *own* prose *as the content to be transformed into a deliverable* (formalize-this-instruction-into-a-skill, codify-this-prose-into-a-doc), treat it like a Port source (Step 3 §Port external skill) — reference material that earns the 2-3 sentence summary anchor — rather than routing it to the "No reference material" confirming line, because it is the authoritative input for the transformation regardless of authorship. This straddles the two rules above (user-authored, yet the transformation source); the transform-source role is the tiebreaker.
   - **Oversized files**: if a reference exceeds the Read token limit, read in sequential offset/limit chunks until fully covered, and say so in the summary. Run a quick `wc -l <path>` first when you suspect a file is large.
   - **Path resolution**: if an inline filename doesn't resolve, glob the parent directory for the closest match (pluralization, hyphen drift, capitalization) before asking the user to re-specify; announce any correction inline — surface it in the reference-summary anchor below (e.g. "Resolved <typed-path> → <actual-path>"), so the audit trail records the resolution.

   **Emit a 2-3 sentence summary** of what the reference material contains as user-facing text before any further exploration — this is the user's audit-trail anchor for verifying you understood the reference. For a multi-item report, summarize its *shape* (item count, structure, headline position) rather than enumerating each item; per-item detail belongs in the Step 3 triage. When the reference makes claims about the codebase or product state, treat them as hypotheses to verify during exploration (sub-step 6), and flag prominently any that exploration contradicts. When there is no reference material at all (pure analysis from the request text), no summary fires — instead **emit a single confirming line** ("No reference material — the request itself is the spec; proceeding to exploration") as the audit-trail anchor before exploration, so the reference-material disposition is never silently empty; it is distinct from the multi-sentence reference summary. A user-named prior internal spec/plan cited as progress context ("the latest spec we implemented was X") is repo-state-for-context per the discriminator above, so this confirming line — not a reference summary — is the correct anchor for it.

   **No reference material?** Emit exactly this line before exploration: "No reference material — the request itself is the spec; proceeding to exploration." This is the audit-trail anchor for the empty-reference case (bug-report-as-spec, repro-output-as-evidence, or codebase-orientation-only request). It fires even in fast diagnostic/triage flows; skipping it leaves the reference-material disposition silently empty.

2. **Topic classification.** Announce the classification on its own line as soon as it's determinable — often right after reading the primary reference's frontmatter/intro, and in all cases before the broader project-context exploration in sub-step 7. Don't let a large multi-file reference read push the announcement arbitrarily late; classify from the minimum evidence that settles the category, then keep reading. Pick the category the brainstorm falls into:
   - **product-behavior**: anything that changes how the app behaves at runtime or what it persists — rules, legal action generation, validation, state transitions, scoring, terminal detection, deterministic randomness, semantic effects, replay/checkpoint/hash behavior, public/private view projection, serialization contracts, bot choices, or effect-log-driven UI/renderer behavior, together with the `engine-core` kernel boundary, the earned-`game-stdlib` boundary, the static-data-is-not-behavior boundary, and the IP/public-private boundary that govern them. These are governed by `docs/FOUNDATIONS.md` and the architecture docs it anchors (`docs/ARCHITECTURE.md`, `docs/ENGINE-GAME-DATA-BOUNDARY.md`, `docs/OFFICIAL-GAME-CONTRACT.md`, `docs/MECHANIC-ATLAS.md`, `docs/AI-BOTS.md`, `docs/UI-INTERACTION.md`, `docs/TESTING-REPLAY-BENCHMARKING.md`, `docs/IP-POLICY.md`, `docs/AGENT-DISCIPLINE.md`).
   - **dev-process / docs / tooling**: work that does not change product behavior — build/test/benchmark scripts, dev ergonomics, repo tooling, skill design, or documentation/template edits that don't change product semantics. But a benchmark/CI/tooling change that moves gating *doctrine* fixed by an accepted ADR or the testing law (a threshold calibration, lane, or budget) is **product-behavior** — classify by the doctrine it moves, not the file it edits, and read FOUNDATIONS accordingly. A dev-process deliverable that creates or alters **brand-facing visual assets** (icons, logos, imagery, public naming) keeps its classification but requires a `docs/IP-POLICY.md` check — originality, no trade-dress proximity to any commercial game — with the check's outcome stated in the design rationale.

   When genuinely ambiguous, give a one-to-two-sentence justification. Example: `Classification: product-behavior (the design adds a validation rule to the legal-action generation path).`

   **Planning/spec artifacts.** A deliverable that is itself a *plan* (a spec, roadmap, design doc, or agent task) is classified by its own nature — it's a document, so it is **dev-process/docs** even when the work it plans is product-behavior. Still apply the Step 4 FOUNDATIONS alignment table to the *work the plan describes* when that work engages a principle or a §12 stop condition (e.g. a spec that establishes the `engine-core` boundary). Classify by the artifact you write, align by the behavior it governs. **A dev-process deliverable that *commands* product-behavior work** — a skill, automation, or agent-task harness that, when run, drives changes to rules / validation / effects / scoring / determinism / views / bots — is classified dev-process by its artifact, but its design must embed the `docs/FOUNDATIONS.md` and `docs/AGENT-DISCIPLINE.md` guardrails the commanded work must respect (the failing-test protocol, the kernel/static-data boundaries, no-leak, determinism): align by the behavior the deliverable *drives*, not only the behavior a passive plan *describes*. **When the deliverable artifact is undetermined at the sub-step 2 announcement** — common in triage/diagnostic brainstorms, where Step 3/5 decides only later whether the fix is tickets, an ADR, or a design doc — classify at announcement time by the *topic/behavior under analysis* (so a bug-behavior diagnosis announces `product-behavior`, which correctly pulls in the FOUNDATIONS read), then re-state the artifact-based classification if Step 5 selects a plan artifact. The Step 4 "Classification pivot check" is that re-statement hook; the two rules don't conflict once the announcement is understood as topic-keyed and the pivot as artifact-keyed. **When the symptom-surface and the root-cause-surface differ** — a CI/gate/build/test-failure diagnostic whose failing *job* is a tooling surface but whose root cause is not yet located ("why is Gate 0 failing") — don't announce `product-behavior` on the speculation that the fix *might* touch product code; default to `dev-process/tooling` (or defer the announcement — but state any deferral explicitly at the sub-step 2 point, e.g. `Classification: deferred pending root-cause location — provisionally dev-process/tooling`, so the deferral reads as a deliberate choice rather than an apparent omission of the "on its own line as soon as it's determinable" rule above) until exploration locates the failing code, then classify by that root-cause surface. The FOUNDATIONS-read decision hinges on where the cause lives, not on which job reported it — a lint or build-script fix stays dev-process even though the gate also fences product behavior. The same divergence arises for a **UI/display defect** whose cause could be either the presentation layer (TS/CSS — dev-process) or a behavior/view-projection surface (product-behavior): announce provisional `product-behavior` so the FOUNDATIONS read primes the boundary check, then pivot via the Step 4 pivot check once exploration locates the cause (a parser or CSS fix settles to dev-process; a view-projection or legality change stays product-behavior).

   **Governance-doc editorial gate.** When a dev-process/docs topic restructures `docs/FOUNDATIONS.md` or the law/architecture docs it anchors (the doc set listed under product-behavior above) — merging, splitting, de-duplicating, moving, or re-indexing them — keep the dev-process/docs classification but apply a semantic-preservation gate: meaning-preserving moves/merges/de-duplication are editorial and need no ADR, whereas any change to a *principle's meaning* requires an ADR superseding it first (FOUNDATIONS: "Supersede only by accepted ADR"). In this case read FOUNDATIONS even though the topic is dev-process (sub-step 3's optional-read carve-out does not apply), state the preservation guarantee in the design, and surface any merge that would force a genuine semantic reconciliation rather than silently resolve it.

3. **Read `docs/FOUNDATIONS.md` for product-behavior topics.** You'll need it in Steps 3 and 4 to validate approaches against the constitution and its §12 Stop conditions (and, for agent-task and review work, the AGENT-DISCIPLINE §13 review checklist and the ADR review checklist in `docs/adr/ADR-TEMPLATE.md`). For dev-process/tooling/docs topics the read is optional — read it only if the topic directly engages a FOUNDATIONS principle. The read is satisfied if FOUNDATIONS content is already in this session's context; re-read only after compaction or when you need a specific principle's wording. **Secondary digests and restatements do NOT satisfy the read** — the CLAUDE.md non-negotiables block, `tickets/README.md`'s §-name list, and principle restatements in archived tickets/specs are summaries, not the constitution; only the file itself counts. When a provisional `product-behavior` classification (sub-step 2) pivots to dev-process before the read happened, the *obligation* lapses with the pivot — but not the citation gate: any FOUNDATIONS § number cited in a deliverable must still be verified against the file per Step 4's premise-verification rule. If a brainstorm reveals that the *right* design conflicts with FOUNDATIONS, the deliverable may need to be an **ADR that supersedes the relevant principle first** — FOUNDATIONS states "Supersede only by accepted ADR" — so flag this prominently and get explicit sign-off; never quietly design against the constitution. The same ADR-first trigger applies when the design **amends or supersedes the doctrine fixed by an existing accepted ADR** (recalibrating a value, threshold, boundary, or lane an ADR established) — even for a **dev-process/tooling topic** where the FOUNDATIONS read is otherwise optional, write the amending ADR before the implementing change; a plain ticket or config edit must not silently weaken accepted-ADR doctrine (this is a FOUNDATIONS §13 ADR trigger, not only a FOUNDATIONS-principle trigger).

4. **Calibrate initial confidence from the reference and request.** A thorough reference (rationale, decisions, structure) or a richly-specified request (problem analysis, evidence, root cause, code locations, a clear ask) can start confidence at 70-95%, reducing the interview to closing operational gaps. Don't ask motivational questions when the user has already demonstrated deep understanding.

5. **Interview-skip threshold.** If exploration and research bring confidence to 95%+ before the interview starts, skip Step 2 and go to Step 3; announce the confidence level. At 85-94% with a user-framed, fully-constrained directive (enumerated alternatives, or a clear "do X" with minimal open choices — triage brainstorms are the common case), also skip the discovery interview: go to Step 3, announce confidence, and carry remaining gaps as named assumptions for the user to correct during approach/triage selection. **Before either disposition below, check whether the gap is independently verifiable** — a fact the environment can settle deterministically (repo visibility / CI config via `gh`, Cargo metadata, a targeted codebase read) is not a user-preference gap: verify it and fold the result into the recommendation rather than asking. Rule 9's verify-don't-ask discipline applies even though the discovery interview was skipped — `AskUserQuestion` and named-assumptions are for genuinely user-unknowable or preference gaps, not for facts you can check in one call. Resolving a recommendation-determining fact this way also prevents asserting a recommendation that a later same-batch `AskUserQuestion` answer would undercut. **When a remaining gap would *materially shape the deliverable*** (which approach, what gets stripped vs. kept, the deliverable's structure or class) rather than only a detail, prefer posing it via `AskUserQuestion` at approach-selection rather than carrying it as a silent assumption; reserve named-assumptions for detail-only gaps. **Precedence under autonomous pre-authorization**: an approach-level gap that does NOT change the deliverable's *shape* (count, class, or required files — the §Guardrails §User pre-authorization triggers) is carried as a named assumption, with the alternatives considered enumerated in the final message; the `AskUserQuestion` preference applies in interactive flows, or when the gap is a true shape change per that guardrail. The two "shape" phrasings are scoped by flow: "structure or class" drives the interactive `AskUserQuestion` preference; "count, class, or required files" drives the autonomous named-assumption fallback. When the chosen deliverable's template carries its own Assumptions section (a spec's "Assumptions (one-line-correctable)", an ADR's open-questions area), land the named assumptions there as one-line-correctable entries and point at them from the design presentation — the template section is the durable carrier; the chat-side list is the preview. Under autonomous pre-authorization the chat-side preview is not optional: the final message enumerates the template's assumption IDs, each with a one-line hook (see §Guardrails §User pre-authorization).

6. **External research.** If the topic needs domain knowledge beyond the codebase (algorithms, game-rule sources, industry practice, competing architectures), launch research agents — or an available research skill (e.g. `deep-research` for in-session multi-source research, `research-brief` for externally-delegated research) — BEFORE the interview and summarize findings for the user first. For bounded grounding (a handful of comparables or practitioner patterns), direct inline searches (`WebSearch`/`WebFetch`) suffice — reserve agents/skills for multi-source synthesis or contested claims. When delegating to a research agent, bound its report in the prompt exactly as sub-step 7 bounds Explore delegates (a line cap, per-claim citations, a one-line quote limit, and "your final message IS the deliverable"). If codebase exploration already yields a clear root cause with concrete evidence, research may be skipped — note the skip. If research was supplied as a reference file or pasted prose, the sub-step 1 summary satisfies the announcement. When the discovery interview is skipped (sub-step 5), there is no natural wait point for a backgrounded research agent — block on its completion (continuing non-overlapping exploration meanwhile) rather than ending the turn mid-flow; the research findings belong in the same turn as the triage/approach they inform. Exception: when the research runs as a harness-tracked background task (e.g. a workflow) that outlives the available non-overlapping exploration, ending the turn with a brief status note and resuming on the completion notification is sanctioned — the obligation shifts to the resuming turn, whose deliverable must carry the research summary together with the triage/approach it informs. Environment-mandated search tooling (a hook or session directive naming a replacement tool) substitutes for the named tools; if it fails hard (quota, outage, or returns no results for a term you have independent reason to believe is present — a stale or unbuilt index), fall back to the named tools and record both the substitution and the degradation; never treat a mandated-tool empty result as proof of absence without a named-tool cross-check. The substitution rule extends to delegated research agents when the mandate's mechanism reaches them (a hook firing inside the subagent, or the mandate passed in its prompt); when it cannot, note the bypass in the degradation record. The mandate also governs ad-hoc `grep`/`rg` invoked via Bash for premise verification (Step 4; Interview rule 9), not only the Grep/Glob/Explore tools — route those through the mandated tool too, falling back only on hard failure. A hard failure already recorded earlier this session or in same-day reference material may be relied on without a fresh retry, provided the skip/degradation note cites that record. If research degrades rather than completes (tool failure, quota exhaustion, blocked sources), record the degradation and its coverage impact as a named assumption in the deliverable, alongside any skip note.

7. **Project context.** Explore relevant existing material before the interview — `docs/`, `docs/adr/`, `docs/plans/`, `specs/`, `tickets/`, `templates/`, prior design docs, and the relevant game module (`games/*`), `engine-core`, schema, or trace files when the topic touches them. Launch Explore agents for broad surveys — and bound each delegate's final report explicitly in its prompt (e.g. "cap the report at ~150 lines; constraints only, cited by file + section; no quote longer than a line"), per CLAUDE.md (global) §Tool Output Discipline: an unbounded survey report can blow the per-turn tool-result budget and get persisted to disk instead of returned. If a report does overflow and persist, recover with a paged `Read` of the persisted file (offset/limit) — do not re-print it through Bash, which just persists it again. The sub-step 6 environment-mandated-search-tooling rule applies to codebase search too: when a hook or session directive mandates a replacement for the local search tools (e.g. `mgrep` for Grep/Glob) and it fails hard (quota, outage, or returns no results for a term you have independent reason to believe is present — a stale or unbuilt index), fall back to the named tools (Grep/Glob/Explore) and record both the substitution and the degradation. For frustration-triggered brainstorms, also explore the history of attempted fixes — repeated remediation or accumulated workarounds signal that Step 3 should include radical options (reset, strip, rebuild) alongside incremental fixes. For triage brainstorms whose input is a report, check whether that report (or an earlier version) was already triaged into an existing ADR/agent task/`docs/triage/` file, and frame the current pass as a delta so already-accepted or already-rejected items aren't silently re-proposed. When the report instead targets a game or area that a **prior implementation spec already shipped** (a `Done` predecessor in `specs/README.md`), the delta is against *shipped code*, not a prior triage: verify each claimed-still-open defect against the current tree before assigning a verdict — a fully-shipped item is `already-resolved` (cite the predecessor spec + its completion date), a partially-shipped item is `accept-with-modification` scoping only the residual. The report's own "delta" self-description does not discharge this check — a delta report routinely re-raises work the predecessor closed. **For UI or runtime-behavior topics with a running instance available**, exploration MAY also include driving the live app (Puppeteer/Playwright — navigate, screenshot, read the DOM) to reproduce the defect and confirm the root cause before proposing; present the reproduction as empirical findings per Step 2 rule 10, and record the driven session's reproduction parameters alongside the findings — a deterministic repo makes the whole audit replayable. Which parameters are load-bearing depends on the defect class: seed, variant, mode, date for state-driven defects; viewport/breakpoint, selected entity, and theme for layout/presentation defects (seed/variant/mode are often irrelevant to a pure CSS/layout bug). **For CI/gate/build/test-failure diagnostics, reproduce the *entire* gate set locally, not just the reported failure** — fail-fast pipelines (`bash -e` steps, fail-fast jobs) abort at the first failure and mask every downstream one, so the reported job is a lower bound on what's broken. Re-run each gate to completion (e.g. all games' coverage/fixtures, the full build/test sweep) and present the *complete* failure inventory; a triage built from only the first-reported failure is incomplete and ships the user a second red CI cycle. The masking shapes — and the candidate-fix-verification technique — to account for:
   - **Fetch the actual CI failure log first** (e.g. `gh run view <run-id> --log-failed`, or the platform's equivalent) to identify the *first-failing* step: that confirms the reported failure and scopes the local reproduction to that step and everything after it, since every step before it is already proven green in the CI run — so "reproduce the entire gate set" in practice means "reproduce from the first failure onward," not blindly re-run steps the log already shows passing.
   - **The failure log shows only the first-failing step**; the masked downstream steps in the same fail-fast job never ran, so they are absent from it — "everything after it" is *not* derivable from the log. To enumerate what "onward" contains, read the failing **job's workflow definition** (`.github/workflows/*.yml`) for its full ordered step list, then run every step at or after the first failure to end-of-job. Do **not** derive the check list from the project's documented command list (e.g. CLAUDE.md §Commands), which may be an incomplete subset of the job's steps — declaring green off that subset is exactly what ships the user a second red cycle.
   - **When the pipeline is `fail-fast: false` (or the CI UI already lists per-job pass/fail across all lanes), CI's own job results ARE the complete *job-level* inventory** — confirm it with the run's job list (`gh run view <run-id> --json jobs`) and scope local reproduction to the failed lane(s) only. But job-level `fail-fast: false` makes only the *job* list complete, not a failed job's *steps*: `bash -e` step sequencing still masks every step after the first failure *within* each failed lane, so reproduce each failed lane **from its first-failing step onward** (per the step-list bullet above) — never treat the lane itself as fully reported just because the matrix is. The entire-gate-set re-run above is the remedy for fail-fast pipelines where downstream steps were masked, not for an already-fully-reported matrix (which means job-level completeness, distinct from the within-lane step completeness just described).
   - **A cascade-skipped dependent job is another masking shape**: when the failed job has dependents (`needs:` it in the workflow) that CI auto-skipped because their prerequisite failed, those jobs never ran and are absent from the results above — `skipped` is not `passed`, so the `fail-fast: false` "CI results ARE the complete inventory" rule does **not** cover them. For each skipped dependent job, read its step list and reproduce any step that shares the failed root cause (a sibling job re-running the same broken script or tool is the common case); otherwise the fix clears the reported job but the skipped sibling fails on the next cycle.
   - **A single step that soft-guards its sub-units is the inverse shape**: when one step runs many sub-units in sequence with non-aborting guards (an `if ! cmd; then failed=1` collector that defers the exit to the end), nothing downstream is masked — every sub-unit runs to completion and that one step's log is already the complete failure inventory, so local re-running is for *verifying the fix*, not enumerating failures. Confirm the inventory is complete by checking the log reached the step's end-of-loop summary — proving no *unguarded* command (e.g. an unguarded `cargo bench … | tee` under `set -e`/`pipefail`) aborted the step early — rather than re-running the whole gate set on the assumption that something downstream was hidden.
   - **A single fail-fast compile/lint/type command is a masking shape the "re-run to completion" remedy does not cover**: when a gate step *is* one such command (`cargo clippy --workspace … -D warnings`, `cargo build`, `tsc`), it aborts at the **first** error and masks every later error within the same invocation — so a single re-run enumerates only up to that first error, not the full set, and "reproduce the entire gate set" cannot be discharged by one run. Treat the first error as a **lower bound**: fix it, re-run the *same* command, and iterate until it exits clean **before** presenting the triage — do not present the finding set as complete after one run. The corollary at verification time: fixing the first error *unmasks* later ones that were never regressions but were hidden behind it (this session: fixing a `wasm-api` `needless_borrow` unmasked a `useless_vec` in a `meldfall_ledger` test that CI never reached), so the §Deliverable-classification *Post-fix verification* re-run must also iterate-to-clean, not assume one pass enumerates. A same-class finding unmasked late despite this discipline is applied inline per Guardrails §Blocker discovery and appended as a new `O<N>` item. **Disposition of fixes applied during enumeration**: an edit applied to a tracked file to unmask later errors is a real change, not a fait accompli — at triage/approach presentation, surface each such edit explicitly as an applied finding and either (a) keep it under the Step 5 inline-ops carve-out, with the Step 6 menu offering a revert / keep / promote-to-ticket choice, or (b) revert it and present it as a proposed fix for approval. A kept edit still owes the HARD-GATE its menu/approval and must never stand un-surfaced — distinct from the candidate-fix bullet below, whose `apply → compile/lint → revert` always reverts.
   - **Verifying a candidate fix**: for build/lint/type diagnostics where a tool's *suggested* fix may not hold (e.g. a clippy lint suggestion that breaks a trait bound), empirically test the candidate — apply → compile/lint → revert — and present the result as empirical findings (Step 2 rule 10); a refuted suggestion materially changes the recommendation. Mutating a tracked file during exploration — **including a `git checkout` revert** — invalidates any prior `Read` of that file, so re-`Read` each target immediately before the real `Edit` (Step 4), or the first `Edit` fails with "File has been modified since read."

   **Sequential / roadmap-driven brainstorms.** When the request is "produce the next X to continue Y" (the next spec for a roadmap, the next stage of a staged plan), the retread risk is cross-session, not just intra-session. Before proposing, read the progress index for that series — a `specs/README.md` spec index, a roadmap status table, or the existing `specs/`/`tickets/` files — pick the lowest unit whose status is not done — honoring any interlock/admission rules the index or its workflow section declares (e.g. open promotion debt closes before the next mechanic-ladder gate; check the register the index points at) — **and skip any index unit the index or roadmap flags optional, subordinate, or below the active priority order** (e.g. a private/optional late-stage gate that FOUNDATIONS §1 ranks beneath polished public work): when the mechanic-ladder gates are all done but such a subordinate unit is the only non-done index row, the live continuation is the higher-priority work tracked elsewhere (a prioritization/ideation doc, or the non-gate sibling-spec series), not that subordinate unit — and frame the deliverable as exactly that increment.

   **Structural template.** The most recent _same-kind_ archived deliverable is the structural template for the new one (a realignment/aftermath spec templates a realignment spec, a game-gate spec templates a game-gate spec); fall back to the literal immediately-preceding unit only when no same-kind predecessor exists, or none exists at all at an epoch boundary. Cover the chosen template fully, using chunked offset/limit reads when it exceeds the Read cap (the sub-step 1 oversized-file rule extends to this read even though the predecessor is repo-state, not reference material) — at minimum its canonical sections plus Sequencing and Outcome, where carried obligations and the verified-by-predecessor anchor often live in the tail; don't judge an unseen tail non-load-bearing without reading it. This coverage rule applies whenever any cycle — sequential or not — adopts an archived deliverable as its structural template: at minimum read the canonical section list and the template's tail sections before authoring. This predecessor-template read does **not** discharge the Step 5 deliverable-row mandated reference read (`references/triage-and-deliverables.md` §Deliverable classification — which governs roadmap/sequential specs, **not only triage**, despite the filename): the two are complementary, not independently-droppable (Step 4 template-structured carve-out), so perform the reference read before authoring even on this direct-authoring path and even when the template read substantially overlaps it.

   **Register sweep & index update.** Before authoring, also **sweep the domain's candidate/deferred registers for rows naming the picked unit** (e.g. grep the mechanic atlas for the unit id): index rows understate armed obligations, and every register row naming the unit is a spec-shaping obligation — this applies on the authoring path, not just the determine-only case below. Make updating the index status part of the deliverable so the next cycle resumes cleanly — and when the unit was picked from a reference prioritization/ideation document (a deferred-work brainstorm, a priority list) rather than, or alongside, the progress index, also include marking that document's item done, with a pointer to the deliverable, in the closeout obligations, for the same anti-retread reason; authoring a spec flips its index row to the *spec-written* status (e.g. `Planned`), not the terminal *done* status, which waits until its exit criteria pass with evidence; when flipping the row, mirror the index's established row format — link, one-line scope summary, and any carried obligations the next cycle must see — rather than a bare status change. When the spec's *own deliverables* include broader index-content edits (intro/workflow/format rewrites), authoring-time index work is limited to the row-add, status flip, and blocking-interlock note; those content rewrites land during implementation as the unit's deliverables, not at authoring.

   **Non-gate placement.** A **non-gate spec** comes in two shapes. A *polish/UX* non-gate spec (e.g. a post-gate UX/presentation pass on a shipped game) takes a non-gate row (e.g. `Order: —`) in the *live* tracker adjacent to the unit it polishes, not a frozen/completed table whose non-gate rows are already closed. A *prerequisite-realignment* non-gate spec (a report-seeded foundation/doctrine/template pass that gates the next ladder gate, like the Phase 0 realignment at `Order 0`) instead takes an **ordered interlock row** positioned before the gate it unblocks, and updates that gate's interlock note to require it — it advances no gate itself but is a hard precondition for one. When a granularity decision merges several pre-existing seed rows into one (e.g. one combined spec chosen for a ROADMAP unit the index had seeded as separate rows), replace all merged rows with a single row, renumber or annotate the `Order`/interlock column so no dangling sub-unit reference remains, and verify that downstream rows naming the merged sub-units (a successor's `pending X` interlock) still resolve. If no index exists yet (greenfield series), create one alongside the first deliverable so future cycles have a tracker; confirm the empty/greenfield repo state explicitly, since "the first unit" is then the load-bearing premise.

   **Determine-only & authoring pick.** When the request is determine-only ("which spec is next?" with no authoring directive), the deliverable is the recommendation itself — the picked unit plus its spec-shaping obligations from the index, atlas, and law docs — closed by a Step 6-style menu whose first option authors that unit. **When the user picks that authoring option**, continue the same cycle (no Step 1 re-entry, no fresh confidence anchor — distinct from the Step 6 related-scope re-entry rule): fire the Step 4 pivot check, perform the Step 5 deliverable row's mandated reference read and premise verification (including grep-verifying every `FOUNDATIONS §N` citation in the spec's alignment table against the constitution itself, not a restatement), then write under the template-structured carve-out — the menu pick is the approval. **When the unit to author embeds original game/rules design** (proposed rules, constants, balance analysis), make the sub-step 6 research-or-skip decision explicit before authoring — name the available research path (e.g. `research-brief` / `deep-research`) — and record a skip as a one-line-correctable entry in the deliverable's Assumptions section.

   **Epoch rollover / batch authoring.** When the request authors *several* units at once ("create as many specs as warranted") or rolls the index to a new phase rather than continuing it linearly, author only the units that are *groundable now* — those not blocked on a prerequisite unit's outputs (an accepted ADR, a ROADMAP edit, or a contract doc that a Phase-0-style unit must produce first) — and seed every blocked downstream unit as a forward index row that names its blocking interlock. Size "how many" with the Step 3 scope-tier mechanism (minimal / standard / comprehensive), but let the dependency structure, not appetite, set the ceiling: a unit whose grounding premise (its ROADMAP gate, its governing doctrine) does not yet exist cannot be honestly specified now. Treat index *replacement* as an explicit deliverable governed by `docs/archival-workflow.md` — archive the prior index date-suffixed under `archive/specs/` and write a new one, rather than only flipping a row — and before moving the index, confirm the repo's doc-link checker's scan roots (it may skip `archive/`, so a moved index's now-stale relative links are tolerated; if it does not, fix them).

8. **Confidence checkpoint.** After all context reading, announce post-exploration confidence in one sentence on its own line (e.g. `Post-exploration confidence: ~90% — reference file is exhaustive, scope fully specified by the request`). This drives the Step 2 decision and is the audit-trail anchor before any triage recommendation. (In a fully collapsed pre-authorized turn this anchor may land solely in the final message — see §Guardrails §User pre-authorization.)

## Step 2: Confidence-driven interview

The core of the skill. Reach **95% confidence** about what the user actually wants before proposing solutions.

### The protocol

After each user answer, display a confidence block:

```
Confidence: X%
Gaps: [specific list of remaining unknowns]
```

Keep asking until confidence reaches 95%, then announce: "I'm at 95% confidence. Moving to approaches." Under auto mode with a 0-2 question interview, an inline prose alternative is fine (`Confidence: ~90% — gaps listed as named assumptions below`) provided the gaps surface as named assumptions in the very next message.

### Interview rules

1. **One question per message.** Never batch questions in the discovery interview. (This does not restrict `AskUserQuestion`, whose multi-question schema is the sanctioned form for batching approach-selection or section-approval decisions outside the discovery loop.)
2. **Prefer multiple-choice** when the answer space is bounded; open-ended when it isn't. Use `AskUserQuestion` with labeled options when its schema is available; inline numbered options are an acceptable fallback (and are preferred in plan mode for speed).
3. **Probe motivations before solutions.** Ask "What problem does this solve?" and "What happens if we don't?" before "What do you want built?" The first request often describes a solution, not the problem.
4. **Challenge premature specificity.** If the user jumps to implementation details, ask why that specific approach matters. The constraint is often softer than stated.
5. **Detect "should want" vs "actually want".** Watch for buzzword-heavy descriptions, over-scoped requests, vague success criteria ("it should be good" — probe what "good" means), and solutions stated as requirements.
6. **Name your uncertainty specifically.** "I don't know whether this needs to handle edge case X" is useful; "I need more information" is not.
7. **Respect user expertise.** If the user gives a clear, well-reasoned answer, don't re-ask it in different words. Advance.
8. **Handle delegation gracefully.** If the user says "you decide," re-evaluate the options against FOUNDATIONS and project constraints, present your reasoned recommendation, and advance. Don't re-ask.
9. **Verify load-bearing technical premises.** When a user (or reference) premise is load-bearing for the recommendation — crate/library status, version compatibility, feature presence, external-tool behavior — verify it (Cargo metadata, docs, targeted codebase read) before adapting the recommendation to it. The 30-second check prevents recommending against a true premise the user momentarily mis-stated.
10. **Present empirical findings before asking questions.** When exploration produces concrete findings (data, root-cause evidence, confirmed/refuted hypotheses), present a concise summary — hypothesis, evidence, verdict per hypothesis — before the first interview question.

### Confidence scoring

| Range | Meaning | Action |
|-------|---------|--------|
| 0-30% | Don't understand the problem yet | Ask about the problem, not the solution |
| 30-60% | Understand the problem, unclear on constraints | Ask about constraints, success criteria, scope |
| 60-80% | Understand problem + constraints, unclear on priorities | Ask about tradeoffs, what matters most |
| 80-95% | Clear picture, a few edge cases/preferences unknown | Ask targeted questions about specific gaps |
| 95%+ | Ready to propose | Move to Step 3 |

Confidence rises from **both** user answers and research findings; when research closes a gap, note which gaps it closed vs which still need user input.

### Early exit

If the user says "just go" or "that's enough questions," respect it: announce current confidence, list remaining gaps as assumptions you'll make, and proceed. Mark those assumptions explicitly in the design so the user can correct them. Use the format `(N) X — assuming Y` (the unknown as X, the assumption after `— assuming`) so a correction is one line.

### Recovery/reset brainstorms

When triggered by frustration indicators ("huge mistake", "wrong approach", "start over", "strip everything"):

1. **Validate the diagnosis before accepting it.** Present what exploration revealed — both what's broken AND what's working — before agreeing with a scorched-earth instinct. The user may be catastrophizing, or the system may be more salvageable than they think.
2. **Focus confidence on "what's actually broken?"** before "what's the fix?" Is the problem structural (architecture, kernel boundary) or tactical (configuration, content, missing inputs)? Is the frustration proportional to the evidence?
3. **Always include a radical option** in Step 3 (strip, delete, rebuild) — even if you recommend a less radical path. Validating the aggressive instinct as a real choice respects the user's judgment.

## Step 3: Propose approaches

Present **2-3 distinct approaches**, each with:

- **Name** — a short descriptive label
- **How it works** — 2-4 sentences
- **Tradeoffs** — what you gain, what you give up

Close with a recommendation: name the chosen option upfront, then justify in 1-3 sentences. The recommendation is one global decision, not a per-approach attribute.

**Sub-variant splitting.** If an option's "How it works" hides two internally distinct paths with materially different cost/risk, split them into sibling options or labeled sub-rows (`C1`, `C2`) rather than bundle — otherwise the user's selection forces an immediate clarification round.

**Comparing concrete artifacts.** When approaches differ in a concrete, comparable artifact (file layout, section set, transformations table, deliverable scope), use `AskUserQuestion` option `preview` fields to render the candidates side by side, so the user compares the actual artifacts rather than labels.

**For product-behavior topics**, annotate each approach with FOUNDATIONS alignment: `Foundations: <principle> @ <surface> (aligns / tensions — reason naming the surface)`. Surfaces include setup/authoring, legal-action generation, validation gate, state transitions, scoring, RNG/determinism, semantic effects, replay/hash, public/private view projection, serialization, bots, effect-log UI/renderer, the `engine-core` boundary, the static-data boundary, and the IP boundary. Naming the surface forces a second-order check that the proposal actually touches what the principle governs. If any approach tensions a FOUNDATIONS principle or trips a §12 stop condition, flag it explicitly and get sign-off — do not recommend a violating approach silently.

**If the problem space is fully constrained** (a reference provides a proven design, or requirements eliminate alternatives), present a single approach with a one-sentence rationale naming the constraint that narrows the space. Don't invent artificial alternatives.

**If the architecture is settled but a scope/density/coverage dimension is the live decision** (common for content-document deliverables — `README.md`, `CLAUDE.md`, config docs — where there is one obvious design and the real question is how much it should contain), present the **tiers along that dimension** as the options (e.g. minimal / standard / comprehensive) rather than forcing distinct architectural approaches. Render the tiers with `AskUserQuestion` `preview` fields per the note above, so the user compares the actual candidate artifacts. This is the natural shape when the "approach" axis is volume, not structure. The same tier mechanism covers a *scope-boundary* decision (which work-classes one deliverable covers versus defers to seeded forward rows), not only volume — for a spec, an explicit count ("create **a** spec") fixes the *number* of specs but leaves that boundary a live tier decision (see `references/triage-and-deliverables.md` §Deliverable classification, Spec, and the groundable-now / seed-forward axis of §Sequential / roadmap-driven brainstorms above).

**For a Port external skill deliverable** (adapting an external skill into this repo), the approach IS a transformations table — per-element strip / replace / preserve decisions, one row per substitution site (not per source line). The **source skill being ported is reference material** — it is the authoritative input for the transformation, not a file read for orientation, so give it the Step 1 sub-step 1 reference summary. Present the table as the approach/design so the user can redirect any row. This inline guidance is self-contained; `references/triage-and-deliverables.md` §Deliverable classification carries the same Port-external-skill rules for cross-reference. **A substitution not itemized in the table is out of scope until itemized** — itemize file drops (e.g. a source `examples/` dir), domain re-mappings, and convention removals as rows, not loose prose. **Row granularity**: one *decision* is one row, not one token — a set of related tokens sharing a single strip/replace decision (a whole stripped domain vocabulary, an entire stripped checklist section) is one row that names the decision and lists the tokens. A large or heterogeneous port may present several grouped tables (strip / replace / preserve / per-co-ported-file) rather than one flat table; the invariant is that every substitution site is itemized and individually redirectable, not that there is exactly one table. **Co-ported dependency files**: a ported skill often depends on co-ported files from the source repo (templates, READMEs, referenced docs the skill hard-depends on); itemize each as its own strip/replace/preserve row — they typically carry the same source-repo residue as the skill itself. This shapes Step 3/Step 4; the Step 5 destination row points back to the same rule.

**If the user challenges a dismissal**, redo the analysis from first principles rather than defending prior reasoning; if it reverses, say where the prior reasoning was incomplete. **If the user clarifies a constraint**, redo the approach proposal under the refined constraint without restarting Step 2 — cite which assumption was sharpened. Brief confirmation suffices when the recommendation still holds; full re-presentation is reserved for clarifications that flip it.

**Wait for the user to choose** (or to ask you to refine/combine) before proceeding.

### Triage brainstorms

When the deliverable is a set of work items — or a single recommended fix — evaluated against a report, finding-set, or diagnostic question, replace the approaches step with a **triage recommendation**: which items warrant action, which are dismissed, and why. The user's approval of the triage serves the same gating purpose as choosing an approach. Render the Step 1 sub-step 8 post-exploration confidence sentence on its own line before the triage recommendation as the audit-trail anchor. **Read `references/triage-and-deliverables.md` §Triage recommendation structure before presenting the triage** (the read fires in the presenting turn, mirroring the Step 5 spec row's mandated read): verdict types, per-item structure, verdict-bucket grouping, the out-of-report and no-source-report cases, and the closing structure live there. **Single-finding carve-out**: for a **single-finding no-source-report diagnostic** (one inline fix keyed `O1`, optionally with dismissed items), the inline guidance in this section already carries the load-bearing structure, so the reference read may be skipped; it stays mandatory for any multi-item triage, any out-of-report finding set, or any verdict beyond accept/dismiss. A "figure out why X" diagnostic that resolves to one inline fix still routes through that reference's §No-source-report diagnostic case — the single answer is the finding, keyed `O1`; a diagnostic yielding several findings keys them `O1`, `O2`, … in that same out-of-report sequence (never a new prefix), and each carries a disposition per the reference's §No-source-report case. A triage recommendation that **cites a governing-doc by section number** — common when the request asks for a "docs-aligned" solution — still owes the Step 4 premise check: grep the section header and confirm the principle's wording before asserting it; a restatement in a script comment, README, or archived deliverable is not verification. This fires even though the triage path bypasses Step 4's section presentation (it mirrors the Sequential-spec path's same rule for `FOUNDATIONS §N` citations).

**Diagnostic-to-design discriminator**: a diagnostic question whose resolution is a *design or insertion-point choice among alternatives* ("where should X live?", "which mechanism enforces Y?", "what fix scope?") routes through the Step 3 **approaches** path, not triage — present the analysis as empirical findings, then the alternatives as 2–3 approaches with a recommendation. Reserve the triage path for a diagnostic that yields a *pre-existing finding/work-item set to disposition* or a single concrete fix. Both paths share the empirical-findings-first presentation, so the split is about the shape of the *answer* (alternatives to choose among vs. items to disposition), not the shape of the question. An approach or scope choice that *expands into* implementation tickets only **after** the user selects it stays on the approaches path — the tickets are a consequence of the pick, not a pre-existing set to disposition — and they then follow `references/triage-and-deliverables.md` §Deliverable classification (Multi-ticket fix set). Don't let the post-selection ticket count pull a diagnose-then-scope-then-decompose request into triage. **Split-disposition (heterogeneous finding set)**: a single diagnostic can yield findings that route differently — some are concrete or already-applied fixes (triage/inline disposition), others carry a live alternative (approaches). Don't force the whole set down one path: present the empirical findings once, disposition the concrete items triage-style (keyed `O<N>` per §Triage brainstorms), and run the approaches path only for the finding(s) with a real choice. The combined deliverable is then a *mixed batch* (`references/triage-and-deliverables.md` §Deliverable classification) — e.g. one inline fix kept under the Step 5 inline-ops carve-out plus tickets for the items needing an approach decision.

**Fused approach + action menu.** When a diagnostic resolves to a recommended approach whose deliverable is unambiguous, the Step 3 approach choice and the Step 6-style next-steps menu may be presented as a **single fused menu** — e.g. options that span both the approach pick (*go with approach B instead*) and the action (*write the ticket / implement now / fold in finding N / done*). This is a sanctioned compression for diagnostic→approaches flows; keep the approach alternatives and the recommended action distinguishable within the one menu, and the HARD-GATE still holds (a fused option that *acts* is still a user choice, not a pre-authorization). When a fused option would execute inline ops / in-place edits on a *non-pre-authorized* request, the menu MUST include an explicit non-acting decline option ("Just explain, don't change") per Step 5 — the user's selection of an acting option over that decline is what carries the HARD-GATE approval.

## Step 4: Present design

**Plan mode**: skip per-section gates. Present key decisions in 1-2 messages with conversation-level checkpoints, then write to the plan file. `ExitPlanMode` carries approval.

**Classification pivot check**: if the design reveals a deliverable type different from the Step 1 classification (e.g. a "docs tweak" now changes validation behavior), re-state the refined classification before presenting Section 1. For a **template-structured deliverable that skips section presentation** (ADR / agent-task ticket / spec — see the Step 4 carve-outs), there is no Section 1, so fire this re-statement **before writing the draft** instead; this is also the hook sub-step 2 relies on when a topic-keyed `product-behavior` announcement resolves to a *plan* artifact (→ dev-process/docs).

Once an approach is chosen, present the design **section by section**, scaling each to its complexity (a sentence for trivial parts, up to ~200 words for nuanced ones). Cover the relevant sections, renaming/combining to fit the topic's natural structure:

1. **Overview** — what this achieves, 1-2 sentences
2. **Architecture / structure** — how the pieces fit together
3. **Key decisions** — important choices and why
4. **Data flow / process** — how information moves
5. **Edge cases** — tricky scenarios and how they're handled
6. **Testing / verification** — how to confirm it works (unit, rule, golden trace, property, simulation, replay, serialization, visibility/no-leak, bot legality, UI smoke, benchmark — per `docs/TESTING-REPLAY-BENCHMARKING.md`). When the deliverable is a visual asset, verification includes rendering it at its smallest real display size (e.g. 16 px for a favicon) and confirming legibility — include the screenshot evidence in the design presentation or summary.
7. **FOUNDATIONS alignment** — for product-behavior topics, a 3-5 row table: principle (`§N <name>`), stance (`aligns` / `tensions` / `N/A`), and a one-line rationale citing the specific design mechanism and the surface it operates at. List only principles the design actively engages or tensions, plus any relevant §12 stop condition the design must clear. Omit for dev-process/tooling/docs topics unless the design touches a FOUNDATIONS principle.

**After each section**, ask "Does this section look right?" and wait for confirmation. If the user pushes back, revise before continuing. After 3 consecutive approvals with no pushback (2 under auto mode), batch remaining sections in groups of 2-3 — announce the batching. Keep a section standalone if it's substantially more complex or higher-risk.

**Carve-outs** (see `references/triage-and-deliverables.md` §Design-presentation carve-outs for detail):
- **Small deliverable** (≤4 decisions, ≥85% confidence): present as a single structured artifact approved in one turn. Announce the consolidation outside auto mode.
- **Template-structured deliverable** (an ADR, an agent-task ticket, a spec with the `specs/README.md` canonical section set, or a skill file with its own canonical sections): the template provides the navigation; present the full draft as one artifact, one approval covering it — and under §User pre-authorization that approval is the recommendation presentation, so the draft may be written directly in the same turn. When the deliverable's Step 5 row mandates a reference read ("**read** … before authoring"), that read fires in the **authoring turn**: classification and authoring often land in different turns, so re-check before writing that the row's mandated read has happened this session — if not, perform it now. **In an interactive turn** — where no collapsed-turn checklist enforces it — state which mandated reference read you performed before writing, so the gate is visible rather than silently skippable; on the sequential spec path the predecessor-template read (Step 1 sub-step 7) substantially overlaps this read, so the two are complementary, not independently-droppable gates. Because this path skips section presentation, the Classification pivot check has no "Section 1" to precede — so if the artifact's nature differs from the Step 1 topic-keyed classification (a `product-behavior` diagnosis resolving to a *plan* artifact → dev-process/docs), **re-state the refined classification before writing the draft**. In a collapsed pre-authorized turn, where mid-turn text may not render, this re-statement lands in the final message — it is a checklist item there (§Guardrails §User pre-authorization).
- **Multi-deliverable triage**: triage approval covers all N deliverables together; each is then approved as a single template-structured draft (no per-deliverable gate).

**Verify operator-introduced design premises before persisting them.** When the design names a specific field, file path, schema shape, crate/module, trace format, or build/test/bench command from the codebase — or cites a governing-doc principle by section number (`FOUNDATIONS §N`, an architecture doc's §-numbered rule) — verify it exists (a quick Read or grep; for a §-citation, grep the section header and confirm the principle's wording matches the claim — a restatement in a README or archived deliverable is not verification; a same-session full Read of the governing doc itself discharges the per-citation grep, since its wording is then verifiably in context) before writing it into the deliverable — the 30-second check prevents a premise falsifying during implementation. For removals, merges, renames, or moves, enumerate ALL references with a broad grep before drafting the files-to-touch and acceptance criteria. When the design **reuses** an existing tool, script, or codepath for a *new input* (a new game, variant, or data shape), "exists" is insufficient — verify it actually accepts that input (a quick run, or a read of its input-parsing path) before quoting a cost/scope estimate; a parser or codepath hardcoded to the original input is a common falsifier, and the unverified reuse premise silently inflates a "minimal change" estimate. **Delegated verification**: a subagent's summarized report discharges this check only when it cites the concrete evidence (registry/file contents, file:line); when two pieces of exploration evidence about the same load-bearing premise conflict (e.g. an agent reports a tool both lacks a game's registration and gates on it in CI), reconcile by direct Read/grep before the premise lands in the deliverable — never ship the claim with the contradiction unresolved. **Verified-by-predecessor and batching**: a path or command quoted verbatim from the immediately-preceding archived deliverable's *completed Outcome* — one whose final verification transcript passed (e.g. the prior gate's archived spec) — counts as verified without a fresh grep; a premise that is *new or extended* relative to that predecessor (a pattern extension, a new registration, changed semantics, a new input to an existing tool) always needs its own fresh Read/grep, and many small premises against the same file may be discharged by one batched grep. (This does not relax the §-citation rule above: archived restatements never verify a principle's wording.) **For in-place-edit deliverables, a grep does not let you edit**: the premise check above may use grep, but the `Edit` tool requires the target file to have been opened with the `Read` tool first — a grep/Bash/`sed -n` view, or in-context content from an earlier session, does not satisfy the Edit validator. `Read` each file you will edit before editing it (mirrors `skill-audit`'s Read-before-Edit step), or the first `Edit` fails with "File has not been read yet."

## Step 5: Persist the deliverable

After design approval, do NOT implement until the user picks an implementation option from the Step 6 menu — the design doc is typically the deliverable; implementation is a separate, user-chosen act. Exception: inline ops/setup and in-place edits are executed during the session (the HARD-GATE still requires approval first), and no separate design doc is written. **Obtaining that approval for an inline-ops/in-place-edit triage on a *non-pre-authorized* request** (a diagnostic question, not a build directive): the HARD-GATE's explicit approval may be carried by the same `AskUserQuestion` that selects the fix's parameters (pin mode, scope, variant) — include an explicit decline option (e.g. "Just explain, don't change") so the user's selection of an acting option over the decline *is* the approval, with no separate round-trip. That same `AskUserQuestion` may also offer a **verify-depth variant** — e.g. "apply + run the project's verification gates" vs. "apply only" alongside the decline — so verification becomes a first-class user choice rather than an ad-hoc afterthought (this honors CLAUDE.md (global) §Pre-Completion Verification). This is the inline-ops counterpart of the §Triage "user's approval of the triage serves the same gating purpose as choosing an approach" rule.

An explicit user-named destination overrides this table; when it creates a new directory or sits outside the repo's tracked workflows (`specs/`, `tickets/`, `docs/`), record that placement and any index/workflow non-obligations as a named assumption.

When persisting ≥3 files (e.g. a dependency-linked ticket set), track one task per file *for interruptible or long-running multi-file work* so progress stays visible — see `references/triage-and-deliverables.md` §Deliverable classification. A short batch of fast, atomic `Write`s whose success confirmations already surface progress may skip the task list; note that `TaskCreate`/`TaskUpdate` are deferred tools requiring a `ToolSearch` load first, so weigh that overhead against the visibility gained. Never commit a deliverable; leave it for user review.

**Quick triage of deliverable shape → destination** (full per-type rules in `references/triage-and-deliverables.md` §Deliverable classification):

| Deliverable shape | Destination |
|---|---|
| Inline ops/setup task or mechanical-fix batch | execute inline + summary; no file persisted |
| New skill design | `.claude/skills/<name>/SKILL.md` |
| Modify existing skill file(s) | edit in place |
| Project documentation & root instruction/config files (`README.md`, `docs/*.md`, `templates/*.md`, `CLAUDE.md`, `AGENTS.md`) | edit/create in place |
| New dev-tooling/CI/config file (`.github/workflows/*`, lint/format/build config) | create in place |
| Architectural decision, or amend/supersede an accepted ADR's doctrine or `docs/FOUNDATIONS.md` (constitutional change) | `docs/adr/NNNN-<slug>.md` from `docs/adr/ADR-TEMPLATE.md` — flag a constitution supersession explicitly, require sign-off |
| Port external skill into repo | new skill file(s) (`SKILL.md` + any `references/`) + delete source + a transformations table itemizing per-element strip/replace/preserve (see the Step 3 Port-external-skill note + `references/triage-and-deliverables.md` §Deliverable classification) |
| Replaces an existing artifact | new file + delete old + update cross-references |
| Implementation work item (agent-task ticket) | filled `tickets/_TEMPLATE.md` → `tickets/<PREFIX>-NNN.md` (decomposing a spec: `/reassess-spec` then `/spec-to-tickets`; **direct, non-spec authoring**: derive `<PREFIX>`/`NNN` per `references/triage-and-deliverables.md` §Deliverable classification — continue an existing series only when the work is the same topic and the series is still open, else coin a fresh prefix starting at `001`; when `tickets/` holds only `README.md`/`_TEMPLATE.md` the active series is likely archived, so recover its prefix and highest number from `git log` (or the governing gate) before deciding continue-vs-greenfield, and consult that reference's ID-assignment rule when an archived series may still be open and on-topic; for ≥2 tickets (dependency-linked or independent), present one combined Step 6 menu, ordering by `Deps` when dependencies exist). **Before writing**, if Step 1 announced topic-keyed `product-behavior`, re-state `Classification: dev-process/docs (agent-task ticket)` per the Step 4 pivot check |
| Decision + its implementing tickets (a fix needing both a recorded decision and the work items that carry it out) | ADR first (`docs/adr/NNNN-<slug>.md`), then tickets that cite it in `Deps`; encode inter-ticket dependencies; one combined Step 6 menu — see `references/triage-and-deliverables.md` §Deliverable classification |
| Spec (roadmap-gate / stage implementation plan) | **Read `references/triage-and-deliverables.md` §Deliverable classification before authoring** (the read fires in the authoring turn — see the Step 4 template-structured carve-out; this read governs **roadmap/sequential specs too**, not only triage-derived ones — the reference filename understates its scope, and the predecessor structural-template read does not discharge it). Then: `specs/<gate-or-stage>.md` + maintain the `specs/README.md` index (progress tracker); the spec IS the design (skip the `docs/plans/` doc); the FOUNDATIONS-alignment table's `§N` citations are operator-introduced premises — grep each cited section header in `docs/FOUNDATIONS.md` and confirm its wording before writing the table (Step 4; archived or contract restatements don't verify a principle's wording, though a same-session full Read of `docs/FOUNDATIONS.md` discharges the per-citation grep); after writing the spec and flipping the index row, run the repo's doc-link checker (`node scripts/check-doc-links.mjs`) before the menu; its Step 6 menu offers reassess-and-decompose (`/reassess-spec` → `/spec-to-tickets`) / refine / done (no direct-implement option — the repo workflow decomposes an accepted spec into tickets before execution) |
| Triage → ≥2 ADRs / ≥3 agent tasks | deliverables + `docs/triage/YYYY-MM-DD-<topic>-triage.md` |
| Triage analysis, all deliverables deferred | `docs/triage/YYYY-MM-DD-<topic>-triage.md` (full verdict content) |
| Design doc (default) | `docs/plans/YYYY-MM-DD-<topic>-design.md` |
| Research brief for an external researcher | `reports/<topic>-research-brief.md` (the `research-brief` skill owns this directory) |

A default design doc consolidates all approved sections into a clean document with a "Brainstorm Context" header (original request, reference file if any, key decisions that shaped the design, final confidence and assumptions). ADRs and agent tasks skip the design doc — the ADR/task IS the design. An **ADR** follows `docs/adr/ADR-TEMPLATE.md`'s section set (Context, Decision, Alternatives considered, Consequences, Determinism / Replay-hash / Visibility / Data-Rust-boundary / `engine-core`-contamination / UI / Bot / IP / Benchmark impact, Migration notes, Review checklist); assign its ID by scanning `docs/adr/` for the highest existing `NNNN-` filename prefix and claiming the next integer, zero-padded to four digits (first is `0001`; the filename carries no `ADR-` prefix; `ADR-TEMPLATE.md` is not numbered). An **agent-task ticket** fills `tickets/_TEMPLATE.md` (Problem, Assumption Reassessment, Architecture Check, Verification Layers, What to Change, Files to Touch, Out of Scope, Acceptance Criteria, Test Plan) following the `tickets/README.md` authoring contract — do not invent a different structure when the template exists. Decomposing a spec into tickets routes through `/reassess-spec` (assumption reassessment), then `/spec-to-tickets` (one ticket per reviewable diff).

If working in a git worktree, resolve every output path against the **worktree root**, not the main repo root, and confirm the file landed there before the Step 6 menu.

Do NOT commit the file. Leave it for user review.

## Step 6: Next-steps menu

| Deliverable category | Step 6 action |
|---|---|
| Plan mode active (any category) | Call `ExitPlanMode` — skip the menu |
| ADR / agent task / triage / proposal / spec — or any deliverable whose options describe NEXT phases (decompose / implement / done) | **Menu MANDATORY**, even when written this same turn — writing the file is pre-implementation, and skipping the menu strips the user's named next-phase options |
| Inline-completion (skill design; port external skill; small inline ops/setup; new dev-tooling/CI/config files; new or in-place-edited project-doc / root-instruction files — `README.md`, `docs/*.md`, `templates/*.md`, `CLAUDE.md`, `AGENTS.md`; in-place edits to existing ADR/agent-task/docs where the edit IS the completed work) | Skip the menu; summarize what was done |

Categorize by the middle row first; the inline-completion skip fires only when the deliverable falls outside it. For a mandatory menu, present options like:

```
Design doc written to docs/plans/YYYY-MM-DD-<topic>-design.md

What would you like to do next?
1. Record the decision as an ADR (write to docs/adr/)
2. Decompose into tickets — spec-decomposition only (`/reassess-spec`, then `/spec-to-tickets` → `tickets/<PREFIX>-NNN.md`)
3. Start implementing directly
4. Done for now — I'll review the design doc later
```

Adjust options to the deliverable: for an ADR, offer decompose-into-agent-tasks / implement / done; for agent tasks, offer begin-first-task / refine / done; for an ADR-plus-implementing-tickets set, present one combined menu covering the set (record/accept the decision / refine / implement the first ticket / done) rather than a separate menu per file. For a **directly-authored** ticket set (approaches path, no source spec), the menu is begin-first-task / refine / done — the `/reassess-spec`→`/spec-to-tickets` decompose option does **not** apply and must be omitted; offer a *validate-assumptions-before-implementing* option in its place when the tickets carry unconfirmed Assumption-Reassessment items. Use `AskUserQuestion` when its schema is available; inline numbered options are an acceptable fallback. Under autonomous/headless operation (a harness directive that the user is not watching in real time and blocking questions stall the work), prefer the inline numbered menu in the final message over a blocking `AskUserQuestion` — the menu content requirement is unchanged. If the user picks an option that invokes another skill, invoke it. If they pick "done," end.

**If a Step 6 option spawns a follow-up brainstorm cycle for related scope** (a next agent task alongside the just-written one, a related ADR the menu surfaced — or **the user revisits an item the prior triage dismissed or flagged out of scope**, which is a related-scope follow-up even though it was never a menu option; all distinct from re-triaging the prior verdicts): re-enter at Step 1, not Step 2; emit the Step 1 sub-step 8 post-exploration confidence anchor for the new cycle (even when exploration is trivial, and equally inside a pre-authorized collapsed turn — see §Guardrails §User pre-authorization's final-message contents); the prior cycle's context typically elevates initial confidence so the interview-skip threshold applies.

## Guardrails

- **YAGNI ruthlessly.** Strip optional extras from every design unless the user explicitly asked for them.
- **No scope inflation.** The design covers what was asked. Resist "while we're at it" improvements.
- **One question at a time** during the Step 2 discovery interview. (Does not restrict `AskUserQuestion` batching outside the loop.)
- **No implementation before approval.** The HARD-GATE means what it says, subject only to the §User pre-authorization carve-out.
- **FOUNDATIONS is authoritative.** For product-behavior topics, if an approach violates a FOUNDATIONS principle or trips a §12 stop condition, flag it immediately and get explicit sign-off. Never design against the constitution silently; a feature that genuinely needs to diverge requires an ADR superseding that principle first (FOUNDATIONS: "Supersede only by accepted ADR").
- **Respect early exit.** If the user wants to skip ahead, let them — list your assumptions clearly.
- **Worktree discipline.** If working in a worktree, all file paths use the worktree root.
- **Mandated search tooling.** When a session hook/directive mandates a search tool (e.g. `mgrep`), route *all* searches through it — including ad-hoc Bash `grep`/`rg` used for premise verification, not just the Grep/Glob/Explore tools — falling back to the named tools only on hard failure (see sub-step 6 for the full substitution-and-degradation rule).
- **Blocker discovery during implementation.** If implementation reveals an architectural issue or a cost/scope re-estimate that blocks the deliverable or invalidates the approach ranking, present the blocker, options, and a recommendation. If the fix is small (< 50 lines, single file, no kernel-boundary or public-interface change — a *backward-compatible* generalization where no existing caller breaks and a regression test covers the new path does not count as a "public-interface change" here, whereas a signature or contract change that breaks existing callers does), fix it inline and continue; otherwise create an agent task and either work around it with a documented constraint or implement the fix with user approval. **Pre-write case:** when the blocker is surfaced not during implementation but during a Step 4/5 verification *before* any deliverable is written, and its fix would add a deliverable of a new class or cross the deliverable count, the §User-pre-authorization shape-confirmation rule takes precedence over this fix-inline allowance — pose a one-shot `AskUserQuestion` confirming the expanded shape before writing, even under pre-authorization, however small the fix.
- **Auto mode** applies when a system reminder instructs continuous operation, the user issues an explicit "just go" directive, or a harness `auto_mode` flag is set. Under auto mode: clarifying questions are minimized (gaps become named assumptions), the Step 4 batching threshold drops from 3 to 2 consecutive approvals, and small/template-structured deliverables may be written in the same turn as approval. Approach-selection and section-approval gates still hold, and the HARD-GATE is never relaxed.
- **User pre-authorization.** When the request pre-authorizes a deliverable contingent on the verdict ("write an ADR if warranted", "implement if changes are needed") — or issues an unconditional build/port/"adapt" directive naming a specific deliverable ("adapt this skill into a repo-local one", "port X", "build Y"), which is an even stronger authorization than the contingent case — the HARD-GATE is satisfied by presenting the design or triage recommendation (for a Port external skill, the transformations table) — no separate approval round-trip. Pre-authorization satisfies the HARD-GATE; it does NOT skip the Step 6 menu. The user can still redirect between presentation and write by issuing a counter-instruction. (For how this presentation/write sequencing applies to *triage* brainstorms, whose approval model differs, see `references/triage-and-deliverables.md` §Design-presentation carve-outs, Non-plan-mode fast-track.) Under autonomous/headless operation there is no "between" — mid-turn text may never render, so presentation and write collapse into one turn whose **final message must carry the complete triage/design** (findings, finding→deliverable mapping, the post-exploration confidence anchor — sub-step 8, required in every cycle including Step 6 re-entry cycles — and the named assumptions in `(N) X — assuming Y` form; for a template-structured deliverable, enumerating the template's assumption IDs each with a one-line hook satisfies the assumptions requirement) so post-hoc redirection stays cheap; the redirect window then follows the message rather than preceding the write. In a fully collapsed turn, the final-message anchor is also the sole required emission of the sub-step 5/8 confidence announcements — mid-turn emission is optional narration, not a separate obligation. When a remaining gap would materially change the deliverable shape (count crossing a previously-stated boundary — 1↔multi, or N↔N+ when the addition is a new class — type changing class, or a new deliverable file becoming required), prefer `AskUserQuestion` to confirm the shape before writing even under pre-authorization. **Reversible-trivial carve-out**: a 1↔multi expansion whose added unit is **fully reversible, already verified, and mechanically trivial** (e.g. one more line wired into a CI step, both variants confirmed passing) may instead proceed with an explicit redirect flag in the final message — naming the added unit and inviting a pull-back — in lieu of a blocking `AskUserQuestion`. This rule takes precedence over the §Guardrails "Blocker discovery" fix-inline allowance when a Step 4/5 pre-write verification surfaces the blocker (see that bullet). **User-named type-fork carve-out**: when the pre-authorizing request itself enumerates the deliverable-type (class) choice ("a spec or tickets", "agent task or ADR"), the `references/triage-and-deliverables.md` §Deliverable pivot rule governs in place of the prefer-`AskUserQuestion` default — select by exploration scope evidence without re-prompting, naming the scope basis (and any clearly mis-sized class, e.g. a roadmap-gate spec for a two-line shared-CSS fix) in the deliverable lead; this is the type-axis analogue of the Reversible-trivial carve-out. A genuinely open class choice the user did *not* enumerate still prefers `AskUserQuestion`. **Exception:** if the recommendation would reverse a decision the user previously committed to (an accepted ADR, a `docs/triage/` decision record, a deliberately-rejected item), surface the reversal prominently and pose a direction-confirmation `AskUserQuestion` before writing — pre-authorization does not extend to silently overturning a prior user decision. **Amendment counterpart:** distinct from reversal, when *authoring* a pre-authorized **ticket** (or config edit) reveals it must **amend or clarify** the doctrine fixed by an accepted ADR (a §13 ADR trigger — see Step 1 sub-step 3), that amendment is its own ADR-first deliverable: split it out per the `references/triage-and-deliverables.md` §Deliverable classification *Decision + implementing tickets* shape (write the amending ADR first, then the ticket citing it in `Deps`) and re-confirm the expanded shape via `AskUserQuestion` before writing — do not bundle the ADR edit as a mere ticket step, which is the silent weakening sub-step 3 forbids.
  **Collapsed-turn final-message checklist** — the prose above is the rationale; this list is the compliance surface. Under autonomous pre-authorization the final message includes each of:
  - the empirical findings and the finding→deliverable mapping (for sequential/authoring cycles with no finding-set, "findings" = the unit-selection evidence — index row, interlock state — and "mapping" = the deliverable summary);
  - the post-exploration confidence anchor, restated (Step 1 sub-step 8) — mid-turn emission alone does not satisfy it;
  - named assumptions in `(N) X — assuming Y` form (or the template's assumption IDs, each with a one-line hook);
  - alternatives considered for any approach-level gap carried as an assumption (sub-step 5 precedence note);
  - any skip notes the flow incurred (e.g. sub-step 6's research skip or degraded-research note);
  - confirmation that the deliverable row's — or triage path's — mandated reference read (if any) was performed this session and survives in context (re-perform after compaction; matches the Step 4 template-structured carve-out's window) — name the file read, or perform it before writing;
  - the refined classification re-statement, when the artifact's class differs from the Step 1 topic-keyed announcement (satisfies the Step 4 pivot check's pre-write re-statement in collapsed turns, where mid-turn text may not render).
