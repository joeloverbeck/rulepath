# Decomposition Patterns (Step 3)

Analyze the spec and identify discrete work units:

- Each ticket is a **reviewable diff** — small enough for comfortable manual review.
- Map **dependencies** between tickets and name them in each ticket's `Deps`.
- Determine **priority ordering** from the dependency graph and criticality.
- Ensure **every spec deliverable is covered** — no silent skipping. If a deliverable seems wrong, flag it with the 1-problem / 3-options / 1-recommendation format rather than omitting it.
- Use the spec's **Out of Scope** / **Forbidden changes** sections to populate each ticket's Out of Scope — these are pre-validated non-goals.

## Deliverable-coverage categories

Most deliverables get their own ticket. These categories are exempt from per-deliverable ticket production — document them in the Step 4 summary and Step 6 coverage mapping instead:

- **No-change deliverable** — the spec explicitly states no change is needed ("No new crate"). Note it if non-obvious; produce no ticket.
- **Cross-spec consumer-side deliverable** — the work is consumed by a sibling gate's tickets rather than producing a discrete output here. The change is needed, but it happens in another spec's decomposition. Note the ownership as `cross-spec: <sibling-gate>` in Step 4 and `not ticketed (cross-spec: …)` in Step 6.
- **Distributed deliverable** — one conceptual deliverable encoded as a contract distributed across multiple tickets (e.g. a per-game applicability rule carried as a predicate on each affected module, with no single owner). Note it as `distributed across <ticket-list>`; produce no separate "container" ticket.
- **Informational-pointer deliverable** — a section whose body only points to sibling deliverables ("handled in §X above and §Y below"). The actual work is in the pointed-to siblings, each ticketed. Note the routing; produce no ticket.
- **Already-landed deliverable** — the intent was satisfied by an edit landed during a prior in-session skill invocation (e.g. `/reassess-spec`'s user-authorized in-session edit). Produce no ticket; record `not ticketed (already landed during <skill> session <date>)` in Step 6.
- **Inferred deliverable** — named or implied by §Objective / §Acceptance evidence / §Exit criteria but absent from §Deliverables. This DOES get a ticket; document the inference in the ticket's Assumption Reassessment with a citation to the motivating section, and flag it in Step 4 Notes as `inferred from <section>`. When the motivating reference is ambiguous or would expand scope, route through a Step 2 Issue instead.
- **Process / no-artifact work-item** — a Work-breakdown item that produces no reviewable diff: an intake/orientation step, "reassess assumptions against current code," "confirm all referenced paths," or similar. Its substance is discharged by every ticket's Assumption Reassessment section, not by a discrete output. Produce no ticket; record `not ticketed (process step — folded into per-ticket Assumption Reassessment)` in Step 6. (Common as the leading WB row, e.g. a `G<gate>-00 intake` item.)

## Sizing rules

- **Merge** — when multiple deliverables share the same file set and cannot be implemented independently, merge into one ticket. Note merged deliverables in Step 4 Notes.
- **Split** — when a single deliverable is too large for one reviewable diff (it spans, say, a contract surface + a schema + a validation rule + a bot emitter), split by logical sub-concern (e.g. write path vs. read/verify path). Note the split in Step 4 and show `D<n>→00x+00y` in Step 6.
- **Candidate-AGENT-TASK granularity is a floor, not a 1:1 mapping** — when the spec's Work-breakdown table labels each item as a candidate AGENT-TASK (e.g. a `Becomes AGENT-TASK: yes` column), treat that as the author's intended *granularity floor* — the coarsest acceptable grouping — not a binding one-ticket-per-WB-item rule. Honor it as the default, but apply the Split rule when a labeled item exceeds one reviewable diff (the highest-risk surface — `engine-core` kernel growth, a full `games/*` crate — usually does). Surface every divergence from the spec's item count in the Step 4 table with `D<n>→00x+00y`, and offer the coarser-grained alternative ticket count so the user can opt back down to the spec's granularity. **Coarse "workstream"-shaped Work breakdowns**: some specs (e.g. a `Workstream | Required outcome | Hard constraints` table) enumerate broad workstreams with no per-item AGENT-TASK label, and the natural decomposition — by the `docs/OFFICIAL-GAME-CONTRACT.md` pipeline or by logical sub-concern — is both finer than and orthogonal to those rows. Treat the workstream count as a soft lower bound (not a 1:1 target), decompose by sub-concern, and surface the fan-out in the Step 4 table (e.g. `9 workstreams → 16 tickets`) with the coarser-grained alternative offered, the same way the labeled-column case surfaces `D<n>→00x+00y`. **Spec-supplied candidate ticket IDs**: when a spec's Work breakdown pre-labels candidate ticket IDs in a prefix different from the run namespace (e.g. §14 lists `RULSUR-001…006` but the run namespace is `RULDISSHASUR`), treat those IDs as granularity-floor candidates only — renumber into the requested namespace and map the spec's IDs to the produced tickets in the Step 6 coverage mapping (e.g. `RULSUR-002 → 003+004`); never adopt the spec's prefix into the produced ticket filenames. **Same-prefix collision after a split/merge**: when the spec pre-labels candidates in the *same* prefix as the run namespace (e.g. §15 lists `VICEXPSHASUR-001…008` and the run namespace is `VICEXPSHASUR`) AND a split or merge shifts the spec's own later numbers off the produced files (splitting the spec's `-004` into produced `004…008` makes the spec's `-005` mean a *different* ticket than the produced `-005`), do NOT key the Step 6 coverage map by the spec's ticket IDs — they now collide in meaning with the produced filenames. Key the map by the spec's **section identifier** instead (`§15.4 → 004+005+006+007+008`, `§15.5 → 009`), per the Step 6 item-3 "key by the spec's own identifiers" rule, and flag in Step 4/Step 6 that the spec's `PREFIX-NNN` and the produced `PREFIX-NNN` diverge past the split point so a reviewer does not conflate them.
- **Same file / same new directory** — when all deliverables modify the same file or land under one new directory (a new crate, a new `games/*` dir), decompose by logical section/module, not by file boundary. Each ticket targeting a different section is a valid reviewable diff. Note the shared file once in Step 4.
- **Intra-batch create-then-modify chains** — when ticket B's Files to Touch contains a `(modify)` entry for a file ticket A creates `(new)` in the same batch, declare `Deps: A` on B explicitly. The file-creation is a load-bearing structural dependency (B fails at implementation time if A hasn't landed). This is distinct from §Shared-file overlaps (Step 6 item 5), which covers parallel adds by mutually-independent tickets to a shared file — whether pre-existing or created in-batch by a common foundation ticket they each `Deps:` on.
- **Greenfield foundation batch** — when the spec is the repo's first code (no pre-existing code tree, e.g. the Gate 0 repository-skeleton spec), every ticket's Files to Touch is legitimately all-`(new)`; do not treat that as a suspicious under-statement. The structural ordering is carried by intra-batch create-then-modify chains (above), and the deliverables' consumers are the sequenced later gates, not present code — mirroring the greenfield note in `reassess-spec`'s `references/codebase-validation.md`. **New crate / new `games/*` dir in an existing repo**: the same all-`(new)` legitimacy applies — a gate that adds a new crate or game directory will have most of its Files to Touch be `(new)` under that dir, with ordering carried by the crate-skeleton ticket (which creates the dir) plus intra-batch create-then-modify chains, not by pre-existing files. Do not treat a new-game gate's all-`(new)` tickets as under-statements.

## Ticket shapes

**Trailing-shape co-occurrence**: a single spec closeout deliverable often bundles docs-finalization **and** exit-evidence; split it into both shapes below — a cross-cutting docs ticket (Deps = the surface-citing implementation tickets) plus a capstone (Deps = the leaf set, including the docs ticket). The `Done`-status flip (spec Status → `Done` and the `specs/README.md` index row) defaults to the **capstone**, since it is gated on exit evidence passing; the docs ticket carries doc content and any earlier `Planned` update, not the completion flip — unless the spec's §Documentation-updates assigns the index flip to the docs ticket, in which case the §Cross-cutting docs ticket `Deps: <capstone>` exception applies.

### Capstone integration ticket

A single trailing ticket whose scope IS the spec's §Exit criteria / §Acceptance evidence, exercising every prior implementation ticket end-to-end. It introduces no new production logic; it exercises the pipeline the earlier tickets composed.

**Deliverable-doubles-as-capstone**: the "no new production logic" constraint applies to *acceptance-only* capstones. When the gate's own verification harness is itself a deliverable (e.g. a browser-E2E smoke ticket that adds a Puppeteer dependency + harness code), that ticket MAY serve as the capstone even though it ships new test infrastructure — note the dual role in the Step 4 table. And when no single ticket exercises the whole gate end-to-end, acceptance MAY be distributed across the trailing tickets (e.g. ABI smoke + browser smoke + a11y/no-leak + docs/status-flip); record the distribution in the Step 6 deliverable-coverage mapping instead of forcing a synthetic capstone.

- Its acceptance criteria enumerate the spec's exit-criteria / acceptance-evidence bullets as test sub-cases.
- Re-enumerate expected counts from a fixture at test start rather than hardcoding (hardcoded counts go stale).
- **Files to Touch** — `_TEMPLATE.md` makes the section mandatory, but a capstone introduces no production logic, so it is one of: the new e2e/smoke/golden-trace test the ticket adds; `None — verification-only` when the gate is exercised purely by running existing scripts (simulation CLI, replay-check) plus a manual runbook; or the **status-reconciliation surfaces** the §Ticket shapes `Done`-flip default assigns to the capstone (the `specs/README.md` index row, the spec's own Status, the `docs/MECHANIC-ATLAS.md` first-use / atlas row, `progress.md` / root `README.md` — but **excluding** the root `README.md` "current official games are" catalog list, which for a web-exposed game gate co-lands with the web-smoke ticket per §Validator-consumed catalog docs, never the capstone) when no separate cross-cutting docs ticket owns them — a capstone that performs status reconciliation legitimately lists those docs. Do NOT list the *upstream tickets'* files — the capstone exercises them, it does not modify them (the status-reconciliation surfaces above are the capstone's own, distinct from upstream files).
- **Manual-runbook variant** — when an exit-criterion cannot run in the project's test infra — either it requires invoking an LLM-driven skill (not runnable from test code), **or** it is a UI/manual smoke with no browser-automation harness in the project (the WASM web shell) — structure the ticket as a hybrid: a *manual runbook* (in the What to Change section) the implementer follows by hand — for an LLM skill: fixture copy strategy, skill invocations against the copy, expected post-skill state with verification commands; for a UI smoke: the click-path with each step's expected observable result — plus an automated-test subsection for any test-runnable portion (golden-trace assertions, replay/hash checks, no-leak negative tests, bot-legality checks). The acceptance criteria distinguish CI-runnable assertions from implementer-checklist runbook steps.
- **`Deps`** — prefer a single `Deps: <transitive-head-ticket>` when the upstream DAG is a linear chain (the head's own `Deps` reach every prior ticket). When the upstream DAG has parallel branches (e.g. a docs ticket running parallel to implementation tickets), enumerate the **leaf set** — every leaf whose transitive `Deps` collectively cover the full chain.

### Cross-cutting docs ticket

A single trailing ticket whose Files to Touch are markdown docs (READMEs, architectural docs, skill reference files, `specs/README.md` index updates — see the §Ticket shapes `Done`-flip default) that must land atomically once all upstream implementation tickets ship.

- One Files-to-Touch entry per docs surface the deliverable names.
- Acceptance criteria as grep-proofs against the post-implementation tree (exact-string matches for new/renamed symbols, count assertions for status-line updates).
- Out of Scope names the implementation surfaces this ticket does NOT touch, so reviewers know docs-only is the contract.
- **`Deps`** — list every implementation ticket whose surface the docs reference (docs reference each surface independently, so transitive-head is insufficient here). **Exception** — when the docs ticket only annotates *aggregate completion* (a `specs/README.md` status flip / "verified by Gate N" note) rather than citing individual symbols, and a capstone ticket already gates every referenced surface, `Deps: <capstone>` is sufficient and more legible than re-enumerating the surface set. Enumerate per-surface only when the docs cite individual symbols/APIs that could go stale independently.
- **When NOT to use** — if each docs change can co-locate with its implementing ticket without a staleness window, per-ticket docs are cleaner. Use this shape when ≥1 docs surface requires all upstream surfaces to exist coherently before it can land (status-line counts, cross-tool tables, multi-surface skill guidance).
- **Tool-validated docs** — a game doc that a validator *tool* consumes (in Rulepath, `tools/rule-coverage` reads `RULES.md` / `RULE-COVERAGE.md` / `BENCHMARKS.md`) MUST co-land with (or before) that tool's registration ticket, not in a trailing docs-only ticket — otherwise the tool registration has nothing valid to check. Split the official-game doc set by *which ticket validates each doc* (rules docs front-loaded per OGC §3; `RULE-COVERAGE.md` with the rule-coverage registration; `BENCHMARKS.md` with the benches; bot-strategy docs ordered relative to the bot whose evidence they document — before a Level 2 policy, or trailing and depending on the bot ticket when an evidence pack documents a Level 1 bot's fixtures; the rest trailing), not by a blanket docs-vs-code boundary. **Cross-ticket dependency when one validator reads docs split across tickets**: `tools/rule-coverage` validates `RULES.md` + `RULE-COVERAGE.md` + `BENCHMARKS.md`, but the placement above puts `BENCHMARKS.md` with the benches ticket — a *different* ticket from the rule-coverage registration. So the rule-coverage registration ticket's fully-green `rule-coverage --game <g>` state depends on the benches ticket having landed `BENCHMARKS.md`. Either declare `Deps:` from the rule-coverage ticket onto the benches ticket, or flag the expected partial-green window in Step 4 / Step 6 (mirroring the `check-catalog-docs` red-window note), so a reviewer running `rule-coverage` before the benches ticket lands isn't surprised by an incomplete coverage report.
- **Validator-consumed *catalog* docs (web-exposed game gates)** — `scripts/check-catalog-docs.mjs` (run in `gate-1-game-smoke.yml`) is a second validator-consumed-docs surface, keyed off the `crates/wasm-api/src/lib.rs` `GAME_*` catalog const as source of truth. It mechanically checks three README surfaces: the `apps/web/README.md` intro catalog list, the root `README.md` "current official games are" list, and the `apps/web/README.md` Smoke Layers `smoke:e2e` bullet (the `apps/web/README.md` Shell Surface renderer bullet is process-enforced, not mechanically checked). For a web-exposed game gate these three surfaces MUST co-land with the web-smoke / gate-1-CI ticket that wires `smoke:e2e` (and references the new `GAME_*` const) — NOT in a trailing docs-only ticket or the capstone — or the catalog README drift gets deferred to an aftermath pass (the Gate 9 `token_bazaar` aftermath was exactly this miss). Note the expected red-CI window: because `check-catalog-docs` runs continuously and keys on the wasm-api catalog const, every PR between the WASM-registration ticket (adds the const) and the README-reconciliation ticket shows a red `check-catalog-docs` step — flag it in Step 4 / Step 6 as an expected mid-gate state resolved at the web-smoke ticket, so a reviewer reading PRs in order isn't surprised.

### Conditional / decision-gated ticket

A ticket worked only if an upstream **decision ticket** resolves a certain way — e.g. a `game-stdlib` helper extraction worked only if a §4 primitive-pressure ledger decides *promote*, or a migration step worked only if an investigation confirms a root cause. Coverage is not silently skipped: the deliverable still gets a ticket, with its conditionality made explicit.

- **`Deps`** — depend on the deciding ticket (the one that records the branch outcome).
- **Problem / Out of Scope** — state the trigger condition in Problem ("worked only if <decision ticket> decides X") and, in Out of Scope, "proceeding at all if <decision ticket> decided otherwise (this ticket is then not-applicable)."
- **Fallback** — name where the work goes if the condition is *not* met (e.g. "rules-core implements the logic locally instead"), so the alternate path is owned, not orphaned.
- **Step 4 / Step 6** — flag it in the Step 4 table Notes as `CONDITIONAL on <ticket>` and record in Step 6 that it closes not-applicable if the decision goes the other way. A sibling ticket that would *consume* the conditional output (e.g. rules-core using the promoted helper) should NOT hard-`Deps` on the conditional ticket; instead it `Deps` on the stable predecessor and notes the conditional consumption, so its dependency graph stays resolvable whether or not the conditional ticket is worked.

## Official-game-gate pattern

Gates that admit a new official game under `games/*` (the dominant Rulepath gate type per `docs/ROADMAP.md`) share a recurring ~16–22-ticket shape. For a gate that carries the §4 primitive-pressure decision (ledger + atlas update + conditional `game-stdlib` extraction), the `directional_flip` / Gate 6 decomposition (`archive/tickets/GAT6DIRFLI-*`, 21 tickets) is the closest precedent — it is the freshest full board-game gate and the only one whose decomposition includes the primitive-pressure decision + conditional-extraction shape; a gate that *also* introduces a novel boundary (a compound action tree, multi-segment WASM export) trends toward the upper end (~22, e.g. Gate 7 `draughts_lite`). For a **full public-game gate** without that decision (Level 2 bot + web renderer + WASM + benchmarks + public-release checklist), the `column_four` / Gate 5 decomposition (`archive/tickets/GAT5COLFOUPUB-*`, 18 tickets) is the closest precedent; the lighter `three_marks` / Gate 4 decomposition (`archive/tickets/GAT4THRMARBOA-*`, 16 tickets) is the board-smoke precedent for a gate without the full public-polish surface. The closest match for a gate that adds one new game with bots, WASM, web, and e2e but **no** §4 decision is a hidden-info card game: the `high_card_duel` / Gate 8 decomposition (`archive/tickets/GAT72GAT8HIG-*`, 22 tickets, with a viewer-scoped export split) and — fresher, and the closest twin for a **Level 2** authored-policy hidden-info card game — the `poker_lite` / Gate 10 decomposition (`archive/tickets/GAT10POKLITBET-*`, 18 tickets: betting/showdown with a viewer-scoped export split, a Level 2 bot, and an atlas *second-use* card/private-hand note rather than a §4 hard-gate decision). `poker_lite` is also the immediate second-use sibling that a §4 *third*-use card/private-hand gate compares against (e.g. `plain_tricks` / Gate 10.1, which adds the third-use ledger hard gate on top of the `poker_lite` shape). For a **new public game with a Level 1 (not Level 2) bot and no §4 decision** (the `token_bazaar` / Gate 9 shape — resource/economy, carrying an atlas *first-use* row rather than a promotion decision), the `token_bazaar` / Gate 9 decomposition (`archive/tickets/GAT9TOKBAZBRO-*`, 18 tickets: 001–018, ignoring post-hoc bugfixes 019–020) is the freshest and most exact **Level-1** concrete precedent — it is the `column_four` (~18-ticket) shape with the Level 2 bot replaced by a Level 1 heuristic plus trailing bot-evidence docs. It is also the closest precedent for a commitment/reveal or other hidden-info-adjacent Level-1 successor game (e.g. `secret_draft` / Gate 9.1), which additionally mirrors `high_card_duel`'s commitment/reveal/visibility machinery. The two newest gates extend this lineage and are usually the closest twin: `masked_claims` / Gate 11 (`archive/tickets/GAT11MASCLABLU-*`) is the freshest hidden-info bluff/reaction sibling, and `flood_watch` / Gate 12 (`archive/tickets/GAT12FLOWATCOO-*`, 19 tickets) is the freshest and closest precedent for a **budgeted multi-action, per-seat/per-faction Level-1, no-§4-decision** new-game gate carrying atlas *first/second-use* rows — including the model for a perfect-information(-adjacent) competitive game whose visibility and replay surfaces **merge into one ticket** (no viewer-scoped export split). Refresh these named precedents to the two newest shipped game gates whenever a new game gate lands, so the closest twin is never older than the list. Default shape, in dependency order:

1. **Rules + IP source docs first** (`RULES.md`, `SOURCES.md`) — front-loaded per `docs/OFFICIAL-GAME-CONTRACT.md` §3 (rules prose precedes implementation).
2. **Primitive-pressure ledger decision** — *only when the gate is a §4 third-use mechanic comparison* per `docs/MECHANIC-ATLAS.md` (e.g. Gate 6 `directional_flip`, whose spec calls this "the architectural center of the gate"). Author the ledger + atlas update + decision (reuse / promote / defer-reject / ADR) **before** the crate skeleton — FOUNDATIONS §4 makes the third use a hard gate that must resolve before the game proceeds. The crate skeleton (and any conditional extraction ticket) `Deps` on it. A *promote* outcome spawns a conditional `game-stdlib` extraction ticket (see §Conditional / decision-gated ticket); a defer/reject outcome leaves the coordinate/scan logic local in rules-core. Skip this step for non-third-use gates.
3. **Crate skeleton + workspace wiring** — creates `games/<game>/` (all-`(new)`; see §Greenfield foundation batch, new-crate clause).
4. **Pipeline tickets** by `OFFICIAL-GAME-CONTRACT` sub-concern: rules core → public view/visibility (co-locate the Rust `ui.rs` UI-metadata file here — labels / token / accessibility metadata / preview copy; it is viewer-facing Rust output, distinct from the TS renderer) → effects → replay → bots → tests → golden traces/fixtures → benchmarks → WASM exposure → web renderer → web smoke.
5. **Bot-strategy docs** (`COMPETENT-PLAYER.md`, `BOT-STRATEGY-EVIDENCE-PACK.md`) when the gate ships authored bot-evidence docs — placed *before* the bot for a Level 2 policy, or *trailing and depending on* the bot ticket when a Level 1 policy's evidence pack documents that bot's fixtures (the evidence is produced by the bot tests, so the docs follow them). A required evidence pack is not Level-2-only: a Level 1 bot gate (e.g. `token_bazaar`) that must ship `BOT-STRATEGY-EVIDENCE-PACK.md` + `COMPETENT-PLAYER.md` uses the trailing placement.
6. **Tool-validated docs with their validators** (see §Cross-cutting docs ticket → Tool-validated docs): `RULE-COVERAGE.md` with the rule-coverage registration; `BENCHMARKS.md` with the benches.
7. **Tool + CI registration** — per-tool match arms (`simulate` / `replay-check` / `fixture-check` / `rule-coverage`), then the CI workflow steps.
8. **Trailing game docs** (MECHANICS / UI / AI / ADMISSION / PUBLIC-RELEASE-CHECKLIST) + a **capstone** (mechanic atlas, status hygiene, spec/index `Done`-flip). §Exit-criteria acceptance is typically **distributed** across the bench / wasm-smoke / tool / browser-smoke / CI tickets rather than forced into a synthetic capstone.

Honor the spec's Work-breakdown item count as the granularity floor; split a WB item (commonly the docs item, which fans out by validator) per the §Sizing rules.

## Non-game / infrastructure-spec pattern

Not every spec admits a new `games/*` game. Cross-cutting specs — a shared web-UI surface, a docs/contract obligation, a tooling/validator addition (e.g. `specs/rules-display-shared-surface.md`, decomposed as `tickets/RULDISSHASUR-*`) — have no `OFFICIAL-GAME-CONTRACT` pipeline to follow. They share a recurring shape, in dependency order:

1. **Contract / docs foundation first** — the template, foundation/area-doc amendments, and the `specs/README.md` index row (`Planned`) that make the obligation permanent. Everything else depends on the contract it defines.
2. **Tooling / validators** — the deterministic CI guard scripts (e.g. copy + fail-closed check) that enforce the contract. Wire the CI step with the ticket that first makes it go green (avoid a multi-PR red-CI window), not necessarily the script-authoring ticket.
3. **Content authoring** — the per-item content the contract requires (e.g. per-game player docs), often split pilot-vs-rest so the pilot anchors house style; the pilot `Deps` the contract + tooling tickets.
4. **Presentation infrastructure** — the shared component + presentation-only state (FOUNDATIONS §2: no legality decided in TypeScript), independent of content authoring and parallelizable with it.
5. **Integration** — wire the access/entry points to the presentation infra; `Deps` the component ticket (structural producer/consumer).
6. **Verification smoke** — the e2e / accessibility / no-leak smoke; a §Deliverable-doubles-as-capstone ticket (it ships new test infra), `Deps` the content + integration tickets.
7. **Capstone closeout** — verification-only acceptance evidence + the `specs/README.md` `Done`-flip (`Deps` the smoke leaf and the index-row-creating ticket, a create-then-modify on `specs/README.md`).

§Exit-criteria acceptance is typically **distributed** across the tooling / smoke / capstone tickets rather than forced into a synthetic capstone. Honor the spec's Work-breakdown count as the granularity floor (§Sizing rules) and split oversized items (authoring, UI) per the §Split rule.
