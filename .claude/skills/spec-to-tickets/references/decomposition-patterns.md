# Decomposition Patterns (Step 3)

Analyze the spec and identify discrete work units:

- Each ticket is a **reviewable diff** — small enough for comfortable manual review.
- Map **dependencies** between tickets and name them in each ticket's `Deps`.
- Determine **priority ordering** from the dependency graph and criticality.
- Ensure **every spec deliverable is covered** — no silent skipping. If a deliverable seems wrong, flag it with the 1-problem / 3-options / 1-recommendation format rather than omitting it.
- Use the spec's **Out of Scope** / **Forbidden changes** sections to populate each ticket's Out of Scope — these are pre-validated non-goals.

## Deliverable-coverage categories

Most deliverables get their own ticket. These categories are exempt from per-deliverable ticket production — document them in the Step 4 summary and Step 6 coverage mapping instead:

- **No-change deliverable** — the spec explicitly states no change is needed ("No new crate"). Note it if non-obvious; produce no ticket.
- **Cross-spec consumer-side deliverable** — the work is consumed by a sibling gate's tickets rather than producing a discrete output here. The change is needed, but it happens in another spec's decomposition. Note the ownership as `cross-spec: <sibling-gate>` in Step 4 and `not ticketed (cross-spec: …)` in Step 6.
- **Distributed deliverable** — one conceptual deliverable encoded as a contract distributed across multiple tickets (e.g. a per-game applicability rule carried as a predicate on each affected module, with no single owner). Note it as `distributed across <ticket-list>`; produce no separate "container" ticket.
- **Informational-pointer deliverable** — a section whose body only points to sibling deliverables ("handled in §X above and §Y below"). The actual work is in the pointed-to siblings, each ticketed. Note the routing; produce no ticket.
- **Already-landed deliverable** — the intent was satisfied by an edit landed during a prior in-session skill invocation (e.g. `/reassess-spec`'s user-authorized in-session edit). Produce no ticket; record `not ticketed (already landed during <skill> session <date>)` in Step 6. **Partially pre-landed multi-location deliverable**: when a deliverable spans several sub-locations and *some* are already committed (at spec-authoring time or by a prior in-session edit) while others remain — e.g. a `specs/README.md` deliverable whose index row + a blocked-gate row already landed but whose intro/spec-format/workflow rewrites are pending — it is **not** a whole-deliverable drop: produce a ticket covering the **remaining** locations and list the pre-landed ones as **verify-only** in that ticket's Out-of-scope / Acceptance (e.g. "Locations 2–3 already committed — verify-only; this ticket lands Locations 1/4/5"). Note the split in Step 4 Notes so a reviewer isn't surprised the ticket skips part of the deliverable.
- **Inferred deliverable** — named or implied by §Objective / §Acceptance evidence / §Exit criteria but absent from §Deliverables. This DOES get a ticket; document the inference in the ticket's Assumption Reassessment with a citation to the motivating section, and flag it in Step 4 Notes as `inferred from <section>`. When the motivating reference is ambiguous or would expand scope, route through a Step 2 Issue instead.
- **Process / no-artifact work-item** — a Work-breakdown item that produces no reviewable diff: an intake/orientation step, "reassess assumptions against current code," "confirm all referenced paths," or similar. Its substance is discharged by every ticket's Assumption Reassessment section, not by a discrete output. Produce no ticket; record `not ticketed (process step — folded into per-ticket Assumption Reassessment)` in Step 6. (Common as the leading WB row, e.g. a `G<gate>-00 intake` item.)

## Sizing rules

- **Merge** — when multiple deliverables share the same file set and cannot be implemented independently, merge into one ticket. Note merged deliverables in Step 4 Notes.
- **Spec-declared admission rule overrides Merge** — when the spec itself declares a one-surface-per-diff / no-bundled-diff admission rule (e.g. 8C-R1 §11.3: "bundled 'all six games' or 'update snapshots' diffs are inadmissible even when green"), treat it as a **hard granularity floor that overrides the Merge heuristic and the shared-file-set merge temptation**: keep each declared surface its own ticket even when sibling tickets share a file set, surfacing the shared file per §"Same file / same new directory" rather than merging across the spec's declared surface boundary. (8C-R1 kept six WASM-output tickets separate though all touch `crates/wasm-api/src/seats.rs`.)
- **Split** — when a single deliverable is too large for one reviewable diff (it spans, say, a contract surface + a schema + a validation rule + a bot emitter), split by logical sub-concern (e.g. write path vs. read/verify path). Note the split in Step 4 and show `D<n>→00x+00y` in Step 6.
- **Candidate-AGENT-TASK granularity is a floor, not a 1:1 mapping** — when the spec's Work-breakdown table labels each item as a candidate AGENT-TASK (e.g. a `Becomes AGENT-TASK: yes` column), treat that as the author's intended *granularity floor* — the coarsest acceptable grouping — not a binding one-ticket-per-WB-item rule. Honor it as the default, but apply the Split rule when a labeled item exceeds one reviewable diff (the highest-risk surface — `engine-core` kernel growth, a full `games/*` crate — usually does). Surface every divergence from the spec's item count in the Step 4 table with `D<n>→00x+00y`, and offer the coarser-grained alternative ticket count so the user can opt back down to the spec's granularity. **Coarse "workstream"-shaped Work breakdowns**: some specs (e.g. a `Workstream | Required outcome | Hard constraints` table) enumerate broad workstreams with no per-item AGENT-TASK label, and the natural decomposition — by the `docs/OFFICIAL-GAME-CONTRACT.md` pipeline or by logical sub-concern — is both finer than and orthogonal to those rows. Treat the workstream count as a soft lower bound (not a 1:1 target), decompose by sub-concern, and surface the fan-out in the Step 4 table (e.g. `9 workstreams → 16 tickets`) with the coarser-grained alternative offered, the same way the labeled-column case surfaces `D<n>→00x+00y`. **Spec-supplied candidate ticket IDs**: when a spec's Work breakdown pre-labels candidate ticket IDs in a prefix different from the run namespace (e.g. §14 lists `RULSUR-001…006` but the run namespace is `RULDISSHASUR`), treat those IDs as granularity-floor candidates only — renumber into the requested namespace and map the spec's IDs to the produced tickets in the Step 6 coverage mapping (e.g. `RULSUR-002 → 003+004`); never adopt the spec's prefix into the produced ticket filenames. **Already-topological spec numbering**: when the spec's candidate IDs use a different prefix but an already-topological, gap-tolerant numeric scheme (e.g. wave-grouped `…-101`/`-201`/`-301`), preserving the spec's numeric *suffix* under the run-namespace prefix (`<NS>-101`, `<NS>-201`, …) for near-1:1 traceability is preferred over sequential renumber — provided lower numbers still land earlier (Step 3's topological-order rule still holds) and the Step 6 map is recorded either way. Sequential renumber (`001…NNN`) remains correct when the spec's scheme is not already topological. **Same-prefix collision after a split/merge**: when the spec pre-labels candidates in the *same* prefix as the run namespace (e.g. §15 lists `VICEXPSHASUR-001…008` and the run namespace is `VICEXPSHASUR`) AND a split or merge shifts the spec's own later numbers off the produced files (splitting the spec's `-004` into produced `004…008` makes the spec's `-005` mean a *different* ticket than the produced `-005`), do NOT key the Step 6 coverage map by the spec's ticket IDs — they now collide in meaning with the produced filenames. Key the map by the spec's **section identifier** instead (`§15.4 → 004+005+006+007+008`, `§15.5 → 009`), per the Step 6 item-3 "key by the spec's own identifiers" rule, and flag in Step 4/Step 6 that the spec's `PREFIX-NNN` and the produced `PREFIX-NNN` diverge past the split point so a reviewer does not conflate them. **Spec-internal finer-granularity directive**: when a directive inside the spec itself (e.g. a `/reassess-spec` finding stating "one ticket per `<X>`") specifies a granularity *finer* than the WB floor, treat that finer granularity as the recommended **default** and offer the coarser clustering as the alternative — the reverse of the WB-floor case — since the spec's own stated granularity anchors the recommendation (the user can still opt down to the coarser grouping).
- **Same file / same new directory** — when all deliverables modify the same file or land under one new directory (a new crate, a new `games/*` dir), decompose by logical section/module, not by file boundary. Each ticket targeting a different section is a valid reviewable diff. Note the shared file once in Step 4. When two **mutually-independent** sibling tickets both modify a **pre-existing production** file (conflict-prone — e.g. two same-game migrations editing one `effects.rs`/`setup.rs`/`Cargo.toml`), prefer a `Deps`-serialize over a bare overlap-flag; reserve the Step 6 §Shared-file-overlaps flag for low-conflict **test-file** appends.
- **Intra-batch create-then-modify chains** — when ticket B's Files to Touch contains a `(modify)` entry for a file ticket A creates `(new)` in the same batch, declare `Deps: A` on B explicitly. The file-creation is a load-bearing structural dependency (B fails at implementation time if A hasn't landed). **Transitive linear chain**: B need not name A *directly* when an unbroken predecessor `Deps` chain reaches A — in a sequential pipeline (A creates the stub modules; B→C→D→… each `Deps` its immediate predecessor and modifies those stubbed files), the chain already guarantees A lands first, so the direct-`Deps`-on-creator requirement is satisfied transitively (the same transitive-head logic the §Capstone integration ticket `Deps` uses). Annotate `(modify; created by A)` in Files to Touch as the audit trail, and reserve the *direct* `Deps: A` requirement for the parallel-branch case where A is **not** on B's predecessor chain. **Verify the transitive chain before relying on it**: when you annotate `(modify; created by A)` without a direct `Deps: A`, trace B's predecessor `Deps` chain and confirm it actually reaches A unbroken — a chain *claimed* transitive but not traced (B `Deps` a parallel branch that never descends to A) leaves the real ordering dependency undeclared until Step 6 catches it. **Generated-from dependency**: this create-then-modify relationship extends to a ticket that *generates or derives* an artifact **from** a file another ticket authored even when it never `(modify)`s that file — e.g. ticket B runs `scripts/copy-player-rules.mjs` to generate `apps/web/public/rules/<game>.md` from ticket A's `HOW-TO-PLAY.md`, or finalizes/regenerates a coverage doc A created; B `Deps` A (directly, or transitively with the chain traced) on that source-authoring relationship, not only on a `(modify)` of the same path. This is distinct from §Shared-file overlaps (Step 6 item 5), which covers parallel adds by mutually-independent tickets to a shared file — whether pre-existing or created in-batch by a common foundation ticket they each `Deps:` on.
- **Acceptance-gated ticket** — when a downstream ticket's precondition is the *acceptance/review* of an upstream artifact rather than its mere file existence — e.g. a ROADMAP edit the roadmap header law blocks until an ADR carries `Status: Accepted`, or any ticket the chain must pause on for a human sign-off — `Deps` the authoring ticket AND encode the acceptance precondition as an Acceptance-Criteria / Verification-Layer grep (`Status: Accepted` on the upstream artifact), because `Deps` alone only guarantees the artifact *exists*, not that it was accepted. Flag the human-acceptance pause in the Step 6 suggested-order line so a reviewer knows the chain stops for a decision. This is the decision-precondition analogue of create-then-modify's file-creation precondition.
- **Greenfield foundation batch** — when the spec is the repo's first code (no pre-existing code tree, e.g. the Gate 0 repository-skeleton spec), every ticket's Files to Touch is legitimately all-`(new)`; do not treat that as a suspicious under-statement. The structural ordering is carried by intra-batch create-then-modify chains (above), and the deliverables' consumers are the sequenced later gates, not present code — mirroring the greenfield note in `reassess-spec`'s `references/codebase-validation.md`. **New crate / new `games/*` dir in an existing repo**: the same all-`(new)` legitimacy applies — a gate that adds a new crate or game directory will have most of its Files to Touch be `(new)` under that dir, with ordering carried by the crate-skeleton ticket (which creates the dir) plus intra-batch create-then-modify chains, not by pre-existing files. Do not treat a new-game gate's all-`(new)` tickets as under-statements.
- **Spec WB dependency annotations are advisory** — map each ticket's `Deps` from structural codebase truth (file creation, producer/consumer symbol introduction, shared-surface ordering), not from the spec's Work-breakdown dependency column; a spec-declared dependency that validation shows has no structural basis is dropped, and every divergence from the spec's annotations is surfaced in the Step-4 table Notes.

## Ticket shapes

**Trailing-shape co-occurrence**: a single spec closeout deliverable often bundles docs-finalization **and** exit-evidence; split it into both shapes below — a cross-cutting docs ticket (Deps = the surface-citing implementation tickets) plus a capstone (Deps = the leaf set, including the docs ticket). The `Done`-status flip (spec Status → `Done` and the `specs/README.md` index row) defaults to the **capstone**, since it is gated on exit evidence passing; the docs ticket carries doc content and any earlier `Planned` update, not the completion flip — unless the spec's §Documentation-updates assigns the index flip to the docs ticket, in which case the §Cross-cutting docs ticket `Deps: <capstone>` exception applies. **Exception (infrastructure specs)** — for an infrastructure spec whose doc amendments are deferred-to-closeout (the §Non-game / infrastructure-spec pattern's closeout-deferred-docs variant), the single capstone MAY carry both the doc lift and the `Done`-flip rather than splitting into two shapes.

### Capstone integration ticket

A single trailing ticket whose scope IS the spec's §Exit criteria / §Acceptance evidence, exercising every prior implementation ticket end-to-end. It introduces no new production logic; it exercises the pipeline the earlier tickets composed.

**Deliverable-doubles-as-capstone**: the "no new production logic" constraint applies to *acceptance-only* capstones. When the gate's own verification harness is itself a deliverable (e.g. a browser-E2E smoke ticket that adds a Puppeteer dependency + harness code), that ticket MAY serve as the capstone even though it ships new test infrastructure — note the dual role in the Step 4 table. And when no single ticket exercises the whole gate end-to-end, acceptance MAY be distributed across the trailing tickets (e.g. ABI smoke + browser smoke + a11y/no-leak + docs/status-flip); record the distribution in the Step 6 deliverable-coverage mapping instead of forcing a synthetic capstone.

- Its acceptance criteria enumerate the spec's exit-criteria / acceptance-evidence bullets as test sub-cases.
- Re-enumerate expected counts from a fixture at test start rather than hardcoding (hardcoded counts go stale).
- **Files to Touch** — `_TEMPLATE.md` makes the section mandatory, but a capstone introduces no production logic, so it is one of: the new e2e/smoke/golden-trace test the ticket adds; `None — verification-only` when the gate is exercised purely by running existing scripts (simulation CLI, replay-check) plus a manual runbook; or the **status-reconciliation surfaces** the §Ticket shapes `Done`-flip default assigns to the capstone (the `specs/README.md` index row, the spec's own Status, the `docs/MECHANIC-ATLAS.md` first-use / atlas row, `progress.md` / root `README.md` — but **excluding** the root `README.md` "current official games are" catalog list, which for a web-exposed game gate co-lands with the web-smoke ticket per §Validator-consumed catalog docs, never the capstone) when no separate cross-cutting docs ticket owns them — a capstone that performs status reconciliation legitimately lists those docs. Do NOT list the *upstream tickets'* files — the capstone exercises them, it does not modify them (the status-reconciliation surfaces above are the capstone's own, distinct from upstream files).
- **Test Plan** — a verification-only capstone that adds no test files uses a parallel New/Modified-Tests fallback, e.g. `None — verification-only capstone; it exercises the acceptance suite composed by prior tickets and adds no tests` — distinct from the `_TEMPLATE.md` `None — documentation-only ticket` line, which is reserved for docs-only tickets and reads as factually wrong for a verification capstone. A capstone that carries **both** docs reconciliation and exit-verification (the §Ticket-shapes trailing-shape co-occurrence / infrastructure-spec exception) fits neither single line — use a combined fallback, e.g. `None — docs-reconciliation + verification capstone; it reconciles the named docs/status surfaces and exercises the prior tickets' acceptance suite, adding no test file`. A **behavior-neutral adoption ticket** — one that modifies production code but is byte/hash/visibility-neutral and adds no test (a helper-adoption pilot whose proof is the existing replay/visibility suites staying byte-identical to the characterization baseline) — is neither a docs-only ticket nor a capstone; it uses a parallel fallback, e.g. `None — behavior-neutral adoption; the existing replay/visibility suites and the characterization baseline are the regression guard`. A **code ticket whose test suite lands in a paired sibling ticket** — a validator/checker split off from its own tests (e.g. a `scripts/check-*.mjs` checker shipped in one ticket, its `*.test.mjs` + fixtures in a separate ticket) — adds no test file itself and uses a parallel fallback, e.g. `None — checker logic only; its Node test suite + fixtures land in <sibling ticket>; verification at this landing is the green run against the real repo`. The Step-6 Test-Plan-fallback-contradiction check accepts any of these no-tests fallbacks.
- **Manual-runbook variant** — when an exit-criterion cannot run in the project's test infra — either it requires invoking an LLM-driven skill (not runnable from test code), **or** it is a UI/manual smoke with no browser-automation harness in the project (the WASM web shell) — structure the ticket as a hybrid: a *manual runbook* (in the What to Change section) the implementer follows by hand — for an LLM skill: fixture copy strategy, skill invocations against the copy, expected post-skill state with verification commands; for a UI smoke: the click-path with each step's expected observable result — plus an automated-test subsection for any test-runnable portion (golden-trace assertions, replay/hash checks, no-leak negative tests, bot-legality checks). The acceptance criteria distinguish CI-runnable assertions from implementer-checklist runbook steps.
- **`Deps`** — prefer a single `Deps: <transitive-head-ticket>` when the upstream DAG is a linear chain (the head's own `Deps` reach every prior ticket). When the upstream DAG has parallel branches (e.g. a docs ticket running parallel to implementation tickets), enumerate the **leaf set** — every leaf whose transitive `Deps` collectively cover the full chain.

### Cross-cutting docs ticket

A single trailing ticket whose Files to Touch are markdown docs (READMEs, architectural docs, skill reference files, `specs/README.md` index updates — see the §Ticket shapes `Done`-flip default) that must land atomically once all upstream implementation tickets ship.

- One Files-to-Touch entry per docs surface the deliverable names.
- Acceptance criteria as grep-proofs against the post-implementation tree (exact-string matches for new/renamed symbols, count assertions for status-line updates).
- Out of Scope names the implementation surfaces this ticket does NOT touch, so reviewers know docs-only is the contract.
- **`Deps`** — list every implementation ticket whose surface the docs reference (docs reference each surface independently, so transitive-head is insufficient here). **Exception** — when the docs ticket only annotates *aggregate completion* (a `specs/README.md` status flip / "verified by Gate N" note) rather than citing individual symbols, and a capstone ticket already gates every referenced surface, `Deps: <capstone>` is sufficient and more legible than re-enumerating the surface set. Enumerate per-surface only when the docs cite individual symbols/APIs that could go stale independently.
- **When NOT to use** — if each docs change can co-locate with its implementing ticket without a staleness window, per-ticket docs are cleaner. Use this shape when ≥1 docs surface requires all upstream surfaces to exist coherently before it can land (status-line counts, cross-tool tables, multi-surface skill guidance).
- **Tool-validated docs** — a game doc that a validator *tool* consumes (in Rulepath, `tools/rule-coverage` reads `RULES.md` / `RULE-COVERAGE.md` / `BENCHMARKS.md`) MUST co-land with (or before) that tool's registration ticket, not in a trailing docs-only ticket — otherwise the tool registration has nothing valid to check. Split the official-game doc set by *which ticket validates each doc* (rules docs front-loaded per OGC §3; `RULE-COVERAGE.md` with the rule-coverage registration; `BENCHMARKS.md` with the benches; bot-strategy docs ordered relative to the bot whose evidence they document — before a Level 2 policy, or trailing and depending on the bot ticket when an evidence pack documents a Level 1 bot's fixtures; the rest trailing), not by a blanket docs-vs-code boundary. **Cross-ticket dependency when one validator reads docs split across tickets**: `tools/rule-coverage` validates `RULES.md` + `RULE-COVERAGE.md` + `BENCHMARKS.md`, but the placement above puts `BENCHMARKS.md` with the benches ticket — a *different* ticket from the rule-coverage registration. So the rule-coverage registration ticket's fully-green `rule-coverage --game <g>` state depends on the benches ticket having landed `BENCHMARKS.md`. Either declare `Deps:` from the rule-coverage ticket onto the benches ticket, or flag the expected partial-green window in Step 4 / Step 6 (mirroring the `check-catalog-docs` red-window note), so a reviewer running `rule-coverage` before the benches ticket lands isn't surprised by an incomplete coverage report.
- **Validator-consumed *catalog* docs (web-exposed game gates)** — `scripts/check-catalog-docs.mjs` (run in `gate-1-game-smoke.yml`) is a second validator-consumed-docs surface, keyed off the `crates/wasm-api/src/lib.rs` `GAME_*` catalog const as source of truth. It mechanically checks three README surfaces: the `apps/web/README.md` intro catalog list, the root `README.md` "current official games are" list, and the `apps/web/README.md` Smoke Layers `smoke:e2e` bullet (the `apps/web/README.md` Shell Surface renderer bullet is process-enforced, not mechanically checked). For a web-exposed game gate these three surfaces MUST co-land with the web-smoke / gate-1-CI ticket that wires `smoke:e2e` (and references the new `GAME_*` const) — NOT in a trailing docs-only ticket or the capstone — or the catalog README drift gets deferred to an aftermath pass (the Gate 9 `token_bazaar` aftermath was exactly this miss). Note the expected red-CI window: because `check-catalog-docs` runs continuously and keys on the wasm-api catalog const, every PR between the WASM-registration ticket (adds the const) and the README-reconciliation ticket shows a red `check-catalog-docs` step — flag it in Step 4 / Step 6 as an expected mid-gate state resolved at the web-smoke ticket, so a reviewer reading PRs in order isn't surprised.
- **Validator-consumed *player-rules* docs (game gates)** — `scripts/check-player-rules.mjs` (run in `gate-1-game-smoke.yml`) validates each game's `HOW-TO-PLAY.md` and its generated `apps/web/public/rules/<game>.md` (produced via `scripts/copy-player-rules.mjs`), keyed off the wasm catalog const. So `HOW-TO-PLAY.md` and the generated player-rules markdown MUST co-land with the **WASM-registration ticket** (the one that adds the catalog const) — not a trailing docs-only ticket — or `check-player-rules` shows a red window exactly like `check-catalog-docs`. For a **hidden-info game**, that same WASM ticket also registers the game in `HIDDEN_INFO_GAMES` (`scripts/check-player-rules.mjs`) and fills the `HOW-TO-PLAY.md` hidden-information section, which the checker requires for listed games (the `flood_watch` / Gate 12 and `event_frontier` / Gate 14 precedent). The trailing player/mechanic docs ticket then only *syncs* `HOW-TO-PLAY.md` and authors `MECHANICS`/`UI`/`AI` — it does not own the player-rules surface.
- **Validator-consumed *outcome-explanation* docs (web-exposed game gates)** — `scripts/check-outcome-explanations.mjs` (run in `gate-1-game-smoke.yml`) is a third validator-consumed-docs surface, keyed off the wasm catalog const. It requires four co-dependent surfaces per game: the stable scoring/terminal rule IDs in `RULES.md`, the "Outcome / victory explanation" section in `games/<game>/docs/UI.md`, the viewer-safe rationale mirror in `apps/web/src/wasm/client.ts`, and the static copy keys in `apps/web/src/components/outcomeExplanationTemplates.ts`. Because these split across the rules-doc, web-renderer, and trailing-docs tickets, `check-outcome-explanations` shows a red window from the WASM-registration ticket (which adds the catalog const) until the *last* of the four surfaces lands — typically the `UI.md` outcome section in the trailing docs ticket. Co-land the web surfaces (`client.ts`, `outcomeExplanationTemplates.ts`) with the web-renderer ticket and flag the interim red window in Step 4 / Step 6, mirroring the `check-catalog-docs` red-window note, so a reviewer reading PRs in order isn't surprised.

### Conditional / decision-gated ticket

A ticket worked only if an upstream **decision ticket** resolves a certain way — e.g. a `game-stdlib` helper extraction worked only if a §4 primitive-pressure ledger decides *promote*, or a migration step worked only if an investigation confirms a root cause. Coverage is not silently skipped: the deliverable still gets a ticket, with its conditionality made explicit.

- **`Deps`** — depend on the deciding ticket (the one that records the branch outcome).
- **Problem / Out of Scope** — state the trigger condition in Problem ("worked only if <decision ticket> decides X") and, in Out of Scope, "proceeding at all if <decision ticket> decided otherwise (this ticket is then not-applicable)."
- **Fallback** — name where the work goes if the condition is *not* met (e.g. "rules-core implements the logic locally instead"), so the alternate path is owned, not orphaned.
- **Step 4 / Step 6** — flag it in the Step 4 table Notes as `CONDITIONAL on <ticket>` and record in Step 6 that it closes not-applicable if the decision goes the other way. A sibling ticket that would *consume* the conditional output (e.g. rules-core using the promoted helper) should NOT hard-`Deps` on the conditional ticket; instead it `Deps` on the stable predecessor and notes the conditional consumption, so its dependency graph stays resolvable whether or not the conditional ticket is worked.
- **Self-deciding (audit-within-ticket) variant** — when the deciding step is an audit the ticket *itself* performs rather than a separate upstream decision ticket (e.g. an O13-style "audit what the wasm bridge exposes for the bots → render the §15 affordance or record a named follow-up"), there is no deciding ticket to depend on: `Deps` the stable predecessor (or `None`), state in Problem that the in-ticket audit gates the build, put both branches (implement / record follow-up) in What to Change, and put the not-applicable path in Out of Scope. `Engine Changes` MAY carry a conditional value (e.g. `Conditional — Yes (presentation-only) if implemented; None if the audit defers`). Flag it in Step 4 Notes as `CONDITIONAL (self-deciding audit)`.
- **Spec-declared-optional (user-droppable) variant** — when the spec itself marks a deliverable optional via an Assumption (e.g. a teaching aid "included; drop if unwanted — no dependents") rather than gating it on a decision ticket or an in-ticket audit, the deliverable still gets a ticket: state "optional per spec Assumption `A<N>`; droppable with no dependents" in Problem and the not-applicable path in Out of Scope; `Deps` the stable predecessor; flag Step 4 Notes as `OPTIONAL (spec Assumption A<N>)` and surface the count delta per SKILL.md §"Contingent ticket count … spec-level conditionality". A *consuming* sibling must not hard-`Deps` it (per the Step 4 / Step 6 rule above); but a **closeout/capstone ticket that only verifies or reconciles the optional ticket's surface** MAY list it in `Deps` with an explicit Out-of-scope "remove this row if the optional ticket is dropped" note as the audit trail — distinct from a consumer, since the capstone exercises rather than consumes it.

## Official-game-gate pattern

Gates that admit a new official game under `games/*` (the dominant Rulepath gate type per `docs/ROADMAP.md`) share a recurring ~16–22-ticket shape. For a gate that carries the §4 primitive-pressure decision (ledger + atlas update + conditional `game-stdlib` extraction), the `directional_flip` / Gate 6 decomposition (`archive/tickets/GAT6DIRFLI-*`, 21 tickets) is the closest precedent — it is the freshest full board-game gate and the only one whose decomposition includes the primitive-pressure decision + conditional-extraction shape; a gate that *also* introduces a novel boundary (a compound action tree, multi-segment WASM export) trends toward the upper end (~22, e.g. Gate 7 `draughts_lite`). For a **full public-game gate** without that decision (Level 2 bot + web renderer + WASM + benchmarks + public-release checklist), the `column_four` / Gate 5 decomposition (`archive/tickets/GAT5COLFOUPUB-*`, 18 tickets) is the closest precedent; the lighter `three_marks` / Gate 4 decomposition (`archive/tickets/GAT4THRMARBOA-*`, 16 tickets) is the board-smoke precedent for a gate without the full public-polish surface. The closest match for a gate that adds one new game with bots, WASM, web, and e2e but **no** §4 decision is a hidden-info card game: the `high_card_duel` / Gate 8 decomposition (`archive/tickets/GAT72GAT8HIG-*`, 22 tickets, with a viewer-scoped export split) and — fresher, and the closest twin for a **Level 2** authored-policy hidden-info card game — the `poker_lite` / Gate 10 decomposition (`archive/tickets/GAT10POKLITBET-*`, 18 tickets: betting/showdown with a viewer-scoped export split, a Level 2 bot, and an atlas *second-use* card/private-hand note rather than a §4 hard-gate decision). `poker_lite` is also the immediate second-use sibling that a §4 *third*-use card/private-hand gate compares against (e.g. `plain_tricks` / Gate 10.1, which adds the third-use ledger hard gate on top of the `poker_lite` shape). For a **new public game with a Level 1 (not Level 2) bot and no §4 decision** (the `token_bazaar` / Gate 9 shape — resource/economy, carrying an atlas *first-use* row rather than a promotion decision), the `token_bazaar` / Gate 9 decomposition (`archive/tickets/GAT9TOKBAZBRO-*`, 18 tickets: 001–018, ignoring post-hoc bugfixes 019–020) is the freshest and most exact **Level-1** concrete precedent — it is the `column_four` (~18-ticket) shape with the Level 2 bot replaced by a Level 1 heuristic plus trailing bot-evidence docs. It is also the closest precedent for a commitment/reveal or other hidden-info-adjacent Level-1 successor game (e.g. `secret_draft` / Gate 9.1), which additionally mirrors `high_card_duel`'s commitment/reveal/visibility machinery. The two newest gates extend this lineage and are usually the closest twin: `flood_watch` / Gate 12 (`archive/tickets/GAT12FLOWATCOO-*`, 19 tickets) is the freshest precedent for a **budgeted multi-action, per-seat/per-faction Level-1, no-§4-decision** new-game gate carrying atlas *first/second-use* rows — including the model for a perfect-information(-adjacent) competitive game whose visibility and replay surfaces **merge into one ticket** (no viewer-scoped export split); and `frontier_control` / Gate 13 (`archive/tickets/GAT13FROCONASY-*`, 17 tickets) is the freshest and closest precedent for an **asymmetric graph-map / area-control gate carrying §4 atlas reviews that resolve defer/reject (no extraction)** — graph topology, faction-disjoint action sets, per-faction Level-1 bots, and a primitive-pressure ledger ticket (its `-002`) that records the reviews without a promote/extraction outcome (distinct from Gate 6's promote-with-conditional-extraction shape). (`masked_claims` / Gate 11 (`archive/tickets/GAT11MASCLABLU-*`) remains the freshest hidden-info bluff/reaction sibling.) The two newest shipped game gates are **`briar_circuit` / Gate 16** (`archive/tickets/GAT16BRICIRTRI-*`, 18 tickets) and **`vow_tide` / Gate 17** (`archive/tickets/GAT17VOWTIDOHHEL-*`); `river_ledger` Gate 15.1 is the §Existing-game behavior-extension precedent below. `briar_circuit` is the closest precedent for a **hidden-info trick-taking** gate (the *second* close trick-taking use: follow-suit / led-suit comparator / winner-leads). `vow_tide` / Gate 17 (Oh Hell) is the shipped precedent for a §4 *third*-use trick-taking gate that **promotes** a `game-stdlib` trick helper and back-ports the prior trick-taking games (`plain_tricks`, `briar_circuit`) in-gate; a later trick-taking gate that reuses those promoted helpers **unchanged** — no new third-use gate, e.g. Gate 18 `blackglass_pact` — compares against both. They are the freshest closest twins for any trick-taking decomposition. Refresh these named precedents to the two newest shipped game gates whenever a new game gate lands, so the closest twin is never older than the list. Default shape, in dependency order:

1. **Rules + IP source docs first** (`RULES.md`, `SOURCES.md`) — front-loaded per `docs/OFFICIAL-GAME-CONTRACT.md` §3 (rules prose precedes implementation).
2. **Pre-skeleton governance gates** — author these *before* the crate skeleton (it `Deps` on them). **(a) forward-v1 reuse-first scaffolding audit + implementation admission — required for *every* new-game gate** since Unit 8F / ADR 0008 (FOUNDATIONS §11 makes it a universal acceptance invariant; §12 makes skipping it a stop condition): a ticket authoring the `MECHANICS.md` C-01…C-10 reuse-first audit, the lawful-shared-home review, and `GAME-IMPLEMENTATION-ADMISSION.md`, which **blocks** serious code (the crate skeleton `Deps` on it); `Engine Changes: None` (game-local governance docs). Its matching **post-build `ci/scaffolding-audits.json` `forward-v1` receipt + register/atlas reconciliation** ticket lands in the step-8 closeout group. Precedent: the 8F governance unit `archive/tickets/PREGAT18FORSCA-*`. **(b) Primitive-pressure ledger decision** — *only when the gate is a §4 third-use mechanic comparison* per `docs/MECHANIC-ATLAS.md` (e.g. Gate 6 `directional_flip`, whose spec calls this "the architectural center of the gate"). Author the ledger + atlas update + decision (reuse / promote / defer-reject / ADR) **before** the crate skeleton — FOUNDATIONS §4 makes the third use a hard gate that must resolve before the game proceeds. The crate skeleton (and any conditional extraction ticket) `Deps` on it. A *promote* outcome spawns a conditional `game-stdlib` extraction ticket (see §Conditional / decision-gated ticket); a defer/reject outcome leaves the coordinate/scan logic local in rules-core. Skip this step for non-third-use gates. **Spec-pre-decided promotion (committed, not conditional)**: when the spec — commonly via a prior `/reassess-spec` pass — has already *locked* the promote decision, the extraction ticket and the back-port tickets are **committed, not conditional** (drop the CONDITIONAL framing and the deciding-ticket `Deps`). FOUNDATIONS §4/§11 require every already-shipped matching game to adopt the primitive in-gate, so the promote yields the extraction ticket **plus one behavior-preserving back-port ticket per matching shipped game** (Gate 17 `vow_tide` / Oh Hell, `archive/tickets/GAT17VOWTIDOHHEL-*`, extracted a trick helper and back-ported both `plain_tricks` and `briar_circuit`), each `Deps` the extraction ticket and gated on `replay-check --all` byte-identical hash/trace preservation, with the §10A no-promotion-debt receipt carried on the closeout capstone.
3. **Crate skeleton + workspace wiring** — creates `games/<game>/` (all-`(new)`; see §Greenfield foundation batch, new-crate clause).
4. **Pipeline tickets** by `OFFICIAL-GAME-CONTRACT` sub-concern: rules core → public view/visibility (co-locate the Rust `ui.rs` UI-metadata file here — labels / token / accessibility metadata / preview copy; it is viewer-facing Rust output, distinct from the TS renderer) → effects → replay → bots → tests → golden traces/fixtures → benchmarks → WASM exposure → web renderer → web smoke. Golden traces co-locate with the module ticket whose behavior they capture (a setup ticket carries its setup traces, the showdown ticket its split traces, the visibility ticket its no-leak traces) — validated cross-cuttingly by the `replay-check` registration ticket — rather than a single dedicated traces ticket; consolidate into one traces ticket only when no single module owns them. engine-core's replay recorder already exists, so a module ticket may author its golden traces before the game's own `replay_support.rs` lands (which adds only the public-export redaction layer).
5. **Bot-strategy docs** (`COMPETENT-PLAYER.md`, `BOT-STRATEGY-EVIDENCE-PACK.md`) when the gate ships authored bot-evidence docs — placed *before* the bot for a Level 2 policy, or *trailing and depending on* the bot ticket when a Level 1 policy's evidence pack documents that bot's fixtures (the evidence is produced by the bot tests, so the docs follow them). A required evidence pack is not Level-2-only: a Level 1 bot gate (e.g. `token_bazaar`) that must ship `BOT-STRATEGY-EVIDENCE-PACK.md` + `COMPETENT-PLAYER.md` uses the trailing placement.
6. **Tool-validated docs with their validators** (see §Cross-cutting docs ticket → Tool-validated docs): `RULE-COVERAGE.md` with the rule-coverage registration; `BENCHMARKS.md` with the benches.
7. **Tool + CI registration** — per-tool match arms (`simulate` / `replay-check` / `fixture-check` / `rule-coverage`), then the CI workflow steps. `ci/games.json` (the gate-1 per-game `{id, sim_flags, e2e}` registry) co-lands with the **e2e-smoke ticket**, because its `e2e` field names the smoke file that ticket creates (a create-then-reference ordering dependency) and its `sim_flags` mirror the `simulate` arm — place it there, not in the skeleton or tool-registration ticket, to avoid a dangling `e2e` reference (`Engine Changes: Yes (audit + e2e)`). When a tool arm is *consumed* by an earlier sub-concern's ticket — e.g. the `simulate` arm drives the bots ticket's seeded-simulation 1,000-match smoke — place that arm with its **consumer** ticket at Step 4 (not bundled with the other tool registrations), so the placement lands in the approved table rather than being relocated across a deliverable boundary at Step 5.
8. **Trailing game docs** (MECHANICS / UI / AI / ADMISSION / PUBLIC-RELEASE-CHECKLIST), a **forward-v1 governance closeout** (the post-build `ci/scaffolding-audits.json` `forward-v1` receipt + `MECHANIC-ATLAS.md` / `MECHANICAL-SCAFFOLDING-REGISTER.md` / primitive-pressure-ledger reconciliation, gated by `node scripts/check-scaffolding-governance.mjs`; `Engine Changes: Yes (CI evidence receipt)`; pairs with the step-2(a) pre-code audit), + a **capstone** (mechanic atlas, status hygiene, spec/index `Done`-flip). §Exit-criteria acceptance is typically **distributed** across the bench / wasm-smoke / tool / browser-smoke / CI tickets rather than forced into a synthetic capstone.

Honor the spec's Work-breakdown item count as the granularity floor; split a WB item (commonly the docs item, which fans out by validator) per the §Sizing rules.

## Non-game / infrastructure-spec pattern

Not every spec admits a new `games/*` game. Cross-cutting specs — a shared web-UI surface, a docs/contract obligation, a tooling/validator addition (e.g. `specs/rules-display-shared-surface.md`, decomposed as `archive/tickets/RULDISSHASUR-*`) — have no `OFFICIAL-GAME-CONTRACT` pipeline to follow. They share a recurring shape, in dependency order:

1. **Contract / docs foundation first** — the template, foundation/area-doc amendments, and the `specs/README.md` index row (`Planned`) that make the obligation permanent. Everything else depends on the contract it defines. **Closeout-deferred-docs variant**: some infra specs deliberately defer their doc/law amendments to a trailing closeout (the spec marks them "applied at closeout, not before") because the amended doctrine *describes machinery that must exist first* — there the amendments land in the §Capstone closeout (step 7), not this foundation ticket, and only the `specs/README.md` index row (`Planned`, often already present) is front-loaded. Spec-declared sequencing overrides this docs-first default (consistent with §"Spec WB dependency annotations are advisory").
2. **Tooling / validators** — the deterministic CI guard scripts (e.g. copy + fail-closed check) that enforce the contract. Wire the CI step with the ticket that first makes it go green (avoid a multi-PR red-CI window), not necessarily the script-authoring ticket.
3. **Content authoring** — the per-item content the contract requires (e.g. per-game player docs), often split pilot-vs-rest so the pilot anchors house style; the pilot `Deps` the contract + tooling tickets.
4. **Presentation infrastructure** — the shared component + presentation-only state (FOUNDATIONS §2: no legality decided in TypeScript), independent of content authoring and parallelizable with it.
5. **Integration** — wire the access/entry points to the presentation infra; `Deps` the component ticket (structural producer/consumer).
6. **Verification smoke** — the e2e / accessibility / no-leak smoke; a §Deliverable-doubles-as-capstone ticket (it ships new test infra), `Deps` the content + integration tickets.
7. **Capstone closeout** — verification-only acceptance evidence + the `specs/README.md` `Done`-flip (`Deps` the smoke leaf and the index-row-creating ticket, a create-then-modify on `specs/README.md`).

§Exit-criteria acceptance is typically **distributed** across the tooling / smoke / capstone tickets rather than forced into a synthetic capstone. Honor the spec's Work-breakdown count as the granularity floor (§Sizing rules) and split oversized items (authoring, UI) per the §Split rule.

**Multi-layer platform-capability variant** — some infra specs are not a single shared web-UI surface or one validator but a *platform capability* spanning several layers at once: a Rust bridge/kernel-adjacent generalization, a tool generalization, a new shared UI surface, and a new test harness (e.g. an N-seat-readiness spec). The 7-step UI-surface shape above mostly does not map — decompose by **Work-breakdown item / layer** per the granularity-floor rule (the same way the official-game pattern decomposes by sub-concern), and let the §Capstone closeout carry the deferred doc lift + `Done`-flip via the §Ticket-shapes infrastructure-spec exception.

**Pure-docs / governance-realignment variant** — some infra specs are *all documentation*: their deliverables are foundation/area-doc, template, ADR, and `ROADMAP.md` edits with no tooling, validator, presentation, or smoke surface (the doc/law-realignment and aftermath specs — e.g. `gate-9-aftermath-roadmap-realignment`, or a next-phase foundation realignment). Steps 2–6 above (tooling/validators, content authoring, presentation, integration, smoke) are then N/A — do not invent them. The shape collapses to, in dependency order: **(1) foundation** — one or more authorizing ADRs (accepted first) and/or a new contract doc that everything else cross-references, plus — when the spec amends the constitution — a separate `FOUNDATIONS.md` amendment ticket that `Deps` those ADRs and encodes their `Status: Accepted` as an acceptance precondition (mirroring the Governance-doctrine + enforcement-tooling variant's "foundation ADR + constitution first" ordering). When a roadmap/gate reorder is involved, the ADR *is* the foundation and the `ROADMAP.md` edit waits on its acceptance (see §Acceptance-gated ticket); **(2) doc-clarification content** — one ticket per coherent doc cluster, each `Deps` the contract doc it cross-links; **(3) template content** — template-cluster tickets, same dependency; **(4) the `ROADMAP.md` / law edit**; **(5) a status-reconciliation capstone** that flips the `specs/README.md` row and whose "smoke" is the repo's doc gates (`check-doc-links` / `boundary-check` / `check-catalog-docs`). Acceptance is distributed onto that capstone's gate run, not a test harness. Every ticket is `Engine Changes: None` (docs/status-only). **Forward-link red window**: an authority-map / index entry (a `docs/README.md` authority-map line, a `specs/README.md` row) that links a doc **another ticket in the batch creates** must co-land with — or `Deps` — that doc-creating ticket; otherwise `check-doc-links` (the pure-docs "smoke") goes red on the dangling link until the target lands. This is the docs-spec analogue of the `check-catalog-docs` / `check-player-rules` forward-reference red windows: prefer letting each new-doc ticket add its own authority-map/index line over front-loading all of them in the foundation ticket (so a foundation ticket with no `Deps` doesn't link not-yet-created docs), and flag any unavoidable interim red window in Step 4 / Step 6. **`check-doc-links` validates heading anchors, not just file paths** (it slugifies each `##`/`###` heading into an anchor set and checks `file.md#fragment` targets), so this red window also opens when a ticket's prose links to a *new section another ticket adds to an **existing** doc* (e.g. a contract/template/agent-law ticket linking `MECHANICAL-SCAFFOLDING-REGISTER.md#forward-per-game-maintenance-cadence` before the register ticket adds that heading): the citing ticket `Deps` the section-adding ticket, exactly as for a new-file link. This is a create-then-link ordering dependency (see §Intra-batch create-then-modify chains, generated-from clause).

**Governance-doctrine + enforcement-tooling hybrid variant** — some governance specs are *mostly* the Pure-docs shape (foundation/area-doc, template, ADR, `ROADMAP.md`, `specs/README.md` edits) but *also* ship a CI guard that enforces the new doctrine: a `scripts/check-*.mjs` validator, a static evidence receipt it consumes (e.g. `ci/<unit>-audits.json`), the validator's Node test suite + fixtures, and one CI-wiring step. Neither the Pure-docs variant (which excludes tooling/validator surfaces) nor the infra 7-step shape (no content-authoring/presentation/integration layers) maps alone — decompose as the **union**: the Pure-docs doc/law/template tickets (foundation ADR + constitution first, then the area-doc/contract/atlas/register clusters, then agent/testing/roadmap/tracker law, then the templates) **plus** the infra pattern's step-2 tooling sub-chain in dependency order — **receipt → checker → test-suite+fixtures → CI-wiring** (wire the CI step at the ticket that makes it green, per §Non-game step 2) — closed by a status-reconciliation capstone whose "smoke" runs both the repo doc gates *and* the new checker, and which owns the `specs/README.md` `Done`-flip. The receipt/checker/test/CI tickets are `Engine Changes: Yes` (not `None` — see SKILL.md §Step-5 CI-data-file / tooling categories); the doc/law/template/capstone tickets are `None`. Precedent: the Pre-Gate-18 forward scaffolding-reuse governance unit `8F` (`tickets/PREGAT18FORSCA-*`, 21 tickets — 16 doc/law/template + a 4-ticket tooling chain + a capstone).

**Non-gate game-presentation / UX-enhancement variant** — some specs enhance an *already-shipped* `games/*` game's presentation (additive Rust view fields + WASM projection + web renderers across the existing crate and shell) without admitting a new game, a new shared UI surface, or a new validator — so neither the Official-game-gate pattern nor the shared-infra shapes above map. Decompose **1:1 by the spec's Work-breakdown items** (the granularity floor), splitting only an oversized item per §Sizing rules; most tickets are `Yes (presentation-only)` web or additive-Rust-view, and the closeout is a docs-reconcile + `specs/README.md` `Done`-flip capstone (the §Capstone Test-Plan combined docs+verification fallback applies). The closest precedent is the archived River Ledger presentation decomposition (`archive/tickets/RIVLEDSHO-*`, the showdown-legibility pass).

**Trace-invalidating correctness fix (existing game) — defer trace reconciliation, flag the red window.** When this variant carries a *correctness* fix (not just additive presentation) — a logic ticket that changes an existing game's serialized/replay output (winner order, allocation order, labels, scoring) that a *later* reconciliation ticket regenerates — the golden traces and `replay-check` go red the moment the logic ticket lands and stay red until the reconciliation ticket reviews and regenerates them. This is the existing-game analogue of the validator-consumed-docs red windows (§Cross-cutting docs ticket → Tool-validated / catalog / player-rules), and it does not map to the new-game golden-trace co-location rule (where traces are *authored* with the module, not invalidated). Handle it: (i) scope the logic ticket's gate to **targeted native tests** (the new regression unit/property tests) and explicitly defer full `replay-check`/golden-trace assertions to the reconciliation ticket — so "leave its scoped tests green" means the logic ticket's own tests, not the to-be-regenerated traces; (ii) state the deferral in the logic ticket's Out of Scope (`replay-check may be red until <reconciliation ticket>`); (iii) make the reconciliation ticket `Deps` the logic chain and own the trace regeneration with individually-reviewed hashes (no bulk-accept); (iv) flag the interim `replay-check`/golden-trace red window in Step 4 / Step 6 as an expected mid-decomposition state, mirroring the `check-catalog-docs` red-window note. (River Ledger's `RIVLEDSHOSEA` decomposition: the pot/order/label fixes deferred all trace reconciliation to one trailing evidence ticket.)

**Existing-game behavior-extension / version-bump (vN cutover) variant** — some specs add *substantial new behavior* to an already-shipped `games/*` game (e.g. an all-in/side-pot layer over a base poker game) AND bump the per-game rules/data version — beyond the additive-presentation §Non-gate game-presentation variant and beyond a §Trace-invalidating *correctness fix*. Neither the Official-game-gate pattern (the crate and rules-source already exist) nor the existing-game variants above map cleanly. Decompose by the §Official-game-gate sub-concern pipeline **minus** the rules-source-first and crate-skeleton steps (the game exists and is already WASM-registered), honoring the spec's Work-breakdown count as the granularity floor; carry the §4 primitive-pressure ledger when the gate is a third-use mechanic comparison (defer/reject leaves the constructor/allocator local); and split the closeout into a docs-reconcile ticket + a `Done`-flip capstone (§Ticket shapes trailing-shape co-occurrence). **Version-bump red windows are the distinguishing hazard**: because the wasm catalog const already lists the game, the validator red windows do NOT open at a WASM-registration ticket (as the new-game §Cross-cutting docs ticket → Tool-validated / catalog / player-rules notes assume) — they open at the **version-bump ticket** (the one that advances `RULES_VERSION_LABEL` / `manifest.toml` `rules_version`) and stay red until the trailing docs-reconciliation ticket re-aligns the version-cited surfaces. Specifically, `check-player-rules` (HOW-TO-PLAY formal-rules version ↔ `RULES.md` version), `rule-coverage` (new rule rows unmapped until `RULE-COVERAGE.md` lands), and the golden-trace / `replay-check` set (per §Trace-invalidating correctness fix) all go red from the version-bump ticket onward; flag this interim red window in Step 4 / Step 6 as an expected mid-gate state. `BENCHMARKS.md` / `RULE-COVERAGE.md` validator co-location (§Cross-cutting docs ticket → Tool-validated docs) still applies. Closest precedent: the `river_ledger` Gate 15.1 all-in/side-pot extension (`archive/tickets/GAT151RIVLED-*`, 20 tickets — official-game pipeline minus the crate skeleton, a §4 defer/reject ledger, a v2 replay/hash migration, and the version-bump red window resolved at the trailing docs ticket).

**Behavior-neutral scaffolding / helper-adoption retrofit (multi-game) variant** — some specs adopt already-shipped shared scaffolding (effect-envelope constructors, canonical seat parsers, seat-count validators, parallel versioned-hash wrappers, dev-only evidence-profile drivers) across several *already-shipped* `games/*` games without admitting a new game, changing behavior, or adding a shared UI surface — a bounded byte/hash/visibility-neutral migration. Neither the Official-game-gate pattern nor the existing-game variants above map. Decompose **1:1 by the spec's §Work-breakdown one-surface-per-diff tasks** (the granularity floor), honoring the spec's own admission rule as a hard merge-override (§Sizing rules → Spec-declared admission rule overrides Merge) — keep each byte/hash-bearing migration its own ticket, never merged across games or surfaces even when siblings share a file. The 1:1 floor binds the byte/hash-bearing **migration** tasks (bucket 2 below); the spec's determination/characterization Work-breakdown tasks consolidate into bucket (1), and its consolidation/register/acceptance/status tasks into buckets (3)–(4) — so the migration count stays 1:1 while the governance ends legitimately collapse several Work-breakdown tasks per ticket (surface the resulting WB-task→ticket delta per Step 4's decomposer-initiated merge rule). **Ticket numbering**: this consolidation forces fresh numbers only for the foundation (`001`) and the register/capstone (`901`/`902`); preserve the spec's wave-grouped suffixes for the bucket-(2) migration body (`<NS>-101…`) per §Sizing → "Already-topological spec numbering" rather than sequential-renumbering the whole set, which loses the spec's wave-group→ticket traceability. **C-08 profile-adapter granularity**: dev-only profile adapters are test-only, not byte-bearing, so the 1:1 floor does not strictly bind them — keep them 1:1 by default but treat per-game merging as a Step-4 sizing-latitude option (surface the count delta). Shape, in dependency order: (1) a **characterization-baseline foundation ticket** pinning the pre-migration byte/hash/seat baseline + verdict matrix, which every migration `Deps`; (2) the **per-surface migration tickets** grouped by helper family (constructors, parsers, output/seat canonicalization, count validators, parallel-hash wrappers, dev-only profile adapters); (3) a **register-receipts + checkpoints ticket** (`Deps` all migrations) recording one receipt/exception per surface (checkpoints folded here include the C-06 reverse-edge proof, C-09 divergence receipts, C-10 local-behavior bundles, and adjacent-C-05 exception receipts — a standalone spec checkpoint task folds in rather than becoming its own ticket unless it carries an actual source edit such as a dev-dep edge fix or an accepted RNG migration); (4) a **verification-and-closeout capstone** (`Deps` the register ticket) that runs the spec's full acceptance command set, audits the candidate byte-diff against the admission inventory, and flips the `specs/README.md` row to `Done`. Surfaces the spec marks already-discharged by a prior pilot are **not** re-ticketed — verify them in the characterization + register tickets. **Report-edit discipline**: per-migration after-receipts are recorded in each migration ticket's Acceptance Criteria and *consolidated by the verification capstone* — they are **not** appended to the shared characterization report by every migration (which would force a many-way merge conflict on one markdown file, since the migrations are mutually independent); each migration `Deps` the foundation ticket for the pinned *baseline*, not to co-edit the report. This holds even when the spec frames its report as "append-only during execution" — the capstone performs the consolidation. Most byte-bearing migration tickets are `Yes (deterministic evidence)` and the dev-only profile adopters are `Yes (dev-only profile adapter)`; a C-06-style `[dev-dependencies]` edge ticket is `Yes (dev-only dependency)` and a C-07/visibility no-leak test-geometry ticket is `Yes (no-leak test geometry)`; `Engine Changes: None` is reserved for the characterization report, register, and capstone; a byte-neutral evidence/compatibility audit ticket that touches code but changes no bytes (e.g. a WASM seat-roster compatibility audit) takes `Yes (compatibility evidence)`, never `None`. Closest precedent: the 8C-R1 public/fixed-seat scaffolding retrofit (`archive/tickets/8CR1PUBFIXSEA-*`, 38 tickets across six games). Bodies are moderate full-template (~90–130 lines); per the §Step-5 rehearsal cadence, start ≤3 Writes/batch on a cold session with a post-batch self-check (a 38–50-ticket run is ~⌈N/3⌉ batches), dropping to 1/batch only on a divergence or longer bodies.
