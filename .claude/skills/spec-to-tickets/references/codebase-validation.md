# Codebase Validation (Step 2)

Before decomposing, validate the spec's assumptions against the actual Rulepath codebase. Collect everything; surface Issues to the user before Step 3.

**Parallel grep-batch guard**: when validating via `grep` / `find` / `test` calls batched in one parallel turn, guard each with `|| true`. A non-zero exit from one call (grep finding nothing, `test` on a missing path) cancels its siblings. Zero-match results are expected and valid during validation. Prefer several small independent Bash calls over one long `{ …; } | head` compound — a sub-command that dies mid-block silently loses every later command's output; if a compound block returns fewer results than commands, re-run the missing checks individually rather than trusting the partial output.

## Full Validation Path (default)

- **File paths** — Glob/Grep every path the spec references and confirm it exists where stated. Targets: spec files (`specs/*`), skill files (`.claude/skills/*/SKILL.md` and `references/*`), doc files (`docs/*`), template files (`templates/*`), and the code tree (`crates/*`, `games/*`, `apps/*`, `tools/*`). Distinguish existing paths (must exist now) from proposed paths (exist after implementation); proposed paths must not collide with an existing file and must follow conventions (kebab-case, `<gate>-<slug>.md` for specs, `<PREFIX>-NNN.md` for tickets, crate dirs under `crates/`, game dirs under `games/`, tools under `tools/`, per `docs/ARCHITECTURE.md`). **Generated-artifact check for edited paths**: for each existing file a deliverable proposes *editing*, check whether it is a generated/derived artifact (grep the path across `scripts/`, build configs, and codegen sources; `diff` it against a suspected source). If generated, the ticket must edit the canonical source and re-run the generator, with the parity/check guard (e.g. a `scripts/check-*.mjs` source↔generated assertion) as a Test-Plan command — a ticket that hand-edits a generated file is wrong. (Rulepath pattern: `apps/web/public/rules/<game>.md` is copied from `games/<game>/docs/HOW-TO-PLAY.md` via `scripts/copy-player-rules.mjs`, guarded by `scripts/check-player-rules.mjs`; see §"Validator-consumed *player-rules* docs" in `decomposition-patterns.md` for the placement rule.)
- **Types / schema / contract fields** — Grep each type, trait, struct, or schema/contract field the spec names. The schema/contract-authoritative sources are `docs/ARCHITECTURE.md` and `docs/ENGINE-GAME-DATA-BOUNDARY.md` (action-tree, command-envelope, effect-envelope, diagnostic, public/private view, checkpoint, hash, serialization-boundary vocabulary; static-data manifest entries) and `docs/OFFICIAL-GAME-CONTRACT.md` (per-game deliverable set). Verify field names against these and the actual crate code, not against FOUNDATIONS prose (prose carries *intent*; the boundary/architecture docs and code carry the *actual* names, and the two can drift). **Apply this with equal rigor to fields named in the spec's proposed corrections / fix prose, not only fields it names as already-existing** — a field a spec proposes to *write* still needs its host structure's schema confirmed to define it; a wrong field in a proposed fix propagates through decomposition into a ticket exactly as a wrong field in a bug claim does.
- **Data-availability / projection** — when a deliverable *consumes* a value (a label, flag, or field a downstream surface renders/reads), verify it is present on the **specific surface the consumer reads** — the catalog entry, the WASM bridge JSON, the public/private view projection — not merely that it exists *somewhere* upstream (a `*.toml` manifest, a Rust struct). A value living in an upstream manifest but never projected to the consumer surface is unavailable to that consumer: the deliverable needs an added projection (Rust-side), or the consumer (TypeScript) would invent it — a §2 behavior-authority risk (TS inventing presentation data). A field-existence grep passes silently because the value exists upstream; the gap is the missing projection hop, and it understates the ticket's true Files to Touch exactly like §"Enum / consumer blast radius" under-enumeration. **Spec-sanctioned presentation realization**: when the spec explicitly sanctions a §2-clean presentation realization that *avoids* the projection (e.g. a `game_id`-keyed TS/CSS mapping the spec's Assumptions permit), choosing it over the projection is a legitimate decomposition decision, not a §2 risk — surface it as the SKILL.md Step-4 spec-declared-implementation-latitude decision rather than forcing the projection hop.
- **Functions / commands / surfaces** — Grep each; confirm signature, location, crate placement (`engine-core` vs `game-stdlib` vs `games/*` vs `ai-core` vs `wasm-api` vs `tools/*`), and that proposed commands/flags match existing conventions (`tools/*` naming, the legal-action API surface) and `docs/ARCHITECTURE.md`.
- **FOUNDATIONS references** — Grep every FOUNDATIONS principle, acceptance invariant, or stop condition the spec cites; confirm the section/rule still exists in `docs/FOUNDATIONS.md` with the claimed semantics.
- **Spec dependencies** — confirm each predecessor/successor gate in the spec's `authority order` and Sequencing section resolves to a file under `specs/` (or is an expected not-yet-specced gate per the `specs/README.md` index). An entry resolving to no file that the spec treats as existing is a broken-dependency Issue; a not-yet-specced sibling the spec correctly marks as such is a LOW annotate-the-reference finding.
- **Enum / consumer blast radius** — when the spec introduces, retires, or changes a string-enum value (a diagnostic code, effect kind, visibility scope, dispatch token) or a new field on an existing schema/contract, pick the analogous existing sibling and grep every consumer site across `crates/*`, `games/*`, `apps/*`, `tools/*`, `.claude/skills/*`, `docs/*`, `specs/*`. New values need an arm at each dispatch site; retired values need every consumer updated or retired with them. Under-enumeration of the consumer set is the most common way a deliverable silently understates a ticket's true Files to Touch — surface the consumer count explicitly.

- **New-game gates** — when the gate admits a new game (`games/<new>`), harvest the template sibling game's full file inventory during validation (`benches/<game>.rs` + `thresholds.json`, the `tests/golden_traces/` filenames, the `data/` layout, the wasm-smoke harness path under `apps/web/scripts/`, the per-tool registration arms), so new-file Files-to-Touch paths are convention-accurate before the write phase rather than discovered mid-write. For the `apps/web/scripts/` smoke harnesses (`smoke-load-wasm.mjs` for `smoke:wasm`, `smoke-ui.mjs` for `smoke:ui`, `smoke-effect-feedback.mjs` for `smoke:effects`) specifically, do not stop at the path — grep them for **hardcoded per-game assertions** (e.g. `catalog.some(... game_id === "<sibling>")`, a `["<sibling>"]` start-match): if present, they are **`(modify)`** targets of the WASM-registration and web-renderer tickets (a consumer-set under-enumeration per §"Enum / consumer blast radius"), not merely reference paths; if they auto-discover from the `crates/wasm-api/src/lib.rs` catalog const, they are not. Resolving this at harvest time prevents the mid-write discovery that otherwise forces editing already-written tickets to add the `(modify)` entry.

**Flag** any stale reference, missing file, renamed entity, or FOUNDATIONS-violating assumption as an **Issue**, with one carve-out: **mechanical drift** (a path-convention swap where intent is unambiguous, a capitalization variant, an extension normalization, an unambiguous relocation of the same artifact) propagates the corrected reference silently into tickets — note the propagation inline in the Step 2 summary rather than opening an Issue cycle. **Substantive drift** (renamed type, removed file, shifted semantics, FOUNDATIONS-violating assumption, a field that doesn't exist on the named structure) is an Issue requiring disposition. Decision rule: if a competent reader would immediately see the spec's reference and the codebase's actual reference are the same artifact (no semantics or scope change), it's mechanical; if the divergence raises a question about intent (rename vs. removal vs. typo vs. API shift), it's substantive.

A distinct case is a spec disagreeing with *itself* — a deliverable's prose count differing from its own enumeration (e.g. "seventeen traces" above a list that names eighteen). When the enumeration is authoritative, handle it as a ticket-level Assumption-Reassessment note authored to the enumeration (decompose to the eighteen), not a Step-2 Issue requiring disposition; flag the cosmetic mismatch inline in the Step 2 summary.

Present Issues to the user before Step 3. For each, obtain one of five dispositions:
- **fix-before-decomposition** — a small spec correction lands first (via `/reassess-spec`), then decompose against the corrected spec.
- **defer-to-follow-up-ticket** — with a named dependency.
- **reject-with-rationale** — spec too materially wrong to decompose this round; route back to `/reassess-spec`.
- **expand-scope-in-place** — validation surfaced materially understated scope; the spec's intent is preserved but tickets are sized against the wider surface the codebase requires (the spec is not edited, the decomposition is).
- **drop-as-moot** — the deliverable's named target doesn't exist AND its intent is satisfied by sibling deliverables OR is a structural no-op under the actual codebase shape; drop it from the decomposition with a Step 2 note naming the missing target and the covering siblings.

**Brainstorm-produced specs**: when `<spec_path>` was produced by the `brainstorm` skill earlier this session, the Full Validation Path still applies — brainstorm's triage validation is less systematic than `/reassess-spec`'s discipline. Treat brainstorm's in-context exploration as already-gathered evidence for the bullets it demonstrably covered, but run fresh greps for everything it did not explicitly verify (notably the schema/contract-field validation, since brainstorm's spec-drafting does not guarantee proposed-fix fields were schema-checked against `docs/ENGINE-GAME-DATA-BOUNDARY.md`). A multi-item brainstorm triage also writes a companion `docs/triage/<date>-<topic>-triage.md`; check its follow-up-items section and surface those follow-ups in the Step 6 cross-spec follow-ups list. When a `/reassess-spec` pass on a brainstorm-produced spec also ran this session and resolved its findings, that pass **supersedes** this Full-Validation-Path requirement — it already performed the systematic schema/contract validation this rule mandates (so this spec's brainstorm origin does not re-impose the full path) — and the Abbreviated Spot-Check Path below applies.

## Abbreviated Spot-Check Path (when `/reassess-spec` ran in-session)

If `/reassess-spec` was run on this spec in the current session and all findings were resolved, Step 2 may be abbreviated to a targeted spot-check (3–5 greps). Verify at least:

- **(a) Primary references** — the spec's primary crate / doc / type references still exist at the stated paths.
- **(b) Schema version** — if the spec modifies a structured schema/contract (action-tree fields, command/effect-envelope fields, public/private view shape, static-data manifest entry), verify it hasn't drifted since reassessment.
- **(c) Sibling specs** — no new specs under `specs/` reference the same surfaces.
- **(d) Additive extension** — for specs extending an existing schema/contract, consumers have been updated, or the extension is additive-only (new optional field with a default).
- **(e) Rename / removal blast radius** — for tickets renaming or removing a crate/module, doc-governed contract, acceptance invariant, schema field, or exported symbol, grep repo-wide (`crates/`, `games/`, `apps/`, `tools/`, `.claude/skills/`, `docs/`, `templates/`, `specs/`) for every symbol. Any match outside the ticket's Files to Touch either (i) joins Files to Touch, or (ii) splits into a follow-up ticket with an explicit dependency, or (iii) when the match lands in a *sibling spec's* committed deliverables/exit-criteria, is surfaced as a flagged Step 4 consideration (held even under auto mode, since altering a sibling spec's tested behavior is a cross-spec scope decision).
- **(f) New-field / restructured-field spot-check** — when the spec adds a field to, or restructures the shape of, an existing schema/contract (string → object, scalar → enum, single → array, or a `required` → optional relaxation that breaks consumers asserting presence), spot-check that the host structure's boundary/architecture docs, validation rules, and producing/consuming crates are covered by the spec's deliverables. Route any gap per (e).
- **(g) Test-vehicle existence** — when the spec names a test type, runner, or facility its acceptance depends on (e.g. "unit tests", "fake timers", a coverage threshold, a new bench lane), confirm that harness actually exists in the target tree (grep the relevant `package.json` / `Cargo.toml` / test dirs / smoke-script set). Absence is a **count-affecting Issue** requiring disposition — introduce-the-runner (its own ticket, shifting the count) vs. realize the tests in the repo's existing convention (e.g. the node `apps/web/scripts/smoke-*.mjs` pattern, with `node:test` mock timers when fake timers are needed). A spec naming "unit tests" in a tree with no unit-test runner otherwise yields tickets with non-existent test commands; this also fixes the per-ticket Test-Plan vehicle, so resolve it before sizing.

After spot-checks, render the exercised sub-checks as a compact inline list (e.g. `Spot-checks: (a) ✓, (b) ✓, (c) skipped — no new sibling specs, (d) ✓, (e) N/A — no renames, (f) N/A — no new field, (g) ✓ — tests use the existing node-smoke convention`).

**New-game gates** — even on the abbreviated path, harvest the template sibling game's file inventory (see the New-game-gates bullet under §Full Validation Path) so new-file Files-to-Touch paths are convention-accurate, including the smoke-harness enumeration-style check (hardcoded per-game assertions → the harness is a `(modify)` target, not just a reference path).

If `/reassess-spec` ran but some findings were **deferred** by the user, treat deferred items as out of scope for decomposition. Note them in the Step 6 summary as "deferred reassessment findings that may warrant separate tickets." Do not silently incorporate deferred findings into ticket scope.

## Validation Rules This Skill Upholds

- **§11 Fail-closed validation** — when a deliverable introduces validation, verify it stays deterministic and blocking, distinguishes warnings from blockers, rejects unknown fields by default, refuses behavior-looking fields (§5/§11), and names what failing means (block? warn?). Unaddressed second-order effects are Improvement findings at minimum.
- **§2 Behavior authority / determinism** — for deliverables touching legal-action generation, state transitions, scoring, RNG, replay, hash, serialization order, or traces, verify behavior stays in Rust (TypeScript presentation-only), no nondeterministic input enters canonical forms, and identical inputs + versions produce identical output. A change to replay/hash *semantics* additionally trips the §13 ADR trigger. Violations are CRITICAL.
- **§3 `engine-core` kernel boundary** — for deliverables touching `engine-core`, verify no mechanic/domain noun leaks into the kernel. A game-specific type in `engine-core` is a boundary-failure CRITICAL Issue.
- **§11 No-leak visibility firewall** — for deliverables affecting public/private view projection, previews, effect logs, bot explanations, candidate rankings, UI test IDs, or replay exports, verify no path lets hidden information reach a viewer the deterministic views forbid. Missing firewall is CRITICAL.
- **No silent retcon** — enforced at ticket-write time: every ticket modifying existing behavior cites the change rationale in its Assumption Reassessment (`tickets/README.md`). This skill's contribution is catching drift introduced *between* reassessment and decomposition; durable architecture-changing decisions are gated by §13 ADR triggers.
- **§12 Stop conditions** — any §12 stop condition the decomposition would cross (mechanic nouns in `engine-core`, procedural static data, YAML/DSL without ADR, TS deciding legality, hidden-info leakage, bot bypass, unbounded agent scope) is a CRITICAL Issue blocking decomposition until the user dispositions it.
