# Research Brief

Author the comprehensive, paste-ready prompt for an **external deep-research session** (ChatGPT-Pro) — here, where Claude has direct access to the whole repository — instead of reconstructing it interactively in ChatGPT-Pro.

This skill replaces **"Session 1"** of the user's two-stage routine:

- **Session 1 (replaced by this skill)**: explore the repo, interview the user to 95% confidence about their *actual* intent, then author a requirements-style prompt.
- **Session 2 (ChatGPT-Pro, the actual deep researcher)**: receives the uploaded manifest + the emitted prompt, explores and researches online as deeply as needed, and **produces the deliverable directly** — it does not re-interview.

The emitted brief is **self-contained**: ChatGPT-Pro Session 2 has none of this session's context, so everything it needs must live in the prompt, the uploaded manifest, and any out-of-band inputs the brief explicitly enumerates (e.g. uploaded source documents).

**Consumer variants.** Session 2 is **ChatGPT-Pro by default** (it receives the brief + uploaded manifest(s) and reads the repo via the upload / raw-URL / GitHub-connector access pattern). It MAY instead be a **local Claude deep-research session** — e.g. a future `/brainstorm` session — with **direct filesystem access** to the repo(s). When the consumer reads the repo directly: (a) frame the brief's §0/§1 around filesystem reads, not uploads or raw URLs; and (b) treat the manifest(s) as pinned **read-checklists** rather than upload bundles (Step 6). (a) and (b) always apply to a direct-filesystem consumer — but the **locking mode is an independent choice, orthogonal to access mode**: a direct-filesystem consumer may be a **locked producer** (the default Step 7 "produce directly, do not ask" contract — handle any single open design axis with a Step 4 **bounded delegation**, recommend-with-default, *not* an open discussion) **or** an **iterating consumer** (the `references/brief-template.md` §7 iterating variant — lock the §3 settled intentions but permit it to surface the open design axes for discussion). Pick per the user's intent: a session asked to *produce a spec/deliverable directly* (then run it through a downstream reassess/decompose pipeline) is a locked producer; a `/brainstorm`-class session asked to *iterate* is an iterating consumer. Do **not** treat "local / direct-filesystem" as automatically meaning "iterating." Everything else (interview to 95%, settled intentions, exploration + research mandate, IP discipline) is unchanged. Where this file says "ChatGPT-Pro Session 2," read "Session 2 (ChatGPT-Pro by default; a local `/brainstorm`-class session when so directed)."

**Out-of-band / private-IP-subject briefs.** Sometimes Session 2's inputs extend beyond the brief + repo manifest — most commonly uploaded source documents (PDFs, datasets) that this authoring session cannot itself read, and whose subject is **private licensed IP deliberately excluded from the repo and the manifest**. When this applies, the brief MUST: (a) enumerate every out-of-band input by name and role up front, so the locked Session 2 treats them as in-scope; (b) state plainly that the manifest excludes the subject material and why (it is private / not in the repo), so the absence reads as intentional, not a gap; and (c) carry an **IP-discipline section** directing Session 2 not to reproduce licensed prose, art, or component text into the deliverable and to keep the subject material out of the repo and every public surface (cite `docs/IP-POLICY.md` and FOUNDATIONS §10); and (d) **stay opaque on every public surface** — by default the emitted brief and the refreshed manifest land in the **public** repo's tracked `reports/` (a public surface), so they observe the no-leak rule: refer to the subject opaquely (e.g. "the first private licensed game"), let the out-of-band uploads convey its identity, and never write the licensed title or identifying source filenames into them. **Surface exception:** when the user (or established lane precedent) directs the brief into a *separate private repository*'s `reports/` precisely so it can carry the subject material — the sanctioned private-game lane is the recurring case — that brief is a **private surface**, like the private-lane deliverable it commissions: its **body MAY name the subject and describe mechanics** (still subject to the IP-discipline section's no-licensed-expression rule). Even then, keep the brief's **filename stem opaque** (mirroring the deliverable's opaque stem — see `references/brief-template.md` §7), and keep every **public**-surface artifact — most importantly the public-repo manifest — fully opaque. (Check the directory is tracked and follow any existing opaque precedent — e.g. a prior same-subject report that already avoids the title.) The opaque-stem requirement **overrides** the Step 2 "mirror the sibling slug" rule and any same-directory precedent: if an existing same-subject sibling's filename *names the subject*, do **not** mirror it — choose an opaque distinct-angle stem and note the directory's inconsistency in the Step 7 summary. This is the one case where the §2 manifest is *not* the whole of Session 2's repository-and-source picture — say so explicitly rather than relying on the default two-input model.

<HARD-GATE>
Do NOT Write the brief file or refresh the manifest until BOTH hold:
(a) the interview has reached ~95% confidence (or the user issued an early-exit "just go" — in which case remaining gaps are written into the brief as explicit, labeled assumptions); AND
(b) the Step 5 brief outline + settled-intentions summary has been presented in chat and the user has approved it (silence on a section while answering other questions counts as approval; an explicit objection re-opens that section and requires re-presenting the correction first).
The skill mutates only `reports/` — for a multi-repo brief, the `reports/` tree of *each* repo the brief names (see **Multi-repository briefs** in Step 6). It NEVER edits `docs/`, `specs/`, `tickets/`, `templates/`, `.claude/skills/`, or source code.
</HARD-GATE>

## Invocation

```
/research-brief "<what ChatGPT-Pro should deep-research>"   [reference_path]
```

## Step 1: Classify the research target

Read `research_target` (and `reference_path` in full if given). Classify into one type — this selects the load-bearing "read in full" set for Session 2 (the type→reads map is in `references/brief-template.md`):

- **new-spec** — what to build/create next for the repo (a roadmap-gate spec).
- **thorny-fix** — diagnose and resolve a stubborn defect or design knot.
- **hardening / boundary-enforcement / anti-leak** — strengthen an existing system against drift, an engine/game/data boundary breach, a hidden-information leak, or weak proof.
- **foundational / doc-overhaul** — overhaul a foundation doc (or the cascade from an upstream doc change).
- **other** — anything else; build the read set from exploration alone.

Announce the classification on its own line as your **first user-facing output**, before any exploration tool call — emit `Classification: <type>` so the audit trail records it independent of your reasoning. When ambiguous, give a one-sentence justification. This first-output classification is **provisional**: it may be restated once you consult the `brief-template.md` §B type→reads map (the type descriptions there — e.g. `other (incl. audit / review / presentation-overhaul)` — are more detailed than the one-liners above). A restated classification on its own line is expected behavior, not a process slip.

## Step 2: Explore the repo to ground the brief

The point of authoring here is that Claude can read the repo directly — so the *user never types out what the researcher should read*. When the user nonetheless **volunteers a read set** in `research_target` (e.g. "have it read `docs/**`, `templates/**`, and `specs/README.md`"), treat that list as a mandatory floor, not the whole job: verify each named path resolves at the baseline commit, carry every one into the brief's §2 with its own one-line load-bearing reason, and then **enrich** (never replace or silently prune) it with the authority-ordered artifacts and code seams the user did not name. When a volunteered entry is a directory or glob rather than a single file (e.g. `docs/**` — "all files and folders"), verify it *expands* (`git ls-tree -r <dir>` against the baseline — a plain `ls <dir>` is acceptable here when `git status --porcelain` is clean, since the working tree then equals the baseline; otherwise use `git ls-tree`) instead of resolving as one path, give per-file load-bearing reasons for the files in it that actually bear on *this* target, and cover the incidental remainder with a single directory-level reason rather than forcing a contrived per-file line. If a named path or tier resolves to nothing — the user's structure is stale or imagined (e.g. numbered tiers `docs/0-foundation/*`…`docs/4-specs/*` when the tree is actually flat `docs/*.md` + `specs/`) — do **not** drop the intent: map it to the actual tree that satisfies it, flag the correction prominently in-session, and carry the corrected paths into the brief's §1/§2 so the locked Session 2 isn't pointed at nonexistent paths. Build, from exploration:

- the **authority-ordered read list** following `docs/README.md`'s ordered index (`FOUNDATIONS.md` → `ARCHITECTURE.md` → `ENGINE-GAME-DATA-BOUNDARY.md` → the relevant area docs → `ROADMAP.md`), then relevant `specs/`, `tickets/`, `templates/`, `docs/adr/`, `docs/plans/`, `reports/`, `archive/`, each with a one-line reason it is load-bearing for *this* target;
- the **relevant code seams** Session 2 should inspect (name files/modules in `crates/`, `games/`, `tools/`, `apps/web/` — don't paste them; Session 2 reads them itself);
- any **prior report / spec / ADR / archived work** that already bears on the target, so the brief frames the task as a delta rather than a cold start. When a prior brief/report cycle on the *same subject* has already been **implemented** (its recommendations shipped — check git history for the touched area), the delta is sharper than "build on prior thinking": the brief's §3 must state the implemented baseline explicitly and instruct Session 2 **not to re-recommend already-shipped work as if it were missing** (build on it, improve where it falls short — do not rebuild). Without that explicit instruction the locked Session 2 will plausibly re-propose shipped surfaces, which is wrong downstream output. A distinct **settled-but-unimplemented** mode applies when the brief commissions the *decomposition or implementation* of a predecessor **design deliverable** — a report/plan that is decided but **not yet built** (e.g. a `/brainstorm` design plan whose decisions are final but whose code/specs don't exist yet): here §3 must instruct Session 2 to **decompose, not re-design** — treat the design's resolved decisions as final and forbid reopening them — which is neither the already-implemented "don't re-recommend shipped work" mode nor a cold start. The same sharp-delta applies when **no prior brief on this subject exists** but the target sits in a **serial / numbered lane** (wave N→N+1, gate→gate): a predecessor or sibling unit (e.g. the `8C-R*` scaffolding waves, sequential gates) has already shipped surfaces the new spec must build on — state that implemented baseline (the named pilots / receipts in the relevant register) and forbid re-proposing those shipped surfaces, exactly as for a same-subject prior brief. The `brief-template.md` §B new-spec row already recognizes this serial-unit class for the read-set; keep the sharp-delta trigger coupled to it. In a serial lane, also read the immediately-prior sibling brief in `reports/` and reuse its section anatomy as the template for the new brief — change the target/game set and re-center the wave-specific delta rather than re-deriving the structure from scratch. Here "section anatomy" means the **brief's** own §0–§9 sections, not the deliverable's internal backbone: **deliverable-type-pivot caveat** — when the immediately-prior sibling's *deliverable type* differs from the target's (e.g. a change-plan sibling but a spec target), mirror the brief anatomy but **re-derive the deliverable's §7 backbone from the target's type** (a spec follows `specs/README.md` + `OFFICIAL-GAME-CONTRACT.md` §1A; an advisory change-plan/audit keeps its own backbone) rather than carrying the sibling's deliverable structure over. **Family-pivot caveat:** when the immediately-prior sibling sits in a *different mechanic family* than the target (a ladder pivot — e.g. trick-taking → rummy → board-topology), split the two roles the serial-lane rule otherwise conflates: mirror that sibling for **section anatomy** and any **governance / process delta** (e.g. the `forward-v1` audit ordinal — Nth user), but draw the **mechanic delta baselines** from the nearest *earlier* siblings that share the target's mechanic family, and instruct Session 2 **not** to carry over the immediate sibling's family-specific surfaces or helpers (e.g. a trick-taking→rummy gate reuses no trick-taking helper). **The same anatomy/baseline split applies to other load-bearing delta axes beyond mechanic family — most sharply the *information model*: when the target's hidden-information posture differs from the immediate sibling's (e.g. a run of hidden-hand gates followed by a perfect-information board game), the brief's §3 must flag the shift and re-derive its consequences rather than mirror the sibling's posture — a perfect-information target flips the MULTI-SEAT §6 no-leak obligation to an explicit all-surfaces-public audit, marks the hidden-info replay-export ADR `not applicable`, and makes bounded perfect-information search bots permissible under AI-BOTS (still no Monte Carlo/ISMCTS).** **Access-mode-pivot caveat:** when the immediately-prior sibling's *consumer access mode* differs from the target's (e.g. a ChatGPT-Pro raw-URL/GitHub-connector sibling but a local direct-filesystem target, or vice versa), mirror the section anatomy but **re-derive the brief's §0/§1 input/access pattern** from the target's consumer (direct filesystem reads + read-checklist manifests vs. uploads / raw-URL / connector — see *Consumer variants* and `references/brief-template.md` §1) rather than carrying the sibling's §0/§1 over. Mirror the sibling's **filename slug pattern** too, not just its section anatomy (e.g. `gate-NN-<game>-next-spec-research-brief.md`), so the brief series stays consistently named — unless a same-subject collision (Step 6.1) forces a distinct angle slug, **and unless the opaque-stem rule overrides it: never mirror a sibling slug that *names* a private licensed subject (Step 6, private-surface clause) — choose an opaque distinct-angle stem instead.**

When the target concerns presentation, UX, or observable runtime behavior, ground it by **exercising the running artifact** — drive the live app and capture the actual rendered state (screenshots, verbatim on-screen strings) — and fold those concrete captures into the brief's context (if no running instance is provided, start it from the project's documented launch commands — see root `CLAUDE.md` / the `/run` skill — before driving it); source and docs alone under-ground a behavior-facing brief, and the live capture is also how you catch exploration findings that the current build has already falsified. **Greenfield-design caveat:** when the target is a *new* presentation that does not exist yet (an interface designed from scratch), there is no target artifact to drive — exercise the **reference / predecessor artifact** instead (the existing shell or sibling the design is modeled on) and capture its rendered state, or state explicitly why source-level grounding of that reference suffices; the under-ground warning then applies to the reference, not a nonexistent target. **Inherited-grounding caveat:** when the target decomposes a **settled predecessor design deliverable that already grounded against the live / reference artifact**, that grounding is **inherited** — cite it and read the reference at source level rather than re-driving the artifact (re-driving would redundantly re-ground the predecessor cycle's work); note the inheritance in-session so the skipped drive reads as intentional, not an omission. Conversely, when the live drive *fails to reproduce* a user-reported defect, record the non-reproduction explicitly in the brief — frame any code-level cause as a hypothesis to confirm rather than a settled finding, and task Session 2 to reproduce it (e.g. seed-hunt) before fixing — so a behavior the current build didn't surface isn't passed downstream as confirmed.

Launch Explore agents for broad surveys, partitioning the fan-out by authority layer / subsystem (foundation + area docs; touched code seams in `crates/`/`games/`/`tools/`/`apps/web/`; planning artifacts in `specs/`/`tickets/`/`reports/`; the worked failing case if there is one) so each agent has a non-overlapping beat. Instruct each agent to return **concise, path-and-symbol-focused findings** — file paths with one-line load-bearing reasons plus the relevant type/field/component names, never whole-file dumps or long prose — so a wide parallel fan stays within the per-turn tool-result budget and no later agent's results get silently dropped or deferred. Pass each agent a concrete bound, not just the adjective "concise": cap it at a fixed digest skeleton (path → one-line reason → relevant symbols), roughly ≤150 lines, with no narrative synthesis or summary tables. Sub-agents reliably over-produce against a qualitative instruction; a structural cap is the lever that actually holds. Verify any repo claim in `research_target` or `reference_path` against the actual tree; flag contradictions prominently. Apply the same skepticism to your **own exploration's conclusions**: before a subagent finding about current behavior or UI shapes the brief's framing (delta vs. cold start), cross-check it against ground truth — recent git history (`git log` for the touched area), where a runnable artifact exists the running app, and **for determination/status claims (use-counts, gate ordering, debt state, what's `Done`) the authoritative trackers/registers themselves (`specs/README.md`, `docs/MECHANIC-ATLAS.md` §10/§10A), not a subagent's restatement of them** — because Explore agents reason from static reads, miss recently-landed wiring, and routinely misreport derived counts. A single miscounted use-count or stale status can invert the brief's central premise (e.g. whether a third-use hard gate fires), so verify the load-bearing one against the register directly. For a **determination-as-target** (a "what's-next" decision), those direct authoritative-tracker reads *are* the grounding — an Explore fan-out is optional and often unnecessary; reserve it for genuinely broad surveys (multi-subsystem code seams, large doc sets) rather than laundering a tracker-settled determination through a subagent's restatement.

## Step 3: Light online research (optional)

Only to **sharpen scope and interview questions** — surface the named techniques, prior art, or decision axes the interview should resolve. The *deep* research is Session 2's job; do not do it here. **If skipped, you MUST still emit the one-liner** in-session (e.g., "Online research: skipped — repo-internal realignment") — treat it with the same discipline and prominence as the Step 1 `Classification:` line, so the audit trail is visible independent of the brief body. Emit it at a fixed point — immediately before you open the Step 4 interview (i.e., before the first `AskUserQuestion` or clarifying question), regardless of how many exploration waves precede it — so it has a structural anchor as durable as the `Classification:` line (which is pinned to "first user-facing output") and cannot silently slip just because the determination was repo-internal or the exploration ran in several passes. (Anchor to the *interview opening*, not to "exploration returns": Step 2 is often iterative — direct reads, greps, then `Explore` agents — so there is no single moment exploration "returns.") And if you will defer the interview to a later turn, defer this one-liner with it (and the pre-batch `Confidence / Gaps` block): emit all three in the response that *opens* the interview, never in an earlier exploration-launch turn — splitting them off into the exploration turn is the exact anchoring slip Step 4 warns against (see Step 4's deferral rule). Deferring also keeps the one-liner from going **stale**: a research judgment emitted before exploration and intent resolve can be *contradicted* by the time you draft (a turn-1 `Online research: skipped` later flipping to `calibrated` once the design work surfaces real external angles) — emit it once, in the interview-opening turn, after the judgment has actually settled.

## Step 4: Interview to 95% confidence

Reach **95% confidence about what the user actually wants** — not what they think they should want — before drafting. Display this block after each answer (or, when a batch is sent, once before the batch and once after it):

```
Confidence: X%
Gaps: [specific remaining unknowns]
```

The Confidence figure tracks **user-intent resolution only**. Gaps you deliberately delegate to Session 2's design scope (UX/interaction details, final naming, anything the user explicitly said is "yours/Session 2's call") do **not** count against the 95% threshold — list them under `Gaps` tagged `(delegated to Session 2)` so they read as out-of-scope-for-the-user, not unresolved intent. The "95% — drafting the brief" announcement fires once user intent is locked, even when such delegated design scope remains open.

Rules: ask one *conceptual* question at a time when probing motivation or uncertainty sequentially, where each answer reshapes the next; but batch independent, already-scoped bounded choices into a single `AskUserQuestion` call (≤4 questions). Prefer bounded multiple-choice (`AskUserQuestion` when available). Probe motivation before solution; challenge premature specificity; name uncertainty specifically; respect demonstrated expertise and "you decide" delegation (re-evaluate and recommend, don't re-ask). Confidence rises from both answers and exploration findings; note which gaps each closes. On receiving batched answers, re-display the `Confidence / Gaps` after-block before proceeding — unless confidence reaches threshold, in which case the "95% — drafting the brief" announcement subsumes it. If the user answers a batched question conceptually, or rejects the `AskUserQuestion` widget to clarify rather than selecting options, treat it as an answer — lock what it resolves, re-display the `Confidence / Gaps` block, and re-ask only the still-open questions in a smaller widget, not the full batch. Announce "95% — drafting the brief" when reached.

You MAY dispatch the Step 2 exploration agents and the Step 4 interview batch in the *same* response — overlapping the two phases to save a turn — but only when the batched questions are already grounded by your own direct reads and do **not** depend on the still-pending exploration results. When you do overlap them, the pre-batch `Confidence / Gaps` block still fires (anchored to findings-so-far): the before/after-batch display is not waived just because the batch was launched alongside exploration. If a question's answer genuinely needs the exploration results, hold the interview until those agents return. When you do hold the interview to a later turn, hold the Step 3 online-research one-liner and the pre-batch `Confidence / Gaps` block *with* it — emit both in the same response as the deferred `AskUserQuestion`, not alongside the earlier exploration launch, so they stay adjacent to the interview opening they anchor to rather than landing a turn early.

**Determination-as-target**: when the research target is itself a decision ("what should we build / fix / spec next"), the interview MUST resolve and *lock* that decision here in Session 1. The brief then instructs Session 2 to **confirm-and-document** the chosen direction — citing the evidence that fixed it (e.g. the lowest non-`Done` gate, an empty promotion-debt register, satisfied predecessor preconditions) — never to re-open the determination open-endedly. A brief that leaves Session 2 to re-decide "what's next" violates the locked / no-questions contract (Guardrails). When the determination sits at a third-use hard gate or an open primitive-promotion decision, the **structural fork is part of the determination to lock**: decide *in Session 1* whether the resolution lives **inline in the named gate's spec** or in a **preceding interlock / debt-closure spec authored first** (the board-space / Gate 7.1 precedent) — because that fork changes what "the next spec" actually is, so it must not be handed to Session 2. A second structural fork is **deliverable count**: when the target naturally decomposes into multiple co-equal units (e.g. a deferred-escalations record naming two independent hardcodes, each needing its own spec), decide *in Session 1* whether the deliverable is **one combined document** or **N separate ones** — and, if separate, their authoring sequence. That choice also changes what "the deliverable" is, so lock it in the interview (a bounded `AskUserQuestion`) rather than leaving Session 2 to infer the packaging. **Explicit-delegation carve-out:** when the user *explicitly delegates* the decomposition count/boundaries/sequence to Session 2 (e.g. "produce as many specs as it makes sense to make"), lock the **combined-vs-separate** decision and the **dependency-marking requirement**, then hand the exact unit count and authoring sequence down as a **bounded design task** (Session 2 produces the dependency-map / index) — that honors the delegation without breaching the locked contract, exactly like the bounded delegation below. This is distinct from a **bounded delegation**: the brief MAY ask Session 2 to *recommend among enumerated options with a required default and justification* for a scoped design sub-choice the user deliberately delegated (e.g. "you pick the delivery mechanism from these two"). That is not re-determination and does not breach the locked contract — the determination and scope stay locked; only a named, optioned sub-decision is handed down, and Session 2 still produces directly rather than asking the user. **Research-depth mismatch**: when a non-game infra / scaffolding / governance unit (8M, 8C, 8C-R*, Infra A–D) is the target but the `research_target` framing requests deep external research ("deep research online", "research papers", "similar implementations"), the right instrument depends on which *kind* of non-game unit it is — do not treat them alike:

- **Behavior-free *code* plumbing** (8C, Infra A–D, and similar mechanically-next units): external prior art is genuinely thin — these are typed kernel/seat/byte helpers under accepted ADRs. Here the deep-research framing usually pictures a richer game gate (which may itself be hard-gated behind the plumbing unit), so surface the mismatch as a *confirming* interview question before locking — confirming the user pictured *this* target (vs. silently proceeding) is the 95%-intent move, not re-opening "what's next."
- **Process / governance / doctrine** units (reuse-doctrine, evidence/registry governance, workflow-obligation changes): external prior art is *real and load-bearing* — refactoring discipline (rule-of-three), paved-road / golden-path platform governance, reuse economics, ADR/registry practice. Here a confirming question is the wrong instrument; instead authorize the depth with a **calibrated §5 research mandate** (per `references/brief-template.md` §A.5) that names the genuinely-relevant external angles and bounds them to *sharpening, not scope expansion*. A confirming question is optional, not required.

This refinement also loosens the trigger: the mismatch applies not only to a *tracker-deterministic* pick (the lowest non-`Done` unit) but equally to a **user-raised governance/process concern not yet on the tracker** that requests deep research — that is a common arrival path for doctrine targets, and it routes to the calibrated-mandate branch above. The determination, where one exists, is still locked off the tracker.

**Feasibility/viability-assessment-as-target**: distinguish this from a *determination*-as-target. When the target is "is it viable / feasible to do X" (a viability assessment, feasibility audit, or thorny-fix-style judgment), the interview locks the **scope and intent** (sources, oracle authority, constraints, what's out of scope) — but the **viability verdict itself is Session 2's analytical output and is NOT locked in Session 1**, exactly like an audit or thorny-fix conclusion. Do **not** apply the Determination-as-target "resolve and lock the decision" rule to the verdict — that would wrongly pre-decide the research product. The brief tasks Session 2 to *render and justify* the verdict against the evidence, not to confirm-and-document a pre-fixed one; locking intent (not the verdict) still satisfies the locked / no-questions contract.

**Early exit**: if the user says "just go," announce current confidence, list remaining gaps, and carry them into the brief as labeled assumptions (`assumption: X`) so Session 2 — which will not ask — treats them as decisions the user can later correct.

## Step 5: Present the brief outline (HARD-GATE)

Before writing, present in chat:

1. the **settled intentions** — the resolved decisions the interview produced (these become §3 of the brief and are what make Session 2 "locked");
2. the **deliverable spec** — exactly which downloadable markdown docs Session 2 must produce (per doc: replace / new / intermediate artifact — for the `new-spec` pipeline the spec is saved to `specs/<gate>-<slug>.md`, then reassessed in place by `/reassess-spec` and decomposed by `/spec-to-tickets`; **the exception is a private-lane / private-IP deliverable bound for a *separate private repository* — the public `specs/`→reassess→tickets pipeline does not apply; name the private repo's spec directory instead (see `references/brief-template.md` §7's private-lane exception)**; see `references/brief-template.md` §7 — plus filenames);
3. the **read-in-full list** (authority-ordered, with the one-line reasons). When the read set is large — e.g. a near-whole-doc-set overhaul spanning many docs plus templates — the gate may present the list authority-ordered with reasons summarized by tier, or deferred to the brief, rather than one reason per file; the brief's §2 must still carry a one-line reason for every load-bearing file regardless (a directory/glob floor like `docs/**` may cover its incidental, non-load-bearing files with one directory-level reason — see Step 2).

If a minor residual ambiguity persists at this gate without an early exit (e.g. the user's wording left a detail genuinely open), surface it here as a labeled `assumption: <X>` for the user to confirm or override, rather than re-interviewing — then carry it into the brief's §3 the same way an early-exit assumption is carried.

Get approval (per the HARD-GATE). Revise on pushback before writing.

## Step 6: Write the brief and refresh the manifest

On approval, do BOTH:

1. **Write the brief** to `reports/<topic>-research-brief.md`, following the canonical anatomy in `references/brief-template.md`. `<topic>` is a short kebab-case slug of the target. First enumerate `reports/` (`ls reports/`) so the same-subject check below runs against the actual directory, not memory — it is the only reliable way to see whether a prior brief exists to avoid colliding with (its slug) or to reference (as the delta baseline). When the brief lands in a **different repo than its serial-lane siblings** (the private-lane case — this brief in the private repo, prior siblings in the public repo), enumerate **both** repos' `reports/`: the landing repo for slug-collision, and the sibling repo (commonly the public one) for the same-subject delta-baseline brief to mirror. Note that a **mature private lane accumulates its own sibling-brief series in the private `reports/`** (e.g. a `private-lane-p1-*` run of briefs co-located with the new one): the serial-lane predecessor to mirror may live in the *landing* repo, not the public one, so enumerate the landing repo for siblings too — don't assume the predecessor is in the public repo just because the build authority is. 2. **Refresh the manifest**: write the current repository path inventory to `reports/manifest_<today>_<shortsha>.txt`, where `<today>` is the real current date (`date +%F`), `<shortsha>` is the fetch-baseline commit's short hash (`git rev-parse --short HEAD`), and the inventory is the exact fetch-baseline commit's tree — `git ls-tree -r --name-only HEAD` (use the same `<baseline>` you pin in the brief's §1, not `git ls-files`, so the manifest provably equals the commit Session 2 fetches from). Encoding the baseline SHA in the filename keeps each brief's manifest pinned: a second brief authored the **same day at a different commit** gets its own file instead of overwriting an earlier brief's manifest — overwriting would silently break that earlier brief's §1 baseline↔manifest correspondence (its §1 still cites the old commit while the file now lists a new tree). **Baseline-commit rule.** The brief instructs Session 2 to fetch every file from one exact commit per repository it reads, so each manifest must list exactly that repository's commit tree. Derive the fetch-baseline commit from verified repo HEAD (`git rev-parse HEAD`) at manifest-refresh time, and generate the manifest from that same commit (`git ls-tree -r --name-only HEAD`) — do NOT use `git ls-files`, which reflects the staged index and silently diverges from HEAD under any uncommitted add/delete/rename. If you do fall back to `git ls-files`, first confirm `git status --porcelain` is clean (or reconcile every listed delta) and note the check in the Step 7 summary, since otherwise the manifest and the §1 fetch baseline will not agree. NEVER adopt a commit string copied from a report, doc, or `research_target` without confirming it contains every file in the §2 read-in-full list (`git ls-tree <commit> <path>` / `git cat-file -e <commit>:<path>`) — a "commit of record" cited inside a report is that report's *own* baseline and often predates later merges. If a referenced source cites a different commit, call out the divergence inside the brief rather than propagating it. Independently of how the baseline was derived (verified HEAD or a cited string), confirm before writing that every §2 read-in-full path resolves at the chosen baseline (`git cat-file -e <baseline>:<path>`); a hand-assembled read-list can name a moved, renamed, or deleted path even at a correct HEAD, and catching it here is cheaper than letting Session 2's self-check surface it after the brief ships.

**Multi-repository briefs.** When Session 2 must read more than one repository — e.g. a public repo for docs/templates plus a *separate* repo holding the code or data the deliverable concerns (the sanctioned private game lane is the recurring case) — author **one manifest per repo**, each generated from *that* repo's own verified HEAD via `git ls-tree -r --name-only HEAD`, written into *that* repo's `reports/`, and pinned to its own §1 fetch baseline. The brief's §0/§1 then carries one fetch-baseline **and access pattern per repo** — a public repo via the raw-URL pattern, a private or otherwise non-raw-fetchable repo via the ChatGPT GitHub connector by SHA (see `references/brief-template.md` §1). The opacity rule holds per surface (Step 6.3): an artifact in a public repo's `reports/` (brief or manifest) stays opaque; an artifact in a private repo's `reports/` is a private surface — a manifest may carry private paths, and a private-lane brief's body may name the subject (its filename stem still opaque). Writing the secondary repo's manifest into *its* `reports/` is within the "mutates only `reports/`" guardrail, scoped to each repo the brief names.

## Guardrails

- **Self-containment is the contract.** Session 2 has none of this session's context. Every path, decision, constraint, and acceptance check it needs lives in the brief, the uploaded manifest, or an out-of-band input the brief explicitly enumerates — never implied. When the source document uses shorthands defined in *another* report (e.g. a `web-seam §N` citation, a named digest), enumerate the shorthand→source mapping so the locked Session 2 can resolve them, and prefer pointing it at the underlying real artifact over the digest.
- **Claude authors; ChatGPT-Pro researches.** Don't perform the deep research here. The brief *commissions* it.
- **Locked, no questions.** The emitted brief instructs Session 2 to produce directly and NOT interview or ask clarifying questions — the interview already happened here. This is the default for every **locked producer**, including a local direct-filesystem session asked to produce the deliverable directly (access mode does not change it). **Exception — iterating consumer:** when Session 2 is *directed to iterate* (e.g. a `/brainstorm`-class session; see *Consumer variants*), lock the §3 settled intentions but permit it to surface the **open design axes** for discussion — the brief's §7 instruction then reads "the settled intentions are final; the open axes may be discussed," not "do not ask anything."
