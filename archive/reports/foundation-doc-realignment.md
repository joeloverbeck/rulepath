## Executive verdict

The next phase can be admitted without weakening the constitution, but the docs and templates need a realignment pass before execution. The center of gravity is not `engine-core`: the kernel already exposes generic `SeatId`, `Viewer`, `Actor`, `VisibilityScope::PrivateToSeat`, `Game::setup(seed, seats, setup)`, and replay `seats: Vec<SeatAssignment>`. The hard ceiling is instead in game crates, tooling, and presentation assumptions: per-game two-variant seat enums, setup rejection for counts other than two, simulator summaries named around `seat_0_wins`/`seat_1_wins`, and web copy/control logic that treats human-vs-bot as “seat_0 human, seat_1 bot.”

The most important documentation change is to add an explicit multi-seat / larger-surface contract underneath the existing foundation law. That contract should not move nouns into `engine-core`, should not introduce YAML or a DSL, and should not relax no-leak or deterministic replay. It should simply tell future specs how to express seat ranges, turn order, topology scale, pairwise no-leak matrices, per-seat outcome/showdown rationale, and benchmark scale targets before a game is built.

## Proposed ADR outline — Next public scaling ladder and Gate P tail placement

**Status:** proposed outline only, not accepted.

**Context/problem.** `ROADMAP.md` currently records a completed public ladder through Gate 14 and a deferred private Gate P. The next phase deliberately continues the public ladder before Gate P, adding N-seat and larger-surface capability through public, IP-safe games. The roadmap's own discipline says gate skip/reorder requires an accepted ADR.

**Decision.** Add a new public scaling phase after Gate 14. Gate P remains in the roadmap but is moved to the very tail after that public phase. The phase proves 3+ seats, larger surfaces, N-player hidden-information safety, multi-seat UI, and larger benchmark envelopes through public games first. Gate P remains private, optional, isolated, and non-architectural.

**Affected sections.** `docs/ROADMAP.md` stage/gate crosswalk and Gate P appendix; `specs/README.md` progress-index guidance; optionally `docs/README.md` if a `ROADMAP-NEXT-PHASE.md` subordinate file is added.

**Consequences.** Specs can be authored for Gate 15+ without ambiguity. Gate P cannot be pulled forward as an architecture driver. Any private monster-game pressure that appears before the public ladder completes is blocked or treated as non-public research with no foundation effect.

## Code and tooling seams that docs should name as evidence

- `engine-core` is not the bottleneck: `Game::setup` accepts `seats: &[SeatId]`; `Viewer` is optional-seat; `VisibilityScope` can be public or private to a specific `SeatId`; replay carries a vector of seat assignments.
- Per-game crates are the bottleneck: `race_to_n` and `poker_lite` define two-seat enums and setup rejects wrong seat counts; `frontier_control` and `event_frontier` set `STANDARD_SEAT_COUNT = 2` with fixed-size state arrays.
- The largest current surfaces are still small: `frontier_control` has seven sites; `event_frontier` has six sites, 21 cards, and three epochs.
- `tools/simulate` summaries are partly fixed around `seat_0_wins`/`seat_1_wins`, even where later games have faction wins, shared wins/losses, splits, or draws.
- The web shell contains useful generic surfaces but also two-seat assumptions: setup mode copy maps human-vs-bot to first/second seat; active actor labels fall back to Player 1/2; action controls compare a single actor seat to a singular `active_seat`; the outcome panel can render arbitrary final standings, so the outcome surface is closer to N-seat-ready than setup/mode controls.
