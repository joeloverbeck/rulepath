## 1. Repository baseline and governing constraints

Rulepath’s governing docs establish that the public polished playable site is the top product priority, Rust owns all behavior, and TypeScript owns presentation surfaces such as the game picker and setup UI while never deciding legality [RP-FOUNDATIONS]. The static-data boundary allows typed inert display content such as display names, short descriptions, icon IDs, theme tokens, accessibility labels, and authored prose, but forbids selectors, triggers, conditionals, behavior-by-naming, hidden identities, legality, state mutation, or behavior-like field semantics [RP-BOUNDARY]. The UI law calls the picker and match setup core public UI, sets the “cozy premium board-game table” direction, requires original SVG/components and color plus shape rather than color alone, and requires a shared How-to-Play/Rules surface from both picker and setup [RP-UI]. The IP policy permits original, permissioned, public-domain, or reviewed generated assets, but forbids copied prose, proprietary art/icons/fonts, screenshots/scans, trademark-forward presentation, and trade-dress mimicry; it explicitly says simple original SVG components are preferable to risky imitation [RP-IP].

The architecture doc confirms `apps/web` owns the picker, setup, layout, and static Markdown rules rendering, while not owning legality, hidden state, bots, or rule resolution [RP-ARCH]. The roadmap’s gate crosswalk confirms the official public roster is 14 games: `race_to_n`, `three_marks`, `column_four`, `directional_flip`, `draughts_lite`, `high_card_duel`, `token_bazaar`, `secret_draft`, `poker_lite`, `plain_tricks`, `masked_claims`, `flood_watch`, `frontier_control`, and `event_frontier`; Gate P is private/optional and subordinate to the public product priority [RP-ROADMAP]. The mechanic-atlas open promotion-debt register is empty at the target commit [RP-ATLAS].

## 3. Similar implementations: catalogs, launchers, portals

Mature game catalogs tend to solve the same three user problems: “What can I play?”, “Which option fits my mood?”, and “How do I get into a configured match?” At Rulepath’s scale of 14 public games, the useful patterns are not storefront monetization or licensed-art marketing, but consistent card anatomy, simple category cues, stable card proportions, accessible affordances, and a short path from browse to setup.

### What the catalog evidence says for Rulepath

Rulepath should adopt a fixed-ratio card grid with consistent slots, original per-game SVG identity, a short text hierarchy, and one secondary How-to-Play affordance per card. The strongest transferable pattern is “collection card → selected setup,” not “storefront browsing.” BGA/Yucata/Tabletopia/itch/Epic show that filters and categories matter when the library is huge, but Hick’s-law and choice-overload research suggest Rulepath should not add a filter sidebar for 14 games [EXT-HICK] [EXT-CHOICE]. The mature launcher pattern of fixed asset safe areas is valuable, but Rulepath must implement it through original geometric SVG and theme tokens, not through box art, screenshots, or figurative illustration [EXT-STEAM] [RP-IP].

## 4. HCI grounding

### 4.1 Aesthetic-usability effect and first impressions

The aesthetic-usability effect is relevant because users often perceive more aesthetically pleasing interfaces as easier to use, and early impressions can influence perceived quality even before deep task completion [EXT-AESTHETIC]. Lindgaard et al. found that users can form stable visual-appeal judgments of web pages after extremely brief exposures around 50 ms, which means the first screen of the catalog needs immediate structure, warmth, and confidence rather than a raw data-list impression [EXT-LINDGAARD]. This does not mean polish can compensate for broken behavior; it means the public picker/setup are themselves product surfaces and should look intentional.

**What this says for Rulepath:** the current text-only cards undercut the public-playable priority. P2 should make every card feel designed at first glance: fixed card shape, distinct icon silhouette, calm tactile surface, title hierarchy, and one-line summary. The visual system should be restrained because the constitution explicitly rejects casino vibes, SaaS-dashboard coldness, debug-console dominance, aggressive skeuomorphism, and trade-dress imitation [RP-FOUNDATIONS] [RP-UI].

### 4.2 Choice, information overload, and a 14-game grid

Choice-overload research challenges the assumption that more visible options always improve choice; large assortments can attract attention but reduce decision quality and satisfaction in some contexts [EXT-CHOICE]. Hick’s law, as commonly applied in UX, recommends reducing apparent choice count, breaking complex tasks into smaller steps, and highlighting recommended/default options when users face many choices [EXT-HICK]. Fourteen games is not a massive catalog, but it is enough to become a wall of undifferentiated text if the cards do not encode difference through shape, category, and hierarchy.

**What this says for Rulepath:** do not add a large filter sidebar, onboarding quiz, or launcher-style navigation. Use lightweight tags already present in catalog metadata—hidden information, cooperative, viewer modes, variant count—and improve the “single vs. multi-variant” line with optional descriptions. The setup screen should carry the selected title forward so the user no longer re-parses the catalog after selecting a game.

### 4.3 Visual hierarchy, scanning, and layout consistency

NN/g’s long-running scanning-pattern research shows users often scan web content in structured patterns such as F-shaped scanning when layouts are text-heavy, and strong visual hierarchy helps users find what matters instead of wandering randomly [EXT-FPATTERN] [EXT-SCANNING]. Grid and list usability research argues that product/category cards need consistent thumbnail sizes, text slots, and metadata placement to remain scannable, especially when users compare similar items [EXT-GRIDS]. Visual hierarchy principles emphasize contrast, grouping, alignment, and predictable rhythm [EXT-HIERARCHY].

**What this says for Rulepath:** use equal-height fixed-aspect cards with consistent icon well, title, summary, flags, and How-to-Play placement. Variable-height cards would make a 14-game catalog feel semirandom and force users to rediscover every card. A fixed aspect ratio plus text clamping is better than cards that grow based on variant copy.

### 4.4 Icon comprehension and abstract iconography

Icon usability research is blunt: icons work best when recognizable, familiar, and paired with clear labels; ambiguous or decorative icons should not replace visible text [EXT-ICONS]. Testing icon comprehension is recommended because designers often overestimate how obvious abstract symbols are [EXT-ICONTEST]. WCAG 2.2 requires that color not be the only means of conveying information, and that non-text UI components/graphical objects meet non-text contrast expectations where they convey meaning [EXT-WCAG] [EXT-WCAG-QUICKREF].

**What this says for Rulepath:** per-game SVGs should be identity marks, not primary labels. The visible display name remains mandatory. Each game identity must differ by shape family as well as color. Icon motifs should be tested at the smallest intended card size and in monochrome/high-contrast settings. Decorative SVGs can be hidden from assistive tech when adjacent visible labels already name the game; if the SVG stands alone in a compact view, it needs a real accessible name [EXT-WAI-DECORATIVE] [EXT-SVG-TITLE].

### 4.5 Progressive disclosure and one-line descriptors

Progressive disclosure reduces cognitive load by exposing primary choices first and moving detail into secondary surfaces [EXT-PROGRESSIVE]. UX microcopy guidance favors concise, clear, informative text that answers the immediate user question without becoming a rulebook [EXT-MICROCOPY]. Variant descriptions should therefore distinguish variants enough to support selection, but not explain legal procedure.

**What this says for Rulepath:** the optional variant-description field should be a one-line, original, neutral sentence fragment that helps the user choose: tone, pressure, duration, scenario feel, or public setup difference. It must not encode behavior, conditions, hidden identities, legality, or procedure. Detailed rules remain in authored HOW-TO-PLAY prose rendered through the Rules surface [RP-BOUNDARY] [RP-UI] [RP-IP].

## 5. Web visual-system research

### 5.1 Design tokens and CSS custom properties

The Design Tokens Community Group published a stable vendor-neutral Design Tokens Format Module in 2025, reflecting industry convergence around named design decisions that can be transformed across platforms [EXT-DTCG]. Rulepath does not need an external token build pipeline for P2, but it does benefit from token discipline: primitive tokens for raw colors/spacing/radii and semantic tokens for UI meaning. CSS custom properties are a native web mechanism for reusable, cascadeable values and scoped theming [EXT-CSSVARS]. The `prefers-reduced-motion` media feature lets CSS minimize nonessential motion for users who request it [EXT-REDUCED-MOTION].

**What this says for Rulepath:** implement a CSS custom-property token layer directly in `apps/web/src/styles.css` first. That approach aligns with the existing single vanilla stylesheet, avoids dependency risk, and supports per-game theme variables on each card/setup header. CSS Modules are a reasonable later containment tool for new component families, but they are not the required default because the catalog/setup redesign can be achieved with tokens plus a disciplined class namespace. A UI framework is unnecessary and explicitly out of scope.

### 5.2 Per-game theming without visual fragmentation

Design-token systems distinguish primitives from semantics so themes can vary accents while shared surfaces stay coherent [EXT-DTCG] [EXT-CSSVARS]. If every game gets arbitrary colors, the catalog becomes a patchwork. If every game only changes color, it fails WCAG’s color-alone rule and becomes fragile under color-vision differences [EXT-WCAG]. The right middle ground is one shared card layout, one shared icon grid, one stroke/fill grammar, and per-game tokens that vary accent pair, shape motif, and pattern.

**What this says for Rulepath:** each game should receive a small inert `catalog_theme` metadata record: `icon_id`, `theme_key`, accent token names, shape/pattern token, and accessibility label. The inline SVG registry remains TypeScript presentation, but the per-game selection of an icon/theme is Rust-owned typed UI metadata, consistent with the existing `ui.rs` pattern and UI-INTERACTION §10A [RP-UI] [RP-UI-FILES].

### 5.3 Card grid responsive patterns

Responsive card grids are most successful when they use fixed slots and fluid columns rather than arbitrary masonry; this preserves comparison and scanning [EXT-GRIDS]. Steam’s asset guidelines are storefront-specific, but the underlying discipline of fixed safe areas and small-size legibility transfers well to a portfolio catalog [EXT-STEAM].

**What this says for Rulepath:** use `grid-template-columns: repeat(auto-fit, minmax(...))`, a fixed card aspect ratio, and text clamping. Keep the icon well in the same location on every card. Avoid masonry, variable art sizes, background screenshots, and layout jumps.

## 6. Original SVG iconography and accessibility

An accessible SVG system needs clear responsibility: decorative SVGs should be hidden from assistive technology when adjacent text already communicates the label, while informative SVGs need proper accessible names via `role="img"`, `<title>`, or equivalent techniques [EXT-WAI-DECORATIVE] [EXT-SVG-TITLE] [EXT-SCOTT-SVG]. WCAG’s color and non-text contrast requirements require shapes, outlines, labels, or patterns in addition to hue [EXT-WCAG] [EXT-WCAG-QUICKREF]. Existing Rulepath `ui.rs` modules already encode shape/pattern labels for several games, proving the project has precedent for non-color-only presentation metadata [RP-UI-FILES].

**What this says for Rulepath:** build a project-authored abstract SVG set on a 24×24 base grid, with a larger 64/96px presentation size generated from the same motif. Use simple geometry, consistent stroke weight, and `currentColor`/CSS variables. Each game should be recognizable in monochrome by silhouette alone. Do not use AI-generated illustration, figurative art, copied symbols, playing-card suit marks as primary identity, casino chips, real board layouts, or anything that resembles a commercial title’s trade dress [RP-IP].

## 7. Variant-description authoring patterns

Variant descriptions are choice-support microcopy, not rules. Effective microcopy is concise, clear, and informative; progressive disclosure keeps procedural detail in the rules surface instead of forcing users to parse it during selection [EXT-MICROCOPY] [EXT-PROGRESSIVE]. Rulepath’s boundary law explicitly permits short descriptions and authored inert prose while forbidding selectors, conditionals, triggers, hidden behavior, or any field whose meaning mutates state [RP-BOUNDARY]. IP policy requires neutral, original prose and forbids copied rulebook text or trademark-forward framing [RP-IP].

**Recommended authoring contract:** optional; one line; recommended 55–95 characters; hard maximum 120 characters; plain user-facing sentence fragment; no raw IDs; no copied/proprietary terminology; no hidden information; no “if/when/then/selector/trigger/valid_if/effect/action/legal” phrasing; no rule procedure; no strategy advice; no casino/trademark terms. Good descriptions say what the variant feels like or what public setup pressure it emphasizes. Bad descriptions encode conditional behavior or try to be a mini-rulebook.

## 8. IP-safe abstraction by game family

The IP policy is conservative because public browser assets count as shipped assets [RP-IP]. For well-known game families, the safest approach is to represent the abstract pressure, not the commercial or traditional look. A 4-in-a-row game should not use a red/yellow vertical plastic grid; it can use four neutral tokens aligned on a path. A draughts-like game should not use a full checkerboard or copied crown treatment; it can use diagonal chevrons and stacked rings. A poker-like game should avoid suits, chips, green casino felt, and gambling vocabulary; it can use abstract slabs, shared pools, and ledger marks. A trick-taking game should avoid bridge/whist branding and suit marks as its main identity; it can use layered neutral cards and turn/order arrows. A bluffing game should avoid theatrical mask clichés that resemble specific games; it can use an offset tile and veil shape. Cooperative event/flood/frontier games should use original terrain, event, waterline, or node-network motifs rather than genre art.

**What this says for Rulepath:** original geometric SVG is not just safer; it is the best match for the project’s “warm, tactile, polished, original” doctrine. The design should include a per-asset IP closeout row for every shipped icon and theme motif, but should not amend `templates/PUBLIC-RELEASE-CHECKLIST.md` because the brainstorm note fixed P2’s closeout as its own asset checklist rather than a template change [RP-BRAINSTORM].

## 9. Synthesis: evidence-backed design decisions for P2

1. **Use original SVG + per-game theme tokens.** This is required by the locked decision and supported by IP, accessibility, and product evidence: fixed card identity improves first impression; original geometry avoids trade dress; color+shape meets accessibility expectations [RP-UI] [RP-IP] [EXT-AESTHETIC] [EXT-WCAG].
2. **Keep the catalog grid simple.** Fourteen games do not need heavyweight storefront filters. Use a responsive fixed-ratio grid, chips, and clear hierarchy. This addresses choice overload without overbuilding [EXT-CHOICE] [EXT-HICK] [EXT-GRIDS].
3. **Surface setup identity, not new behavior.** Carry selected game art/accent into setup, show variant descriptions, and make seat/faction labels prominent. Do not derive legality or rules in TypeScript [RP-ARCH] [RP-MATCHSETUP].
4. **Add optional variant descriptions as inert typed content.** The field is permitted by the boundary only if it remains optional, original, short, non-behavioral prose and does not expose hidden state [RP-BOUNDARY] [RP-IP].
5. **Use a token layer over vanilla CSS as the required default.** Existing CSS architecture is a single vanilla stylesheet with no framework; CSS custom properties deliver theming at low migration risk. CSS Modules are a future containment option, not the default [RP-STYLES] [RP-PACKAGE] [EXT-CSSVARS].
6. **No ADR, no FOUNDATIONS amendment, no mechanic-atlas row.** The redesign stays inside TypeScript presentation, Rust typed metadata, and original inert content; UI-INTERACTION §10A explicitly says repeated presentation shapes like `ui.rs` display metadata are not mechanic-atlas promotion pressure [RP-FOUNDATIONS] [RP-UI] [RP-ATLAS].
