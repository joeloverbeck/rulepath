# GAT202STACROACT-001: Rust active-seat-index catalog metadata for Starbridge Crossing

**Status**: COMPLETED
**Priority**: HIGH
**Effort**: Small
**Engine Changes**: Yes — `crates/wasm-api` (`src/catalog.rs` catalog JSON, `src/tests.rs` catalog test, `tests/snapshots/api_surface.tsv` golden snapshot)
**Deps**: None

## Problem

Starbridge Crossing's browser setup screen mislabels the active seats for 2-, 3-,
and 4-seat matches because the shared web shell derives the active-seat set itself
(slicing the first *N* labels of the catalog's flat six-point ring) instead of
consuming a Rust-owned active-seat-by-count mapping. Starbridge is the first game
whose active-seat selection is **discontinuous** (`SC-SETUP-003`), so "first *N*"
is wrong for it (e.g. 2 seats are North+South, not North+North East).

This ticket lands the Rust half of the fix: the Starbridge catalog entry gains a
Rust-owned, additive declaration of the active ring indices per supported seat
count, sourced from the existing setup authority `active_points_for_seat_count`.
The web shell consumes it in GAT202STACROACT-002. Keeping the active-seat fact in
Rust is the §2 behavior-authority requirement (`SC-UI-001`): the shell must
present setup, not recompute it.

## Assumption Reassessment (2026-06-28)

1. `active_points_for_seat_count` is the ground-truth active-seat mapping
   (`games/starbridge_crossing/src/ids.rs:151`), returning `&[StarPoint]` for
   `{2,3,4,6}` and `None` otherwise. `StarPoint::clockwise_index()`
   (`ids.rs:119`, `pub const fn`) maps each point to its 0–5 ring position. The
   six-point ring labels live in `catalog_starbridge_seat_labels_json`
   (`crates/wasm-api/src/catalog.rs:353`), index-aligned to `StarPoint::ALL`
   (North=0 … North West=5). Both symbols are re-exported at the crate root
   (`games/starbridge_crossing/src/lib.rs:29-30`) and `wasm-api` depends on the
   crate (`crates/wasm-api/Cargo.toml:27`), so the test can call the authority
   directly.
2. The Starbridge catalog entry is built inline in the early-return branch for
   `RegisteredGame::StarbridgeCrossing` (`crates/wasm-api/src/catalog.rs:302-311`),
   which `return`s before the shared `with_catalog_seat_metadata` path
   (`catalog.rs:313-318`); the new field is added to that inline JSON, not the
   shared metadata helper. Spec §4 and Assumption A1 confirm derivation by
   clockwise index, not re-authoring.
3. Cross-artifact boundary under audit: the catalog JSON contract consumed by
   the WASM bridge and snapshotted by `crates/wasm-api/tests/api_surface.rs`
   (asserts `_global/list_games` against `tests/snapshots/api_surface.tsv`,
   regenerated via `UPDATE_API_SNAPSHOT=1`, see `api_surface.rs:236`). Changing
   the catalog JSON requires regenerating the golden snapshot in this same
   ticket, or `cargo test -p wasm-api` goes red.
4. FOUNDATIONS §2 (behavior authority) and `SC-UI-001`: which seats are active
   for a seat count is Rust setup behavior. This ticket keeps that fact in Rust
   so the shell can present rather than compute it.
5. FOUNDATIONS §5 (static data is typed content, not behavior): the new field is
   typed identity metadata — an array of active ring indices per supported count —
   with no selectors, formulas, conditions, or triggers. It is generated by Rust
   from `active_points_for_seat_count`, not hand-authored data. Determinism:
   additive catalog metadata only; no accepted command, state, effect, view,
   hash, or replay surface changes (spec A2). The `api_surface.tsv` diff is the
   single expected additive change.
6. Schema extension: the catalog entry JSON gains one additive field. Consumers
   are the WASM bridge JSON read by the web shell (typed in
   `apps/web/src/wasm/client.ts` `GameCatalogEntry`, updated in
   GAT202STACROACT-002) and the `api_surface.tsv` snapshot (updated here). The
   extension is additive-only; existing consumers that ignore the field are
   unaffected.

## Architecture Check

1. Declaring the active ring indices in the catalog (sourced from
   `active_points_for_seat_count`) is cleaner than the alternative of projecting a
   per-match view pre-match: the setup screen has no match and thus no view, so a
   static per-game catalog mapping is the minimal Rust-owned surface that names
   the discontinuous active set without inventing a new view pathway. Indices
   (not formatted label strings) keep the Rust↔catalog bridge format-free —
   `StarPoint::label()` emits `"north_east"` while the ring labels are
   `"North East"`.
2. No backwards-compatibility aliasing/shims: the field is net-new and additive;
   no existing field is renamed or dual-written.
3. `engine-core` is untouched (no seat-ring/topology noun enters the kernel); the
   mapping stays game-local (`games/starbridge_crossing`) surfaced through the
   existing `wasm-api` catalog plumbing. No `game-stdlib` change.

## What to Change

### 1. Add the active-seat-index field to the Starbridge catalog entry

In `crates/wasm-api/src/catalog.rs`, extend the inline `RegisteredGame::StarbridgeCrossing`
catalog JSON (lines 302-311) with one additive field declaring the active ring
indices per supported seat count — e.g. `"active_seats_by_count"` keyed
`{"2":[0,3],"3":[0,2,4],"4":[0,1,3,4],"6":[0,1,2,3,4,5]}` — produced by a helper
(alongside `catalog_starbridge_seat_labels_json`) that derives each index list
from `active_points_for_seat_count(n)` via `StarPoint::clockwise_index()`. Do not
add the field through `with_catalog_seat_metadata` (Starbridge early-returns
before it). Keep the JSON shape consistent with the existing entry's escaping.

### 2. Regenerate the catalog golden snapshot

Run `UPDATE_API_SNAPSHOT=1 cargo test -p wasm-api --test api_surface` and commit
the single additive Starbridge-row diff to `crates/wasm-api/tests/snapshots/api_surface.tsv`.

### 3. Add the catalog-metadata equality test

In `crates/wasm-api/src/tests.rs`, add a test that parses the Starbridge catalog
entry's new field and asserts, for `n ∈ {2,3,4,6}`, that its index list equals
`active_points_for_seat_count(n).unwrap().iter().map(|p| p.clockwise_index())`.
Compare on indices, not on formatted label strings.
