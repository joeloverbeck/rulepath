## A2. Worked example verification

The live worked example in the brief is exactly the kind of tie-break the committed evaluator and explanation builder support: a six-seat checkdown where **Seat 5 wins with Pair of Jacks**, beating **Seat 2's Pair of Nines**, with the explanation “Both hands are one pair, so the pair rank decides first: Jacks > Nines.”

Evidence chain at `f1d6d39`:

1. `showdown.rs` evaluates only live showdown-eligible seats, combines each eligible seat's two hole cards with the five-card board, and calls the evaluator's best-5-of-7 path (`RL-SHOW-ELIGIBLE-001`, `RL-EVAL-SEVEN-001`).
2. `evaluator.rs` compares category first and tie-break vector second (`RL-EVAL-TIEBREAK-001`). For one pair, the pair rank precedes kickers in the tie-break vector.
3. `showdown.rs` builds `decisive_comparison`, `comparison_basis`, per-seat `hand_name`, `rank_explanation`, `comparison_note`, `best_five`, and `best_five_accessibility_label` (`RL-UI-SHOWDOWN-001`).
4. `visibility.rs` projects the terminal rationale and revealed showdown strength while preserving folded/private redaction (`RL-VIS-SHOWDOWN-001`, `RL-UI-NOLEAK-001`).

**Conclusion:** the tie-break encoding is correct. The gap is not the evaluator; it is the presentation hierarchy. The current panel says the right thing, but makes the player work too hard to see the winner, closest challenger, decisive rank, and card usage at a glance.

# Part B — Showcase UX / presentation redesign

## B0. Product direction: River Ledger as a “tabletop ledger,” not a casino table

**Recommended direction:** make River Ledger feel like a warm tabletop accounting duel: cards, river-like flow, brass ledger marks, parchment summaries, soft shadows, and tactile spacing. The look should be premium and board-game original, not casino mimicry. Avoid green-felt imitation, chip stacks, cash, rake, payout, tournament, or branded card-back tropes. Use the existing design-token system as the substrate and add River-specific tokens instead of globally repainting the app.

Why this direction:

- Poker/client prior art puts players around the edge, community cards and pot/ledger information in the center, and action controls near the active player; this helps users maintain spatial orientation [EXT-10]. River Ledger should use that spatial clarity without copying casino trade dress.
- Browser tabletop services demonstrate that community-card games can be presented as ordinary board games rather than gambling products [EXT-11]. That supports a neutral “showcase game” posture.
- The hardest UX moment is not “what is legal?” but “why did this outcome happen?” HCI guidance supports recognition over recall and progressive disclosure: show the decisive fact first, then let players inspect more detail as needed [EXT-03][EXT-04].
- Explanations of complex decisions are strongest when contrastive and selective: players ask “why Seat 5, not Seat 2?” more than “dump every evaluated hand at full detail” [EXT-05].

**Token-level defaults to add or use:**

| Token family | Recommendation | Boundary |
|---|---|---|
| River surface | `--rl-table-surface`, `--rl-table-rim`, `--rl-table-glow`, `--rl-board-well` built from existing warm/parchment/felt/brass palette, but tuned away from casino green. | TS/CSS presentation only; no behavior. |
| Ledger marks | `--rl-ledger-ink`, `--rl-ledger-muted`, `--rl-ledger-accent`, `--rl-ledger-winner` for contribution units and outcome emphasis. | TS/CSS presentation only; values must not imply money/chips. |
| Card states | `--rl-card-face`, `--rl-card-red-suit`, `--rl-card-black-suit`, `--rl-card-hidden`, `--rl-card-used-ring`. Suit identity still appears through glyph + suit word, not color alone. | TS/CSS presentation only; card identity from Rust. |
| Showdown hierarchy | `--rl-showdown-banner`, `--rl-showdown-challenger`, `--rl-showdown-detail`, `--rl-showdown-collapse-border`. | Shared outcome panel may consume optional River-local class hooks only. |
| Motion | reveal pulse/slide durations through the shared scheduler tokens; no component-local timers. | Shared scheduler boundary; reduced motion required. |

## B1. Showdown centerpiece redesign

### B1.1 Current failure mode

The current outcome panel has the right raw ingredients but the wrong hierarchy:

- winner emphasis is too thin;
- every revealed seat gets a vertically stacked full best-five strip;
- losing notes repeat the same “loses to Pair of Jacks” pattern;
- the community board is not visually tied to each seat's used five;
- the raw `seat_5` id can leak in Rust-authored copy;
- the panel is long enough that a novice must scroll before understanding the result.

This violates the spirit, not the existence, of `RL-UI-SHOWDOWN-001`: Rust already authors the explanation, but the presentation is not yet showcase-grade.

### B1.2 Target information hierarchy

The redesigned showdown surface should answer in this order:

1. **Who won?** `Seat 5 wins — Pair of Jacks`.
2. **Why this winner, not the closest challenger?** `Pair rank decides first: Jacks > Nines`.
3. **Which cards made the winning hand?** Show the board once and mark used board/hole cards for the winner.
4. **How did the field stack up?** Compact ranked standings, winner first, closest challenger second, remaining seats collapsed by default.
5. **Teach me more.** Expanders for hand-ranking ladder, per-seat tie-break vector, and full best-five accessibility labels.

### B1.5 Outcome panel implementation strategy

- Keep `OutcomeExplanationPanel` shared.
- Add a River Ledger-specific `RiverLedgerShowdownV2` subrenderer behind a discriminated optional payload, similar to the existing River Ledger showdown branch.

```text
  status_label: "Waiting",
  round_contribution: { label: "This round", value: "2" },
  hand_contribution: { label: "Hand total", value: "10" },
  hole_card_summary: { label: "Hole cards", value: "2 hidden", accessibility_label: "Seat 2 has two hidden hole cards." }

```

### B2.4 Action panel: stop showing irrelevant metadata

**Problem:** `Fold` and `Check` show `Call price / Adds / Cap left` rows even when those rows do not help the decision.

**Recommendation:** stop rendering raw metadata rows directly. Rust should supply display rows per action. TypeScript renders only those rows.

Rust field shape:

```text
RiverLedgerActionPresentation {
  segment: "fold",
  label: "Fold",
  helper_text: "Leave this hand. No more ledger units added.",
  display_rows: [
    { label: "Adds", value: "0", tone: "neutral" }
  ],
  accessibility_label: "Fold. Leave this hand. Adds zero ledger units."
}
```

Examples:

- `Check`: helper `Stay in; add 0.`, display rows `[Adds 0]`.
- `Call`: helper `Match the current round total.`, display rows `[Call price 2] [Adds 2]`.
- `Bet`: helper `Open this round's ledger.`, display rows `[Adds 2] [Raises left 3]`.
- `Raise`: helper `Call, then add one more unit.`, display rows `[Call price 2] [Adds 4] [Raises left 2]`.
- `Fold`: no `Call price`, no `Cap left`; those are technically public but decision-irrelevant.

This remains compatible with `RL-UI-ACTIONS-001` because Rust still supplies legal action controls and metadata. It improves the presentation contract by making metadata semantic rather than dumping every public counter.

### B2.5 Seat vocabulary

**Problem:** seat panels say `Seat 0` … `Seat 5`, while the shared status line can say `Player 1 to act`.

**Recommendation:** keep the current public seat numbering vocabulary (`Seat 0` … `Seat 5`) for this pass, because it matches current examples and avoids a cross-repo renumbering migration. Eliminate `Player N` from River Ledger status copy. The shared actor label function should consume catalog/Rust seat labels rather than hard-coded two-player `Player 1`/`Player 2` fallbacks.

Rust/static field shape:

```text
SeatDisplayLabel {
  seat_id: "seat_5",
  public_label: "Seat 5",
  short_label: "S5"
}
```

All player-facing River Ledger text should use `public_label`. Raw `seat_5` may remain an internal ID and stable serialization key, but not visible copy or accessibility text.

### B2.6 Raw `seat_5` leak

**Problem:** the visible copy can say `seat_5 wins with Pair of Jacks.` even when adjacent UI correctly says `Seat 5 wins`.

**Recommended fix:** author display labels in Rust and use them inside `showdown.rs`/`visibility.rs` before the string reaches the browser. Do not patch this with a TypeScript regex as the primary solution. `outcomeExplanationTemplates.ts` can keep fallback normalization for legacy generic games, but River Ledger terminal strings should be born clean.

Specific seams:

- `showdown_headline` should use `winner_label`, not `winner.as_str()`.
- `seat_list`, folded summaries, split summaries, and `comparison_note` should use public labels.
- `OutcomeRationaleView` per-seat rows should include `seat_label`.
- `stable_summary` may retain internal ids for deterministic hashes if that surface is not user-facing, but no visible/a11y string should consume it directly.
- Strengthen `scripts/check-presentation-copy.mjs`, `scripts/check-outcome-explanations.mjs`, and the River Ledger e2e/copy audit so `\bseat_\d+\b` fails in visible text, accessibility labels, and terminal outcome strings for public UI surfaces.

### B2.7 Motion and reveal pacing

Use the shared scheduler only. Recommended River Ledger motions:

- board reveal: a short settle/pulse on newly revealed board cards;
- active seat change: a subtle seat-ring emphasis;
- showdown: staged reveal in one scheduler burst: result banner → board usage highlights → standings settle;
- reduced motion: no movement dependency; all reveal facts appear immediately and are announced in text.

No ad-hoc timers. No preloading hidden future board cards into DOM, `aria-label`, `data-testid`, or animation ghosts. W3C accessibility guidance supports giving users a way to prevent non-essential interaction-triggered animation [EXT-08], and repo law already requires reduced-motion-safe scheduler behavior.

## B3. Bot “why?” affordance

### B3.1 Product shape

Add a small, non-debug “Why did the bot do that?” disclosure near the latest bot action or active status. It should answer in one sentence plus optional public facts. It must not display candidate rankings, hidden hand strength, private opponent cards, future board, deck tail, or solver claims.

Example display:

```text
Seat 3 checked.
Why? No call price was owed, so checking kept Seat 3 in the hand without adding ledger units.
Facts used: street Flop; call price 0; raises left 3; live seats 5.
```

For a call:

```text
Seat 4 called 2.
Why? Calling matched the current round total and kept Seat 4 live.
Facts used: call price 2; hand total after action 8; cap left 3.
```

For a fold:

```text
Seat 1 folded.
Why? The bot chose to leave the hand rather than add 2 ledger units.
Facts used: call price 2; street River; live seats before fold 3.
```

### B3.2 Rust-authored field shape

```text
BotDecisionPublicExplanation {
  seat: "seat_3",
  seat_label: "Seat 3",
  action_label: "Check",
  short_reason: "No call price was owed, so checking kept Seat 3 in the hand without adding ledger units.",
  public_facts: [
    { label: "Street", value: "Flop" },
    { label: "Call price", value: "0" },
    { label: "Raises left", value: "3" },
    { label: "Live seats", value: "5" }
  ],
  hidden_information_notice: "Uses only public state and this bot's authorized view."
}
```

For public/observer mode, the explanation uses only public facts. If a future owner-private explanation mentions the bot's own hole-card bucket, it must be visible only to that bot's authorized seat viewer and must still avoid raw card IDs unless those exact cards are authorized to that viewer. This follows `RL-BOT-EXPLAIN-001` and the public bot law: non-random bots should produce concise viewer-safe explanations, but hidden-information bots must not leak opponent secrets or actual hidden worlds.

## B6. Accessibility treatment

- **Color is never the only signal.** Suits retain glyph + suit word; used cards get shape/ring/text indicators, not only color. This follows WCAG color guidance [EXT-06][EXT-07].
- **Contrast:** text targets should meet WCAG AA: 4.5:1 for normal text, 3:1 for large text and non-text boundaries where applicable [EXT-06]. Validate token changes with the existing a11y smoke coverage plus manual contrast checks.
- **Screen readers:** terminal result banner should be a polite status region with atomic announcement. Best-five strings should use Rust-authored `best_five_accessibility_label`; table/seat summaries should use Rust-authored display labels.
- **Keyboard:** ranked field rows and expanders must be keyboard-accessible. Card usage highlights must have text equivalents.
- **Reduced motion:** all reveal facts must be visible without animation; scheduler effects settle to the same final view.
- **No leak:** accessibility labels, hidden DOM text, test IDs, logs, effect logs, storage, dev panels, and replay export must not contain unauthorized private cards, future board, deck tail, burn identities, or bot-private reasons.
